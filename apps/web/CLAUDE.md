# apps/web

The SvelteKit + Bun app. Architectural rules live in the repo-root
`CLAUDE.md` — read that first.

## Test command

`bun --bun vitest`. Vitest must run under Bun so the `bun:sqlite` import in
`lib/space/cache.ts` resolves; running under Node will fail with an unknown
module error. The package script `test:unit` already wraps this:

```
bun run --cwd apps/web test:unit
```

`bun --bun vitest --run` for a single non-watch run (what CI would use).

## Route modules

SvelteKit validates the *shape* of `+server.ts`, `+layout.server.ts`, and
`+page.server.ts` modules at request time, not at import time. Vitest tests
that import handlers directly will pass even when the module's exports
don't match what SvelteKit expects (extra exports, wrong export names,
wrong types). Route modules need at least one runtime smoke test —
`vite preview` against the example-space fixture, or a Vitest test using
`@sveltejs/kit/test` if that lands later. Direct unit tests of handler
functions are necessary but not sufficient.

## Routing

Files in `static/` are served before SvelteKit's route resolver runs. A
static `robots.txt` will silently shadow a dynamic
`routes/robots.txt/+server.ts`. When adding a dynamic route at a path that
could plausibly exist as a static file (`robots.txt`, `sitemap.xml`,
`favicon.ico`), check `static/` for a shadow. Reverse direction too:
removing a dynamic route doesn't restore a static file that was deleted to
make room for it.

## Test fixtures

Fixture `.amber/` directories are regenerated by the test runner, not
committed. Tests that need a hydrated cache should generate it via
`Space.load()` in their setup. Tests that need a fresh start should wipe
`.amber/` in their setup. The global setup in `vitest.setup.ts` ensures
the *initial* state is clean.
