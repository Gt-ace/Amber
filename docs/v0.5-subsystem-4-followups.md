## Open follow-ups

### Minor â€” `InviteRole` duplicates `SpaceRole`

`apps/web/src/lib/server/invites.ts` exports
`export type InviteRole = 'owner' | 'editor'`, parallel to
`apps/web/src/lib/server/permissions.ts`'s
`export type SpaceRole = 'owner' | 'editor'`. Three call sites use
`InviteRole`; the `PublicInvite` and `LoadState` types in the redemption
route use raw `'owner' | 'editor'` literals.

Fix: remove `InviteRole`, re-export or import `SpaceRole` everywhere.
Behaviour change is zero; the value is that a future `SpaceRole`
expansion (e.g., adding `'viewer'`) leaves no stale `InviteRole`
behind.

- First time: `invite.test.ts`'s `beforeEach` hit the default hook

  with what looked like cross-worker contention on the `getSpace()`
  singleton. Single-file re-run was clean; a second full-suite run was
  also clean.

The pattern is parallel `bun:sqlite` workers stressing each other's
temp-dir DB handles. The new subsystem-4 tests' seeding pattern is
correct (mkdtemp â†’ migrate â†’ seed â†’ `_resetAuthSingleton`); the
underlying race lives in `space.test.ts`'s registry/cache handling and
predates this subsystem.

Two possible mitigations:

- `pool: 'forks'` in `vitest.config.ts` (each test file in its own
  process â€” slower but no shared singleton state).
- `--no-file-parallelism` or `singleThread: true` on the suite (same
  effect, simpler config).

Operator workflow already tolerates the flake (`test:unit` is run
manually pre-deploy and is cheap to re-run). Closing the flake is
worth doing before the count of new tests doubles again.

### Open â€” Google OAuth invite-state path needs e2e confirmation
