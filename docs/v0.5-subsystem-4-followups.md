# v0.5 subsystem 4 — invites and per-space permissions follow-ups

Merge `16ee268` shipped subsystem 4 onto `main`. The final-review pass
caught two Important issues and one Minor; the Important pair landed in
the same branch as `f5cabe4` ("revokeIfOwner action + use getRole
helper"). What remains for future work is below — none block deploy.

## Resolved before merge

- **`already-member` redemption state had no action button** (spec §4
  matrix row 3 required owners to see "Revoke this invite"). The wiring
  in the Svelte template was missing; the `revokeIfOwner` action wasn't
  yet defined. Both landed in `f5cabe4`. Owners now see a Revoke form;
  editors see no extra affordance (the prose carries the explanation).
- **Two raw `SELECT role FROM member` queries bypassed the
  `permissions.ts` seam** in `/admin/invite/[token]/+page.server.ts`
  (one inside the `gstate` finalize branch in `load`, one inside
  `redeemAsCurrent`). Replaced with `getRole(userId, slug)` calls in
  `f5cabe4`. The seam now is the only place permission lookups live
  inside the subsystem.

## Resolved post-merge

- **`InviteRole` duplicated `SpaceRole`.** Removed `InviteRole` from
  `lib/server/invites.ts`; `invites.ts` now imports `SpaceRole` from
  `lib/server/permissions.ts`. The two `InviteRole` casts in the
  members admin route (`generateInvite`, `changeRole` — the latter
  was always semantically a `SpaceRole`) and the raw
  `'owner' | 'editor'` literals on `PublicInvite` / `LoadState` in the
  invite-redemption route are now `SpaceRole`. Behaviour change is
  zero; a future `SpaceRole` expansion (e.g., `'viewer'`) leaves no
  stale `InviteRole` behind.
- **Parallel-load test-isolation flake.** `pool: 'forks'` is set on the
  `server` vitest project (`apps/web/vite.config.ts`). Each worker is
  its own process, so the `getSpace()` registry and the better-auth
  singleton in `$lib/server/auth-config` truly reset per file; the
  cross-worker contention on temp-dir `bun:sqlite` handles described
  below no longer has a shared-memory surface to race on. Two
  consecutive full-suite runs are clean (50 files, 489 tests, ~16 s).

## Open follow-ups

### Open — multi-user e2e smoke

The existing `admin.e2e.test.ts` only exercises subsystem 2's single-
admin claim flow. A new e2e (perhaps `members.e2e.test.ts` under
`AMBER_E2E=1`) should walk the round-trip the spec describes in §13:

1. Claim install-admin at `/admin/setup`.
2. Generate an editor invite at `/admin/spaces/<slug>/members`.
3. Sign out; open the invite URL in a fresh browser; redeem as a new
   user.
4. Sign back in as install-admin; visit `/admin/users`; confirm the new
   user appears.
5. Remove the editor; their next request to
   `/admin/spaces/<slug>/edit/…` returns 404.
6. Try `bin/grant-ownership.ts --email <them> --space <slug>`; sign in
   as them; confirm they now see the members page.

Subsystem 4 unit-tested each branch in isolation; the e2e is the
"necessary but not sufficient" other half. Not blocking deploy because
the operator manually verifies before pushing prod; eventually the
operator wants to stop doing that.

### Open — Google OAuth invite-state path needs e2e confirmation

`auth-config.ts`'s user-create hook reads `gstate` from
`getRequestEvent().request.url.searchParams` to allow Google sign-up
during invite redemption. Whether better-auth forwards the original
sign-in query string through to its social-callback runtime depends on
better-auth internals that the unit tests can't exercise (no live OAuth
round-trip). The fallback is correct on inspection and the gstate-
finalize branch in the redemption load handles the post-callback case
either way, but the cold path is e2e-untested.

Fix: add a stubbed-Google-callback unit test (mirroring how the unit
suite tests the OAuth callback elsewhere), or a real-google manual
verification step in the operator's pre-deploy checklist for any
Amber install with Google configured.
