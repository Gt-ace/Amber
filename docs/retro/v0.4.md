# v0.4 retro

Written late — v0.5 is already five subsystems deep. The freshness
window is mostly past; this captures the carry-forward bits, not a
ceremonial recap.

## What shipped

`getSpace()` singleton swapped for a `Map<absolutePath, {space,
watcher}>` registry, with the zero-arg form preserved for single-space
callers and an explicit `getSpace(path)` form for multi-space callers
to consume. Desktop install via `docker compose up` verified
hermetically on a fresh WSL2 clone, backed by `spaces/example/` as the
minimal demo. `docs/self-hosting.md` published as the first full
operator-facing doc. The production compose file renamed
`docker-compose.yml` → `compose.prod.yaml` and pinned with `-f
compose.prod.yaml` in `bin/deploy` and `amber.service` so the box
can't silently swap to a dev compose on the next pull. Dead
`LoadWarning` enum tombstones removed; `bin/deploy` executable bit
restored and tracked; `amber_version` bumped to 0.4.

## What surprised — plan-vs-code drift, in one cluster

The same primitive (`auto_index`) drifted in three different
directions across plan documents, all caught only when actual code
was opened:

- `auto_index.path` is space-root-relative (`notes`), not URL-prefixed
  (`/notes`). The plan assumed URL prefixes.
- `auto_index.sort` values are space-separated (`date desc`), not
  underscored (`date_desc`). The plan used underscores.
- `{{{index_html}}}` is a theme-template variable, injected by the
  page template; it is **not** something a content author writes. The
  plan treated it as an author token.

These were three corrections in one session for one primitive.
Pattern: any non-trivial primitive that appears in both a content
contract and a theme contract needs the parser opened, not the spec
re-read from memory. Refreshing from memory is the failure mode.

Companion finding: the *content-author* side of `auto_index` was
documented only in passing inside `themes.md` (which is the
theme-builder reference, wrong audience). That gap propagated drift;
it's now closed by [`auto-index.md`](auto-index.md).

## What surprised — bin/deploy's "untouchable" lock was wrong

The `bin/deploy` no-touch rule was originally written to protect a
working script from speculative refactors. It turned out the real
risk on the v0.4 deploy was unrelated: production was about to
silently swap to a dev compose file on the next pull because both
compose files were unpinned in the SSH-side `docker compose up`
command. The lock was guarding the script body while the actual
invariant — *which* compose file the box runs — wasn't being
guarded anywhere. Once recognized, both the compose conflict and the
lock got revised explicitly, in writing, not papered over.

Pattern: an "untouchable" rule that was right when it was written can
be wrong by the time it's invoked. The cheaper move is to ask "what
does this rule protect?" and confirm the protected thing is still
the actual risk, before complying.

## What worked — and is now load-bearing for v0.5

- **Status-check section at the top of prompts that assume file
  state.** The first cut of the v0.4 install prompt was written
  without knowing files already existed; the revision added an
  explicit "verify current state of files X, Y, Z" step before the
  build steps. That pattern is now the default for any prompt
  touching existing infrastructure — every v0.5 subsystem prompt has
  one. Cheap to add, would have caught the original drift.
- **Hermetic verification.** The desktop install was verified by
  cloning fresh on a different machine and running the documented
  three commands. A stale dev checkout would have shown a falsely
  green result. The v0.5 e2e smoke (`bun run test:smoke`) carried
  this principle forward: the smoke project builds the production
  bundle from HEAD and boots it under Bun before driving Chromium,
  so it can't be fooled by a stale `build/` directory.
- **"Edges and limits" sections.** `docs/self-hosting.md` ends with
  a "Beyond this doc" block that names hardening, backups,
  monitoring, multi-space as explicitly out of scope. That edge
  marker pattern propagated to subsystem specs (each has an explicit
  non-goals section).

## What carries into v0.5

The path-keyed registry's main motivation was multi-space routing,
which was named at the time as "half-unblocked." Subsystem 3 (multi-
space routing) shipped on top of it cleanly; the abstraction was
right-sized. No registry follow-up was needed during subsystem 3 —
which is the small-but-real evidence that v0.4's refactor was
correctly scoped.

The v0.4 docs (README quick-start, `docs/self-hosting.md`) became
the model for subsequent docs. Subsystem 3's multi-space addendum
to self-hosting and subsystem 5's create-space affordances both
landed with doc updates that match the same shape (concrete commands,
edge markers, no marketing).

## What didn't carry — the off-roadmap item

A Tauri desktop wrapper was raised as a future consideration during
v0.4 and explicitly parked off-roadmap. The self-hostable canvas
identity is the through-line; a desktop wrapper would be a different
product wearing Amber's name. v0.5 is the authoring-layer answer to
the same "easier to edit" pain point that prompted the desktop-app
question — better aligned with what Amber is.

## One paragraph for v0.5 scoping

If v0.5 keeps the pattern that worked: read the parser before the
spec, add status checks before the build steps, ship docs alongside
code (not after), keep edge markers explicit. If v0.5 keeps the
pattern that didn't: assume rules written months ago still guard the
right thing; trust plan-summary phrasing over source code; ship code
before the doc that explains it. The post-mortems will rhyme.
