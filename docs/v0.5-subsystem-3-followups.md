# v0.5 subsystem 3 — multi-space routing follow-ups

PR #4 (`feat/multi-space-routing`) shipped multi-space routing and merged
with two small fixes applied in-PR:

- Admin-elsewhere redirect now preserves `AMBER_PUBLIC_URL`'s scheme
  (was hardcoded `https://`). Commit `5310ba7`.
- `bin/reset-password.ts` now uses `authDbPath()` so the documented
  offline recovery path works in multi-space mode. Commit `93b366c`.

The eight items below were surfaced in the same review and tracked as
GitHub issues. **All resolved on branch `fix/multi-space-routing-followups`.**
The order here is severity-first, not implementation order.

## High — multi-space user-visible regressions

### #5 — `editHref` hardcodes `/admin/edit`, lands in the wrong space

`apps/web/src/routes/[...path]/+page.server.ts:41` builds
`editHref = /admin/edit${url}`. The v0.4-compat shim then redirects to
the alphabetically-first slug (see #7). For any space that doesn't sort
first, the rendered "Edit this page" link silently opens the wrong
space's editor or 404s.

Fix: thread the active slug (available via `event.locals.space` and the
registry) into `editHref` — emit `/admin/spaces/<slug>/edit${url}`.

### #6 — Theme asset URLs lack the mount prefix

`themes.ts` computes `assetBase = /themes/<name>`. The page layout emits
`<link href="/themes/<name>/theme.css">` with no per-space prefix. A
prefix-mounted space ships unstyled: the browser fetches
`/themes/<name>/theme.css`, the resolver matches no prefix, the default
space serves the request, and its theme tree usually lacks that name
(404) — or, worse, serves a same-named theme's bytes from the wrong
space.

Fix: prefix asset URLs with the active mount, and route `/themes/...`
through the same prefix-stripper page paths use.

### #7 — v0.4-compat admin shims pick alphabetical, not `default = true`

Three shims compute `defaultSlug = path.basename(getRegistryEntries()[0].path)`:

- `apps/web/src/routes/admin/(authed)/edit/[...path]/+page.server.ts:14`
- `apps/web/src/routes/admin/(authed)/new/+page.server.ts:16`
- `apps/web/src/routes/admin/api/page/[...path]/+server.ts:13`

`getRegistryEntries()` is insertion order, which mirrors the alphabetical
sort in `spaces-dir.ts:75`. None of the three consult `routing.default`.
A bookmarked v0.4 URL like `/admin/edit/about` always lands in the
alphabetically-first space. The PUT shim is the worst case — buffered
subsystem-2 saves write to the wrong space.

Fix: look up the registry entry whose `routing.default === true`. Fall
back to `entries[0]` only when no default exists.

### #8 — `sitemap.xml` emits unprefixed URLs in prefix-mounted spaces

`apps/web/src/routes/sitemap.xml/+server.ts:17` calls
`buildSitemapXml(space.pages.values(), siteUrl)`. `sitemap.ts:77`
concatenates `siteUrl + page.url` with no awareness of
`locals.mountPath`. For a space at `prefix = "/scratch"`,
`https://site/scratch/sitemap.xml` emits `<loc>https://site/about</loc>`
for every page — those URLs route to the default space (or 404). And
crawlers fetching `https://site/sitemap.xml` (advertised in robots.txt)
only see the default space; prefix-mounted spaces are invisible to SEO.

Fix: prepend `locals.mountPath`'s prefix to each emitted URL. Separately,
consider whether multi-space installs should aggregate sitemaps at the
install root.

### #10 — Manifest redirects don't re-mount under the active prefix

`apps/web/src/routes/[...path]/+page.server.ts:24-30` looks up the
redirect map using the rerouted (mount-relative) URL and calls
`redirect(308, target)` — but `target` is also mount-relative and gets
emitted verbatim. For a space at `prefix = "/scratch"` with
`/old → /new`, the browser ends up at `/new` in the *default* space.

Fix: prepend the active prefix to `target` for prefix-mounted spaces
before redirecting.

## Medium

### #9 — `PUT /admin/api/page/[...path]` shim missing `requireAuthor()`

The shim at `apps/web/src/routes/admin/api/page/[...path]/+server.ts`
lives at `/admin/api/...` (**not** under the `(authed)` group) and the
PUT handler issues a 308 with no auth check. The v0.4 handler called
`requireAuthor()` first and returned 401 uniformly. Unauthenticated PUT
now returns `308 Location: /admin/spaces/<alphabetically-first-slug>/...`,
leaking the install's first slug; with zero spaces loaded it returns
`404 No spaces loaded`, leaking load state.

Not a write-bypass — the redirected endpoint enforces auth — but an info
disclosure regression vs the documented `requireAuthor()`-first contract.

Fix: add `requireAuthor(event)` to the top of the PUT handler.

### #11 — No reserved-prefix guard (PLAUSIBLE)

`apps/web/src/lib/server/space-routing.ts` validates prefix shape but
emits no warning for collisions with reserved roots: `/admin`, `/api`
(especially `/api/auth`), `/themes`, `/sitemap.xml`, `/robots.txt`. A
space declaring `prefix = "/admin"` would get rewritten by `reroute`
before SvelteKit's route matcher; the admin UI becomes unreachable and
recovery requires shell access to edit `space.toml`. Mirrors the
existing `space_routing_admin_host_collision` on the host side.

PLAUSIBLE — needs an integration test of the exact reroute-vs-resolve
interaction to confirm the failure mode.

Fix: add `space_routing_reserved_prefix` warning at parse time; drop the
prefix, keep the space.

## Low

### #12 — `discoverSpaces` silently skips spaces on `statSync` errors

`apps/web/src/lib/server/spaces-dir.ts:53-58` catches `statSync` errors
with a bare `continue` and no warning. Broken symlinks, permission
errors, or transient I/O drop the space from the registry with no log
trace. Spec §5.1 endorses `statSync` but doesn't sanction the silent
skip.

Fix: emit a `space_dir_stat_failed` (or similar) discovery-warning with
path and error message.
