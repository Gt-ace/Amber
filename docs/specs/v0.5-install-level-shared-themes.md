# Install-level shared themes — design spec

**Date:** 2026-06-04
**Status:** approved design, ready for implementation plan
**Companion docs:**
- Problem brief (the seed for this spec): [`docs/install-level-shared-themes.md`](../install-level-shared-themes.md).
- Theme contract (the binding rule this spec revises): [`docs/themes.md`](../themes.md).
- Architectural rules: [`CLAUDE.md`](../../CLAUDE.md) (root) and [`apps/web/CLAUDE.md`](../../apps/web/CLAUDE.md).

This spec covers *how shared themes work*. It changes the binding rule that
themes live per-space. Per `CLAUDE.md`'s "revise the rule here first, then the
change lands" discipline, the implementing change **must** land the
`docs/themes.md` and `CLAUDE.md` revisions in §8 in the same commit series, not
after.

---

## 1. Goal

A freshly created space — `/admin/new-space` scaffolds `amber.toml` + `index.md`
and nothing else — currently has no themes to pick: its theme picker shows only
"Use install default" and the space renders with the unstyled built-in floor
(`available:[]` in the loader log). The three canonical Amber themes
(`amber-default`, `amber-editorial`, `amber-brand`) are pickable today *only*
from `example` and `avp-software`, because each of those spaces ships committed
`themes/<name>/` copies.

After this change, the three canonical themes are **app-bundled shared themes**:
discovered once at install scope, available to *every* space out of the box —
new spaces included, with **zero operator setup** and no per-space file copies. A
space may still ship its own `themes/<name>/` to add a private theme or override
a shared one; per-space stays authoritative where present.

This is the minimum that solves the repro. It is deliberately scoped *not* to
include an operator-controlled override location (see §10, Non-goals).

## 2. Decisions locked during brainstorming

- **On-disk home:** the canonical themes **move** out of
  `spaces/avp-software/themes/` into a new app-owned directory
  `apps/web/themes/`. avp-software's copies are deleted; it resolves them as
  shared. No theme files are duplicated.
- **Override layer deferred (YAGNI):** *no* operator-controlled
  `<install-root>/.amber/themes/` location in this change. App-bundled is the
  only shared source. The override layer can be added later if a real need
  appears; the precedence design in §4 leaves a clean slot for it.
- **`Space.themes` becomes the effective (merged) map.** One merge point feeds
  all four consumers (§3) so a valid shared pick is never mis-classified as
  stale.
- **`BUILTIN_THEME` stays** as the last-ditch in-app floor (§5). It is not
  promoted to a "real" shared theme; the app-bundled `amber-default` satisfies
  resolution step 3 in every real install, leaving the floor reachable only when
  the app bundle's themes are absent (some unit tests).
- **`themes/` stays unwatched.** The shared set is discovered once at boot, held
  in memory; restart picks up new shared theme dirs — consistent with the
  existing per-space rule.

## 3. The merge seam — one effective map, four consumers

The central correctness requirement. The "effective" theme map for a space is:

```
effective = { ...shared, ...perSpace }   // per-space wins on name collision
```

computed **once** and threaded to every consumer. If any consumer saw only the
per-space map, it would mis-handle shared themes — e.g. the picker would classify
a new space's valid `amber-editorial` pick as `staleThemeName` (a missing
theme), telling the operator their working theme is broken. The four consumers:

1. **`resolveActiveTheme`** (`lib/space/themes.ts`) — resolves the active theme
   against the effective map.
2. **The picker's option list** (`routes/admin/(authed)/spaces/[slug]/theme/+page.server.ts`)
   — lists `[...space.themes.values()]`.
3. **`describeThemeSource`** (`lib/space/themes.ts`) — classifies the active
   theme's origin; must check the effective map for "is this declared theme
   real?", and gains a shared/this-space distinction (§6).
4. **The asset route** (`routes/themes/[name]/[...file]/+server.ts`) — must serve
   a shared theme whose files are outside any space root (§7).

### 3.1 Mechanism: thread shared themes through `Space.load`

`lib/space/` must stay free of process globals, env reads, and server holders
(`lib/server/space.ts` header states this). So the shared map is **passed in**,
never imported by the loader. `Space.load`'s existing options bag is extended:

- `Space.load(spacePath, options?: { cache?: boolean; sharedThemes?: Map<string, Theme> })`
  — add `sharedThemes`, defaulting to an **empty map**. Empty default = today's
  behavior exactly, so existing tests calling `Space.load(path)` /
  `Space.load(path, { cache: false })` and the pure-loader tests are unchanged.
- A small exported helper `effectiveThemes(shared, perSpace) → new Map([...shared, ...perSpace])`
  (per-space overwrites shared on collision) is the single merge primitive,
  applied at **both** places where themes are obtained (the cache stores theme
  *placeholders*, so themes are resolved on every load — see §3.2):
  1. **Cold path** — the pure `load(spacePath, sharedThemes?)` in `load.ts`:
     after the existing `discoverThemes(root, log)`, merge shared in, store the
     effective map as `space.themes`, resolve against it. `load()` gains the same
     optional `sharedThemes` param (empty default).
  2. **Hydrate / cache-hit path** — `Space.load` in `space.ts` (today lines
     ~163–174) re-discovers themes fresh because the cache persists none; merge
     shared in there too before `resolveActiveTheme`.
- The `apply()` `space_config_change` handler (theme hot-reload, `space.ts`
  ~line 362) resolves against `this.themes`, which is already the effective map
  (set in the constructor from `initial.themes`) — **no extra shared re-merge
  needed**, and a `space.toml` edit can now hot-switch to a shared theme too.
- The registry (`lib/server/space.ts`), which *does* have install scope, supplies
  the shared map: `loadEntry(spacePath)` calls
  `Space.load(spacePath, { sharedThemes: getSharedThemes() })`. This single call
  site covers **both** boot (`getSpace` ← `bootRegistry`) and the no-restart
  hot-add (`addSpace` ← `/admin/new-space`) — see §9, the literal repro path.

`space.theme` keeps its current type and meaning; it now resolves from the
effective map. Because `Theme` already carries an absolute `path`,
`readTemplate`/`readPartial` work unchanged for a shared active theme.

### 3.2 Cache stays untouched

Verified: the SQLite cache does **not** persist themes. `cache.ts` writes
placeholder `themes: new Map()` / `theme: BUILTIN_THEME` (with a comment that
`Space.load` re-discovers theme state), and the hydrate branch in `space.ts`
re-runs `discoverThemes` + `resolveActiveTheme` every time. So **no `cache.ts`
change is needed**: the shared merge is layered on in the two load paths above
(§3.1), and a cache hit and a cold load produce the same effective map because
both re-resolve themes from scratch. The bundled-themes directory on disk is the
truth for the shared set; nothing about "filesystem is truth, cache is
regenerable" changes.

## 4. Discovery & precedence

- **New module `lib/server/shared-themes.ts`.** Resolves the bundled-themes
  directory (§7.1) and runs the *existing* pure `discoverThemes(dir, log)` once,
  caching the resulting `Map<string, Theme>` in a module-level holder (mirrors
  `lib/server/resolver-index-holder.ts`). `getSharedThemes(): Map<string, Theme>`
  returns it; a `loadSharedThemes()` (or lazy init on first `get`) populates it
  at boot. A test-only reset mirrors `__resetRegistryForTests`.
- Each shared `Theme` carries `path = <bundledDir>/<name>` and the **same**
  `assetBase = /themes/<name>` as a per-space theme. The asset route (§7)
  disambiguates by name lookup, so the URL space is shared and flat — a space
  that overrides `amber-default` with its own copy serves *its* CSS at the same
  `/themes/amber-default/theme.css`, which is correct (the override wins
  everywhere, assets included).
- **Precedence (highest first):** per-space `<space>/themes/<name>/` →
  app-bundled `apps/web/themes/<name>/`. (A future operator override dir, if ever
  added, slots between the two — §10.)

## 5. Resolution chain & warnings

The 4-step chain in `resolveActiveTheme` keeps its shape; only the map it
resolves against changes (now effective, not per-space):

1. `space.toml` `theme` → `effective.get(...)`
2. `amber.toml` `theme` → `effective.get(...)`
3. `amber-default` → `effective.get('amber-default')` — **now hits a real
   shared theme in every install** (it no longer requires a per-space copy).
4. `BUILTIN_THEME` — the in-app unstyled floor, unchanged. Reachable only when
   the bundled themes are absent (e.g. unit tests with no bundled dir).

`space_theme_not_found` semantics are unchanged in spirit: steps 1/2 emit it
when they name a theme that isn't in the **effective** map (previously the
per-space map). A name that exists only as a shared theme now resolves cleanly
and emits no warning — the desired behavior. No new `LoadWarning` codes; the
enum is untouched.

## 6. Picker UI labelling

Data/labelling only — the picker is already on the brand system; no restyle.

- `describeThemeSource` (and/or the picker load) gains the ability to label each
  option and the active theme as **shared** (app-bundled) vs **this space**
  (per-space `themes/`). The picker runs in the server, so it has the effective
  map (`space.themes`) and can read the shared map via `getSharedThemes()`.
  Origin is then derivable without storing a separate pre-merge per-space map: a
  theme is **this space** iff it is absent from the shared map, *or* present in
  it but with a different `path` (the per-space override case); otherwise it is
  **shared**.
- The existing `source: 'space-toml' | 'inherited'` classification and
  `staleThemeName` keep working (now computed against the effective map). The
  shared/this-space label is an **additive** field on each listed theme and on
  the "Currently rendering" line; it does not change the existing chip logic.
- Picker option sort is unchanged (`amber-default` first, then alpha).

## 7. Asset route — serve a shared theme's files

`routes/themes/[name]/[...file]/+server.ts` today hardcodes
`resolve(space.root, 'themes', name)` and serves from there. A shared theme's
files are not under any space root, so the route must learn the shared root.

**Mechanism: directory precedence (not effective-map lookup).** The route
deliberately does *not* consult the discovered theme map — verified by the
existing asset test, which serves a theme dir that ships only `theme.css`/`theme.js`
(no templates) and so is **never discovered**. Requiring a discovered/effective-map
entry would 404 that and regress the route's decoupling from discovery. Instead:

- Compute `perSpaceRoot = resolve(space.root, 'themes', name)` and
  `sharedRoot = resolve(sharedThemesDir(), name)`. Pick the theme root by
  directory existence, per-space first: if `perSpaceRoot` is an existing
  directory use it; else if `sharedRoot` is, use it; else 404. This is the
  per-space-wins rule at the directory level — it never mixes assets from two
  roots.
- Keep the existing path-traversal containment check, now against the chosen
  root (`target` must be strictly below `<root> + sep`). Same guard, correct root.
- The single-segment `name` guard, content-type table, and cache policy are
  unchanged. No second route, no `/themes/_shared/` prefix.

This serves shared themes for spaces with no per-space copy, keeps per-space
overrides serving their own assets, preserves the route's independence from
discovery, and respects the v0.3 "asset-route per-space correctness" work.
`sharedThemesDir()` is exported from `lib/server/shared-themes.ts` (§4).

### 7.1 Bundled-themes directory resolution (the load-bearing mechanism)

Verified against a real `bun run build` of the app:

- `build/client/` is the *only* statically served tree (where `static/fonts/`
  lands → `/fonts/`). A sibling **`build/themes/`** is **server-only** — the
  adapter-node handler never serves it — which is exactly right: theme HTML
  templates (`chrome.html`, `page.html`, `error.html`) must never be public; CSS
  and fonts are served deliberately via the asset route.
- The container runs `bun ./build/index.js` with cwd `/app`, so `build/themes/`
  is addressable at `/app/build/themes`.

Mechanism:

- **Build:** the `build` script copies `apps/web/themes/` → `build/themes/` after
  `vite build` (e.g. `vite build && <copy>`). The Dockerfile's existing
  `COPY --from=build /app/build ./build` then carries it into the runtime image —
  **no new Dockerfile `COPY` line needed**. Use a small portable copy (a tiny
  node/bun script, or `cp -r` given the linux build environment — the plan picks
  one).
- **Prod path:** the Dockerfile gains exactly one line:
  `ENV AMBER_BUNDLED_THEMES_DIR=/app/build/themes` (mirrors the existing
  `AMBER_SPACE_PATH=/space` ENV pattern).
- **Resolution in `lib/server/shared-themes.ts`:**
  `process.env.AMBER_BUNDLED_THEMES_DIR ?? <dev default>`. In the built server
  the env var is always set, so the dev-default branch never runs there. The dev
  default derives the in-repo source dir `apps/web/themes/` from the module's own
  location (`import.meta.url` of `shared-themes.ts`), which is reliable in
  dev/test (module runs from source) and never taken in the bundled server.

This is the one piece that was unproven; the build-layout spike confirms
`build/themes/` is server-only and reachable, and that the existing image copy
carries it. No service added, no dependency added — only files shipped.

## 9. The repro path — `addSpace` hot-add

The literal repro is: create a space via `/admin/new-space`, open its picker,
*without a restart*, and see real themes. The new-space action calls
`addSpace(absPath)` (`lib/server/space.ts`), which calls `loadEntry` →
`Space.load`. Because `loadEntry` passes `getSharedThemes()` (§3.1), the hot-added
space's effective map includes the shared set immediately — the picker lists the
three canonical themes with no restart, and resolution falls to a real shared
`amber-default` if nothing is picked. This must be covered by an integration
test (§11).

## 10. Non-goals

- **No operator-controlled `<install-root>/.amber/themes/` override dir.**
  Deferred per the brainstorming decision. The precedence ladder (§4) leaves a
  clean slot for it between per-space and app-bundled if it is ever specced.
- **No boot-time writes into `.amber/`** (no seeding). App-bundled themes are
  read-only and travel with the image.
- **No change to `cache.db`'s role** or to the watcher (`themes/` stays
  unwatched).
- **No new `LoadWarning` codes**; the enum is untouched.
- **No second asset route / no `/themes/_shared/` URL prefix.**
- **No change to the `example-space` fixture's per-space `minimal` theme** — it
  stays, proving per-space discovery and override still work.

## 11. Testing

**Unit**
- `shared-themes.ts`: discovers the three bundled themes from the resolved dir;
  env-var override vs dev-default resolution; empty/missing dir → empty map, no
  throw.
- Effective-map merge: shared ∪ per-space; **per-space wins** on a name collision
  (a per-space `amber-default` copy shadows the shared one, including its
  `path`/`assetBase`). Cover **both** load paths — cold (`load()`/`pureLoad`) and
  cache-hit (`Space.load` hydrate branch) — and assert they produce the same
  effective map.
- `resolveActiveTheme` against the effective map: step 3 now resolves a real
  shared `amber-default`; `space_theme_not_found` only when the name is in
  neither map.
- `describeThemeSource`: shared vs this-space vs inherited; a valid shared pick
  is **not** flagged stale.
- Asset route: serves a shared theme's `theme.css` from `<bundledDir>/<name>`;
  per-space override serves its own; containment guard rejects `../` escape
  against the matched theme's root; built-in (`path === ''`) → 404.

**Integration**
- New-space → `addSpace` → picker lists the three shared themes with no restart;
  resolution falls to a real shared `amber-default`.
- Existing theme/picker tests repointed to the new theme locations.

**Regression / move**
- avp-software still renders `amber-brand` (now resolved from the shared set; its
  `space.toml`/`amber.toml` already name `amber-brand`, so no config change).
- `example-space` still renders/overrides via its per-space `minimal` theme.

## 12. avp-software move — sizing (so it is not surprise churn)
