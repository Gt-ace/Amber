# v0.5 subsystem 6 — theme-picker UI — spec

Status: **agreed design**, ready for a plan session.
Input: the per-space admin tree shipped by subsystems 3, 4, and 5
(`apps/web/src/routes/admin/(authed)/spaces/[slug]/...`,
`requireSpaceAccess` populating `locals.role`, the
`Space.applySpaceConfigChange()` hot-reload seam already wired by
v0.3's per-space theming work atop v0.1's watcher framework), plus the
per-space theme resolution chain shipped in
subsystem 3 P1 (`lib/space/themes.ts` `resolveActiveTheme`,
`lib/space/config.ts` `readSpaceConfig`, the eight-token dark-mode
contract documented in `docs/themes.md`). Subsystem 5's spec named "a
revision to subsystem 3's hot-reload non-goal for the `theme` field"
as subsystem 6's territory; this spec turns that promise into a
buildable design.

The spec answers *what exactly are we building*. It does not write
implementation code and does not revise `CLAUDE.md` — it names the
revisions; the build makes them.

---

## 1. What this subsystem is

The runtime piece that lets an **owner or install-admin** pick which
discovered theme is active for a space, by writing the `theme = "..."`
field to that space's `space.toml`, and have the change take effect on
the public web **without restarting the process**.

Subsystem 6 owns three new things and revises one existing one:

- A new admin route `/admin/spaces/[slug]/theme` (owner-or-install-admin
  only) that hosts the picker form.
- A new `lib/server/space-config-write.ts` module that takes a desired
  final state and updates `space.toml` in canonical form, preserving
  existing routing fields (`host` / `prefix` / `default`) verbatim.
- A small chrome edit on `/admin/spaces/[slug]/+page.svelte` adding a
  "Theme: …" affordance above the page list, gated on the same
  owner-or-install-admin tier.
- A revision to subsystem 5's "**editing** an existing space's
  `space.toml` from the UI" non-goal — subsystem 6 lifts that non-goal
  *for the `theme` field only*. Editing `host` / `prefix` / `default`
  from the UI is still a non-goal (still requires a restart).

This is structurally a *smaller* subsystem than subsystem 5 because the
hot-reload seam already exists. v0.3's per-space theming work already
wired `space_config_change` dispatch into v0.1's watcher framework on
add/change/unlink of `space.toml`
(`apps/web/src/lib/space/watcher.ts:133`), and `Space.applySpaceConfigChange()`
(`apps/web/src/lib/space/space.ts:360`) already re-reads `space.toml`,
re-resolves the theme via `resolveActiveTheme`, and updates
`this.theme` in place. The picker writes the file; the watcher does the
rest. No new registry-mutation API, no equivalent of subsystem 5's
`addSpace`.

### Non-goals

Explicitly out of scope for subsystem 6. Each is either a later
increment, deliberately not Amber's job, or a different subsystem's
territory:

- **Editing `host` / `prefix` / `default` from the UI.** Routing config
  stays hand-edited; subsystem 5's "creation can hot-add routing, edits
  still require restart" remains in force for these fields.
- **Editing the install-level `amber.toml`'s `theme` field.** That's an
  operator decision at the shell. The picker writes only `space.toml`.
- **Preview of a theme before committing.** Operator opens the public
  site in a new tab via the "view your site ↗" link after submit.
- **Installing or uploading new themes.** Theme directories arrive via
  git/scp at the operator's shell; discovery still runs once at cold
  start (per `docs/themes.md`). Adding a UI to upload themes is a much
  larger change to Amber's "filesystem is truth" stance.
- **Editing `theme.toml` from the UI** (renaming themes, changing
  colors, restyling). That's theme-author work, not space-operator
  work.
- **Surfacing a `/members` link on the per-space dashboard.** A pre-
  existing UX hole (members is reachable only via URL today); fixing it
  isn't subsystem 6's scope.
- **A generic per-space "Settings" sub-route shell with theme as one
  tab.** YAGNI. If a second settings-shaped page lands later, refactor
  then. Subsystem 6 ships a direct `/admin/spaces/[slug]/theme` route.
- **Theme thumbnails / screenshots in the picker UI.** Neither shipped
  theme has one; introducing the convention is speculative.

## 2. Surface and routes

One new route. No new top-level admin chrome — the affordance hangs off
the per-space dashboard.

| Group | Route | Purpose |
|---|---|---|
| `(authed)/spaces/[slug]` | `/admin/spaces/[slug]/theme` | The picker. GET renders the form populated with discovered themes + the current selection; POST handles validation + writing `space.toml`. Owner-or-install-admin only. Editors get 403 from the picker `load`; non-members get 404 from the `[slug]/+layout.server.ts` `requireSpaceAccess` call (subsystem 4's deliberate disclosure choice — merges unknown-slug and not-a-member into one 404). |

The route lives under the existing `[slug]/+layout.server.ts`, which
already runs `requireSpaceAccess(event, params.slug)` (subsystem 4), so
`locals.space` is populated and `locals.role` is
`'install-admin' | 'owner' | 'editor'`, and any non-member (or unknown
slug) has already been 404'd at the layout boundary before any child
`load` runs. The picker upgrades the floor to owner with the same
helper `/members` uses — `requireSpaceAccess(event, params.slug, 'owner')`
— which 403s an editor of *this* space. The check happens in `load` so
editors get a 403 page rather than seeing a form they can't submit. The
action repeats the same call — defense in depth against a stale
session, and the helper re-queries the `member` table rather than
trusting the layout-populated `locals.role` (§7).

The route works in **both** single-space and multi-space modes. Unlike
`/admin/new-space` (which 404s in single-space), every loaded space has
a `theme` that can be picked. In single-space mode the only space is
reachable at `/admin/spaces/<the-single-slug>/theme` because the
registry has one entry whose slug is the space directory's basename;
the per-space `[slug]/+layout.server.ts` matches it directly with no
shim involved. (The v0.4-compat admin shims only *redirect into* this
`[slug]` pattern from old flat URLs; they don't enable it, so an
eventual v0.6 shim removal doesn't touch the theme route.)

### Chrome additions

`/admin/spaces/[slug]/+page.svelte` (the per-space dashboard) currently
shows `<h1>Pages</h1>` and a `+ New page` link. It gains one small
block above the page list, visible only when the viewer is owner or
install-admin:

```svelte
<p class="amber-space-meta">
  Theme: <strong>{data.activeThemeName}</strong>
  <a href={resolve(`/admin/spaces/${data.slug}/theme`)}>Change</a>
  {#if data.publicUrl}
    · <a href={data.publicUrl} target="_blank" rel="noopener">view your site ↗</a>
  {/if}
</p>
```

`Change` is an `<a>`, no JavaScript. The block is omitted entirely for
editors — they can't change the theme and don't need to know which one
is active. Showing a "Change" link they can't use would be misleading.

The dashboard `+page.server.ts` `load` gains three fields:

- `activeThemeName: string` — the resolved theme's name (`space.theme.name`,
  which is the resolver's outcome; never null — the chain has a built-in
  floor).
- `publicUrl: string | null` (derived from `space-routing.publicUrlForSpace`
  + `AMBER_PUBLIC_URL`; see §7 — non-null in single-space mode).
- `canPickTheme: boolean` (= `role === 'owner' || role === 'install-admin'`).

The existing `pages` and `slug` fields stay. The dashboard shows only
the resolved *name* — the "where did it resolve from" detail lives on
the picker page (§3), which is where an operator acts on it.

Unlike the `/members` link (§1 non-goal), the dashboard's
`view your site ↗` link reuses `publicUrl` plumbing subsystem 6 already
needs for the picker page (§7) and sits in the chrome block this
subsystem is adding anyway — it's a one-line render of an
already-computed value, not a new affordance built from scratch.

### Picker route load shape

The picker `load` return is a contract; enumerated here so §3 / §6 / §8
read against one definition:

```ts
{
  themes: Map<string, Theme>;          // the discovered set the radio list renders from
  declaredTheme: string | undefined;   // spaceConfig?.theme — drives the "Selected" chip + initial radio
  resolvedThemeName: string;           // space.theme.name — what actually renders right now
  themeSource: 'space-toml' | 'inherited';  // derived; see §3 + the describeThemeSource helper (§7)
  staleThemeName: string | null;       // declaredTheme when it isn't a discovered theme, else null
  publicUrl: string | null;            // same helper as the dashboard
  preservedRouting:                    // load-time snapshot the action round-trips into the write (§4)
    | { host: string } | { prefix: string } | { default: true } | {};
}
```

`declaredTheme`, `preservedRouting`, and the inputs to `themeSource` /
`staleThemeName` all come from one `readSpaceConfig(locals.space.root)`
call in the picker `load`. `resolvedThemeName` is `locals.space.theme.name`.
`themeSource` / `staleThemeName` are computed by the pure
`describeThemeSource` helper (§7) from `declaredTheme` + `themes` +
`resolvedThemeName` — no resolver re-run, no warning re-emission.

### After-submit redirect

After a successful write, the POST handler issues
`redirect(303, '/admin/spaces/<slug>/theme')` — back to the same page
(post-redirect-get). The picker re-renders with the "Selected" chip on
the newly-chosen row and the "Currently rendering" line updated (§3).
SvelteKit's form action conventions make this the obvious
shape. By the time the redirected GET arrives, chokidar has typically
observed the on-disk write and `applySpaceConfigChange` has fired (see
§5); the redirect-target render reflects the new theme.

A subtle alternative — redirecting to `/admin/spaces/<slug>` (the
dashboard) — would feel like "you're done, return to base" but loses
the "ack: the change took effect on this very page" affordance.

### "View your site" link

The picker page shows the public URL the space is reachable at, as a
`target="_blank"` link beside the "Currently rendering" status line
(§3): `Currently rendering: <strong>amber-default</strong> · view your site ↗`.
The URL is derived from the new `publicUrlForSpace` helper added to
`space-routing.ts` (§7). For an admin-only space with no
`host`/`prefix`/`default`, the link is omitted with a one-line helper:
*"This space has no public URL yet — set `host` or `prefix` in
`space.toml` to make it reachable."*

The dashboard chrome block shows the same link (§7).

No backward-compat shims; there's no prior URL.

## 3. The picker form

Single page, single column, plain admin-chrome aesthetic. Matches the
per-space members page rather than introducing a new visual register.
Amber's admin is utilitarian by design.

### Shape

A vertical list of radio-cards, one per choice, in this order:

1. **"Use install default"** — special row. When chosen, the writer
   omits `theme` from `space.toml`. Helper line: *"Falls through to the
   install-level theme set in `amber.toml`, then to `amber-default`."*
2. **`amber-default`** — pinned second (always shown if discovered; it
   is the documented floor).
3. **Other discovered themes** — alphabetical by directory name.

Each row has:

- A radio input on the left, with the whole row as its `<label>` so the
  full row is clickable (accessibility default).
- Theme name (large), then `theme.toml`'s `description` on a second
  line (if present), then a muted third line with `version` +
  `author`. The "Use install default" row uses its helper line in
  place of the metadata block.
- A **Selected** chip aligned right when the row matches the value
  currently *declared* in `space.toml` (the `declaredTheme` load field).
  This is the declared choice, not the resolver outcome — see the
  "Currently rendering" line below for what actually renders.
  - `declaredTheme === "amber-editorial"` → "Selected" chip on the
    `amber-editorial` row.
  - `declaredTheme === undefined` (no override; the chain inherits) →
    "Selected" chip on the **"Use install default"** row, regardless of
    which theme the chain ultimately picked.
- The initially-selected radio is whichever row carries the "Selected"
  chip.

### Currently-rendering detail

Below the radio list, a one-line muted status driven by the load's
`themeSource` + `resolvedThemeName` + `staleThemeName` (§2 picker load
shape). Three cases, deliberately collapsed to what the operator can
act on:

- `themeSource === 'space-toml'` →
  *Currently rendering: **<resolvedThemeName>** — this space's `space.toml`.*
- `themeSource === 'inherited'`, `staleThemeName === null` →
  *Currently rendering: **<resolvedThemeName>** — inherited install default.*
- `themeSource === 'inherited'`, `staleThemeName !== null` →
  *Currently rendering: **<resolvedThemeName>** — fell back after `space.toml` named `<staleThemeName>`, which isn't a discovered theme.*

This lets an operator who picks "Use install default" understand
*which* theme they'll actually get without leaving the page, and
surfaces the "you set `theme = "foo"` but `foo/` was renamed"
situation. Note the deliberate simplification: `themeSource` does **not**
distinguish "inherited from `amber.toml`'s `theme`" from "inherited from
the `amber-default` floor" — that distinction isn't operator-actionable,
and the resolved *name* already tells them what they'll get. This keeps
the derivation a pure comparison in the `load` handler with no resolver
re-run (the `describeThemeSource` helper, §7), rather than extending the
resolver's return contract.

### Stale-theme handling

Edge case: `space.toml` names a theme that's no longer in the
discovered set (a `space_theme_not_found` `LoadWarning` was emitted at
cold start). Two sub-cases:

1. **`amber-default` is still discovered** → the picker shows all
   discovered themes in the list. The stale `declaredTheme` isn't a row
   (it's not in `themes`), so no row carries the "Selected" chip; the
   initially-selected radio falls back to the "Use install default" row.
   A one-line warning sits above the list: *"`space.toml` names
   `<staleThemeName>`, which isn't a discovered theme. Pick one below
   and submit to fix it."*
2. **`amber-default` is also missing** → the resolver fell to the
   built-in floor. Same warning, same list. The user picks something
   concrete.

We do **not** render an extra "stale theme" radio that could be re-
submitted (that would just re-write the broken value). The fix path is
"pick a real theme."

### Empty-themes state

A space with zero discovered themes (no `themes/` directory, or every
theme directory was skipped by discovery) shows the radio list with
only the "Use install default" row. Submitting it writes nothing useful
in the theme sense — but the on-disk effect depends on what's already
in `space.toml`, per §4: if `space.toml` is absent, the submit is a
true no-op; if it exists with only a `theme` field, the file is
deleted; if it exists with routing fields, the file is re-emitted in
canonical form with the `theme` line omitted (which loses comments and
unknown keys — see §8). Helper text above: *"No themes are discovered
under `themes/`. Add a theme directory and restart to pick it here."*
No errors; this is a configuration state, not a user mistake.

### Validation timing

- Theme-name validity is structural — the radio's value comes from a
  known-good set rendered server-side. The action re-checks the
  submitted value against the live discovered set (defense in depth —
  the discovered set is cold-start-fixed, so a divergence only happens
  across a process restart between GET and POST, but checking is
  cheap).
- A submitted theme name not in the discovered set returns the form
  re-rendered with a form-level error (`theme_not_discovered`). No
  disk write.

### Errors

Form-level errors (write failed, permission denied,
`theme_not_discovered`) sit above the submit button, `role="alert"`,
`aria-live="polite"`. There are no per-field errors — the form has
effectively one field. The stale-theme *warning* (a cold-start load
condition, not a submit error) sits above the radio list, per
"Stale-theme handling" above.

### Submit & cancel

A primary **Save** button and a **Cancel** link back to
`/admin/spaces/<slug>`. Cancel is a plain `<a>`, no form
interception.

### Motion

None for the form chrome itself — this is a settings page, used
occasionally. The submit button gets `transform: scale(0.97)` on
`:active` and a spinner during the async POST (matches subsystem 5).
No page-entry animation, no card-reveal animation.

### Why a radio list and not a card grid

`amber-default` and `amber-editorial` ship today; a third community
theme is hypothetical. With 2-3 choices, a single column with full-
width labels is more legible than a grid that wastes horizontal space
on big card frames. The page is server-rendered, no JS gating; a
future redesign can swap layouts without changing the data shape.

## 4. The disk-writing module — `lib/server/space-config-write.ts`

A single exported function:

```ts
export interface SpaceConfigUpdate {
  // The routing fields to write. Mutually exclusive by space.toml
  // semantics — the caller is responsible for setting at most one of
  // host/prefix/default.
  host?: string;
  prefix?: string;
  default?: boolean;
  // The theme to write. undefined = no `theme` line emitted (the
  // resolver chain falls through). string = `theme = "<value>"`.
  theme?: string;
}

export async function writeSpaceConfig(
  spaceRoot: string,
  update: SpaceConfigUpdate
): Promise<
  | { kind: 'ok' }
  | { kind: 'error'; code: WriteErrorCode; detail?: string }
>;
```

`SpaceConfigUpdate` is the **desired final state**, not a diff. The
route action copies the live values of `host` / `prefix` / `default`
from the `load`-time read of `readSpaceConfig` and applies only the
theme change. The writer doesn't know about "preserve" semantics; it
just serializes what it's given. This keeps the writer pure-ish (it
does I/O but doesn't know about the route, the registry, or HTTP) and
intentionally future-friendly without speculatively building hooks
for fields that subsystem 6 doesn't write.

### Inputs (subsystem 6's only callers)

- Pick a discovered theme `"<name>"`:
  `{ ...preservedRouting, theme: '<name>' }`.
- "Use install default": `{ ...preservedRouting }` — `theme` omitted.

`preservedRouting` is `{}` when the live `SpaceConfig` is `null` (no
existing `space.toml`) or has no routing fields set; otherwise it's
whichever of `{ host }`, `{ prefix }`, or `{ default: true }` was
already on disk.

### What it writes

In this order, all inside `<spaceRoot>/`:

1. Build the canonical text. Lines emitted in fixed key order, present
   iff that field is on `update`:
   ```toml
   host = "<host>"
   prefix = "<prefix>"
   default = true
   theme = "<theme>"
   ```
   At most one of `host` / `prefix` / `default` is ever present (caller
   enforces); `theme` is independent. So at most two lines are written.
   Empty trailing newline at end of file.
2. If the resulting text would be empty (no fields on `update` at all
   — e.g. operator picked "Use install default" and the space had no
   routing config), **delete** `space.toml` instead of writing an
   empty file. Both states resolve identically under `readSpaceConfig`,
   but a missing file is the canonical "no overrides" representation
   and keeps the directory tidy. If no `space.toml` existed to begin
   with, this is a no-op (silently ignore `ENOENT` on unlink).
3. Otherwise, write atomically via tmp-then-rename:
   - `writeFile('<spaceRoot>/.space.toml.tmp.<pid>.<counter>', content, { encoding: 'utf8', mode: 0o644 })`
   - `rename('<spaceRoot>/.space.toml.tmp.<pid>.<counter>', '<spaceRoot>/space.toml')`
   - `<counter>` is a module-local `let n = 0; n++` so two concurrent
     writes inside one process don't collide on the tmp name.
   - The tmp filename starts with `.` so it isn't picked up by the
     watcher.

### TOML escaping

The four values that can land in TOML are three strings (`host`,
`prefix`, `theme`) and one literal `true` (`default`). All three string
values are escaped as TOML basic strings via the shared
`lib/server/toml-escape.ts` helper extracted *verbatim* from subsystem
5's `space-create.ts` (`escapeTomlBasic`, lines 37–53). Its exact rule
set: `"` → `\"`, `\` → `\\`, `\n`/`\r`/`\t` → named escapes, other
control chars (code `< 0x20` minus those three, plus `0x7f`) → `\uXXXX`,
every other character verbatim. Because subsystem 6 imports the function
unchanged, the §8 `toml-escape.test.ts` cases (including "newline →
`\n`") carry straight over.

In practice:

- `theme` is a discovered directory name filtered by `discoverThemes`
  (no leading `.` or `_`), so it's almost always plain ASCII. Escaping
  is defense.
- `host` and `prefix` come from `readSpaceConfig` (which trusts string-
  typed fields). They have already been parsed by `smol-toml`. Round-
  tripping a parsed-then-re-emitted basic string through `smol-toml` is
  the only correctness invariant; our escape rules cover it.

No new TOML serializer dependency. ~15 lines inline. Same pattern as
subsystem 5's `createSpace` writer.

### Atomic-write specifics

```ts
const tmp = join(spaceRoot, `.space.toml.tmp.${process.pid}.${tmpCounter()}`);
const final = join(spaceRoot, 'space.toml');
try {
  await writeFile(tmp, content, { encoding: 'utf8', mode: 0o644 });
  await rename(tmp, final);
  return { kind: 'ok' };
} catch (err) {
  await unlink(tmp).catch(() => {});  // best-effort cleanup; swallow
  return { kind: 'error', code: classifyError(err), detail: (err as Error)?.message };
}
```

`classifyError` is a small local helper mapping `EACCES`/`EPERM` →
`'permission_denied'` and everything else → `'write_failed'`, matching
the error-codes table below. The function returns the discriminated
union rather than throwing — same shape as subsystem 5's writer
(`space-create.ts:91–92,109–112`), so the route action branches on
`result.kind` and never has to catch.

POSIX `rename` is atomic on the same filesystem (we're inside one
directory). On a same-disk rename the new inode replaces the old in
one syscall; readers either see the old content or the new content,
never a torn write. This matters because the watcher could be flushing
while we write — without `rename`, a chokidar `change` event mid-write
would fire `applySpaceConfigChange` against a half-written file.

### Rollback

There is no multi-file partial state. Each failing syscall throws
internally and is caught by the `try`/`catch` above, which converts it
to the error union (never propagating a raw throw to the caller):

- `writeFile` throws → no tmp file (or partial tmp); rename never
  runs; final path untouched. `unlink(tmp)` cleans up; if that fails
  (vanishingly rare), the next write's tmp filename uses a different
  counter — no collision.
- `rename` throws → tmp file may exist; `unlink(tmp)` cleans up; final
  path untouched.
- `unlink(final)` (the delete-when-empty path) throws → final path may
  still exist with stale content. The function returns `write_failed`;
  the user re-submits.

### Error codes (`WriteErrorCode`)

| Code | Trigger | UI surface |
|---|---|---|
| `permission_denied` | `EACCES`/`EPERM` from `writeFile` / `rename` / `unlink`. | Above submit: "Amber doesn't have write access to `<spaceRoot>`. Fix the directory permissions and try again." |
| `write_failed` | Any other I/O error. Detail carries `err.code` + `err.syscall` for log surfacing. | Above submit: "Couldn't update `space.toml`. Check the server logs for details." |

These are distinct from subsystem 5's `space-create.ts` error codes
(`dir_already_exists`, etc., none of which apply here). The two writers
share only the TOML-escape helper.

### Why a separate module from `space-create.ts`

Three reasons. (1) Different write contracts — `createSpace` writes a
directory tree, refuses to clobber; `writeSpaceConfig` updates a single
file, expects the directory to exist. (2) Different rollback surfaces —
`createSpace` rolls back a whole directory; `writeSpaceConfig` writes
one file atomically via tmp-and-rename. (3) Different inputs —
`createSpace` takes a routing-kind enum; `writeSpaceConfig` takes a
flat field set. Sharing only the TOML escape function (~10 lines) is
the right size of overlap.

### What this module deliberately doesn't do

- **Doesn't read the current `space.toml`.** The caller passes the
  desired final state. The writer is pure I/O + serialization.
- **Doesn't validate `theme` against the discovered theme set.** The
  action does that before calling.
- **Doesn't trigger the watcher event.** The on-disk write makes
  chokidar fire `change` (or `add` if `space.toml` didn't exist),
  which the watcher dispatches as `space_config_change`, which the
  existing `Space.applySpaceConfigChange()` handles. The writer just
  writes; the watcher does the rest. This is the load-bearing reuse of
  v0.1 infrastructure that makes subsystem 6 small.
- **Doesn't bypass the watcher with a direct
  `space.apply({ type: 'space_config_change' })`.** The synchronous
  apply would either run twice (once direct, once via chokidar — apply
  is idempotent, but it muddies the log trail) or imply the watcher is
  unreliable. The watcher-only path is the consistent seam to use.

## 5. Watcher coupling

The smallest section, because the work is already done. Documenting it
for spec completeness and to flag the test-shape implication.

### The existing chain

`apps/web/src/lib/space/watcher.ts:133` already dispatches:

```ts
if (rel === 'space.toml') {
  this.dispatch({ type: 'space_config_change' });
}
```

…on add / change / unlink of `space.toml`. `Space.applySpaceConfigChange()`
(`space.ts:360`) then:

1. Re-reads `space.toml` via `readSpaceConfig(this.root)`.
2. Calls `resolveActiveTheme(this.themes, this.manifest, config, log)`.
3. Assigns `this.theme = resolved.theme`.
4. Updates `this.spaceConfigWarnings` (which feeds into the surfaced
   `space.warnings` set on the next `recomputeWarnings`).

Consumers that read `space.theme` on each request (the public page
handler, the sitemap, the layout's chrome injection) see the new theme
on their next request. Templates are read fresh per request
(`readTemplate` does `readFileSync` at call time), so the new theme's
`chrome.html` / `page.html` / `error.html` are picked up automatically.

### What subsystem 6 needs to do here

Nothing. The writer writes; the watcher fires; the apply path updates
`space.theme`; the next request renders with the new theme.

### Test-shape implication

The route integration test for the happy path can't just submit-and-
assert: chokidar's events fire on the OS event-loop pump, not
synchronously with the write. The test needs to wait for the watcher
to observe the file change before asserting `space.theme.name`.
Pattern (matching v0.1 watcher tests): poll `locals.space.theme.name`
every 10ms up to a 500ms cap. Fast in practice (chokidar typically
fires within 20–80ms on Linux), and fails loudly if the watcher chain
regresses.

The hooked-watcher-flush alternative (register a listener for the
specific `space_config_change` event, await its fire, then assert) is
more precise but couples the test to the watcher's internal API.
Polling asserts the *observable* end state, which is what consumers
actually see.

### Acceptance polling in the action handler

The route action does **not** poll the watcher. It writes the file,
then issues a 303 redirect to `/admin/spaces/<slug>/theme`. By the
time the redirected GET request arrives (a network round-trip later),
chokidar will have observed the write and the apply path will have
completed. If the operator's browser is unusually fast or the OS
unusually slow, the redirected GET might see the *previous* theme for
a few milliseconds — the next reload (or the operator's "view your
site ↗" click) sees the new one. We accept this small race as good
enough: the file is on disk, the change is durable, and the in-memory
state self-corrects within a few hundred milliseconds.

This is the same race subsystem 1's editor accepts after a
`/admin/edit` save: the file is the source of truth; the in-memory
index converges asynchronously via the watcher.

### What we don't gain by bypassing the watcher

A tempting design: have the action handler call
`space.apply({ type: 'space_config_change' })` synchronously *before*
the redirect, to guarantee the redirected GET sees the new theme.
Reasons not to:

1. The watcher will *also* fire `space_config_change` for the same on-
   disk event. `applySpaceConfigChange` is idempotent (it re-reads and
   re-resolves; running twice produces the same final state), but
   emitting the same event twice muddies the log trail.
2. The synchronous path silently asserts "the in-process registry is
   the only consumer that needs to know" — true today but a bad
   pattern to establish. The watcher *is* the seam; using it
   consistently keeps the seam load-bearing and tested.
3. If a future deployment runs a multi-process Bun setup (cluster, or
   Bun's eventual worker model), only the *writing* process sees the
   synchronous apply; other processes find out via their own watchers.
   The watcher-only path scales to that future for free.

### Single-space-mode parity

The watcher runs in all modes. Single-space mode has one `Space`
instance, one watcher, one `applySpaceConfigChange` path — identical
to multi-space. The picker route works identically in single-space
(with the existing v0.4-compat admin shims resolving the single
`[slug]`).

## 6. Validation rules and error codes

Validation runs server-side at form submission, before
`space-config-write.ts` touches the disk. Two failure surfaces —
**input-shape errors** (rejected at the form, recoverable, user re-
submits) and **write-time errors** (from §4 — surfaced as form-level
errors above the submit button).

### Form-rejection codes

Returned to the form as a re-render with an error message above the
submit button (the form has effectively one field; per-field errors
aren't useful here).

| Code | Trigger | UI surface |
|---|---|---|
| `theme_not_discovered` | Submitted theme value isn't `""` (the "use install default" sentinel) and isn't in the live discovered theme set. | Above submit: "`<value>` isn't a discovered theme. The list below shows what's available." Re-renders the picker (refreshed against the current discovery set). |

That's the only form-rejection code. Theme name *shape* is
structurally constrained by `discoverThemes` (directory names not
starting with `.` or `_`, must parse `theme.toml`, must have all three
required templates), so we don't separately validate the string
format. Empty string is a valid submission (it means "use install
default") and is the only path through that results in `theme` being
absent from `update`.

The validator is a pure function returning a tagged union. (Subsystem
5's `validateCreateInput` returns a different shape —
`{ valid: ValidatedCreateInput | null; errors: FieldError[] }`,
`space-create-validate.ts:66–69` — because it validates a multi-field
form; a tagged union is the better fit for this one-field form, so the
analogy is "same pure-function discipline," not "same return type"):

```ts
export function validateThemePick(
  submitted: string,             // the radio value: '' or '<theme-name>'
  discovered: Map<string, Theme> // the live themes
):
  | { kind: 'ok'; theme: string | undefined }
  | { kind: 'error'; code: 'theme_not_discovered'; submitted: string };
```

`{ kind: 'ok'; theme: undefined }` is the "use install default" case;
`{ kind: 'ok'; theme: '<name>' }` is the picked-a-real-theme case. The
route action calls this, then derives `SpaceConfigUpdate` from the
result + preserved routing fields.

### Write-time codes

The two `writeSpaceConfig` error returns from §4 — surface above the
submit button:

- `permission_denied` — same copy as subsystem 5 ("Amber doesn't have
  write access to `<spaceRoot>`…").
- `write_failed` — generic write-failed copy with a pointer to logs.

### Load-time warnings (defense-in-depth)

The watcher's reaction to the on-disk write can in principle produce
`LoadWarning`s. None of them block the redirect; all are logged with
`subsystem: "resolver"` (the existing pattern) and re-emitted with
`subsystem: "theme-picker"` for chronological log trace:

| Warning | When it fires after a picker write | Expected? |
|---|---|---|
| `space_config_invalid` | Our hand-written TOML failed to parse. | No — defense via §8's fuzz test. If this ever fires, it's a code bug. |
| `space_theme_not_found` | The chosen theme's directory was removed between validate-and-write and watcher-re-read. | No — themes are discovered once at cold start; can only happen via an out-of-band restart between the two reads, which is essentially never. |

If either fires, the operator's experience is: the redirect lands, the
"Currently rendering" status line shows the resolver chain falling
through unexpectedly, and a one-line warning appears above the picker ("Heads
up: a `space_theme_not_found` warning fired after the write — check
the logs"). They re-pick, the warning clears.

### No new `LoadWarning` codes

Subsystem 6 reuses subsystem 3's vocabulary unchanged. The CLAUDE.md
`LoadWarning` enum and trigger list stay byte-for-byte identical — the
"every code has a defined trigger" invariant holds.

### Race surface explicitly enumerated

The validator reads `locals.space.themes` (the live discovered set)
and the submitted value. Between read and re-read by the watcher, the
discovered set cannot change (discovery is cold-start-only per
`docs/themes.md`). So there's no "validator says theme X exists,
watcher says theme X is gone" race within a single process. The only
race-able state is *file content* on disk, and we make that race
benign:

- **Two tabs racing two writes**: last-write-wins on disk; the watcher
  fires both events; the second `applySpaceConfigChange` wins. The
  losing tab sees a stale active-theme on its redirect-target page; no
  unsaved-state-divergence notice is shown (YAGNI).
- **Operator hand-edits `space.toml` mid-pick**: the picker writes its
  desired final state; hand-edits to non-`theme` fields are clobbered
  (since `update` was built from the load-time snapshot of
  `host`/`prefix`/`default`). Same trade-off as subsystem 5's "trust
  the validator" stance, and explicitly called out in the writer's
  doc-comment.

The race surface is small enough that we don't add optimistic-
concurrency tokens. If operators report problems, that's a v0.7
conversation.

## 7. Permission gating, mode handling, chrome integration

Three small things bundled because each is a few lines.

### Permission gate

The route's `+page.server.ts` `load` and the form action both call the
same helper `/members` uses, with the owner floor:

```ts
import { requireSpaceAccess } from '$lib/server/permissions';

export const load = (event) => {
  requireSpaceAccess(event, event.params.slug, 'owner'); // 403s an editor; install-admin bypasses
  // … build the picker load shape …
};

export const actions = {
  default: async (event) => {
    requireSpaceAccess(event, event.params.slug, 'owner');
    // … validate → write → redirect …
  },
};
```

`requireSpaceAccess(event, slug, 'owner')` is the install-admin-or-owner
gate `/members` already uses on every load and action
(`members/+page.server.ts:34,42`). The helper supports `minimumRole='owner'`
with the install-admin bypass at `permissions.ts:115`, 403s a member
whose role is below the floor at `permissions.ts:128–129`, and — because
it re-runs `resolveAccess`, which re-queries the `member` table
(`permissions.ts:96–103`) — gives stronger defense-in-depth than
trusting the layout-populated `locals.role`. Same helper, same trust
tier, default error message.

The check happens in `load` so editors get a 403 page rather than
seeing a form they can't submit. The action repeats it for defense in
depth against a stale session or an unusual request shape. The layers
above this route have already narrowed who reaches it: the `(authed)`
layout redirects anonymous users to `/admin/login`, and the
`[slug]/+layout.server.ts` `requireSpaceAccess(event, params.slug)`
(no `minimumRole`) has already 404'd any non-member or unknown slug
(`permissions.ts:121–124`, subsystem 4's disclosure choice). So the
only case this `load` gate adds is 403ing a signed-in **editor of this
space** who types the URL directly — a member of a *different* space is
already 404'd at the layout boundary.

### Mode handling

The picker works in **both** discovery modes — unlike `/admin/new-
space`, which 404s in single-space mode. Every loaded space has a
theme that can be picked. No `getDiscoveryMode()` check; no

space settings page for theme.

In single-space mode, the route is reachable at
`/admin/spaces/<the-single-slug>/theme` because the single-space
registry has one entry whose slug is the space directory's basename;
the per-space `[slug]/+layout.server.ts` matches it directly with no
shim involved. (The v0.4-compat admin shims only *redirect into* this
`[slug]` pattern from the old flat `/admin/edit`, `/admin/new`,
`/admin/api/page` URLs — they don't enable it, so the eventual v0.6
shim removal doesn't affect the theme route.) The existing per-space
dashboard, edit, and members routes already work in single-space mode
the same way.

### Two small pure helpers

**`publicUrlForSpace` — extend `lib/server/space-routing.ts`.** The
picker route and the dashboard both need to derive a space's public URL
from its routing config + `AMBER_PUBLIC_URL`. Add one small exported
function to the existing `space-routing.ts`:

```ts
/**
 * Derive a space's public URL from its space.toml routing fields, the
 * install's AMBER_PUBLIC_URL, and the discovery mode. Returns null only
 * in multi-space mode when the space has no public route (no host /
 * prefix / default — an "admin-only" space). In single-space mode the
 * lone space always serves at AMBER_PUBLIC_URL/, regardless of routing
 * fields (hooks.server.ts:74–99 makes it the resolver default and
 * ignores its routing fields), so the helper returns `<publicUrl>/`.
 */
export function publicUrlForSpace(
  config: { host?: string; prefix?: string; default?: boolean } | null,
  publicUrl: string,  // AMBER_PUBLIC_URL — supplies the scheme for `host`
                      //   cases and the base origin for `prefix`/`default`
  mode: 'single-space' | 'multi-space'
): string | null;
```

Logic:

- `mode === 'single-space'` → `<publicUrl>/` (short-circuit; routing
  fields ignored, matching boot behaviour).
- Otherwise (multi-space), by routing field:
  - `host = X` → `<scheme>://X/`, scheme taken from `publicUrl`
    (`https://` in prod, `http://` against localhost in dev).
  - `prefix = /Y` → `<publicUrl>/Y/` (preserve `publicUrl`'s scheme +
    host).
  - `default = true` → `<publicUrl>/`.
  - none of the above → `null` (admin-only space).

The two call sites pass `getDiscoveryMode()` (the getter subsystem 5
added). Subsystem 5's `/admin/new-space` form computes a similar string
inline from form values (`data.adminScheme`/`data.adminHost` + the
typed routing input); subsystem 6 doesn't refactor subsystem 5 — it
adds the helper for its own use. A follow-up could DRY subsystem 5
later if the operator-shape proves identical.

**`describeThemeSource` — new pure helper in `lib/space/themes.ts`.**
The picker `load` needs to tell the operator whether the rendering
theme comes from this space's `space.toml` or is inherited, and whether
a declared theme went stale. This is a pure comparison over data the
`load` already holds — *not* a re-run of `resolveActiveTheme` (which
would re-emit warnings). Lives beside the resolver it mirrors:

```ts
/**
 * Classify where the active theme came from, for the picker's
 * "Currently rendering" line. Pure; emits no warnings, does not re-run
 * resolution. Deliberately does NOT distinguish amber.toml-default from
 * the amber-default floor — both read as 'inherited'; the resolved name
 * already tells the operator what they get.
 */
export function describeThemeSource(
  declaredTheme: string | undefined,   // spaceConfig?.theme
  discovered: Map<string, Theme>,
  resolvedThemeName: string            // space.theme.name
): { source: 'space-toml' | 'inherited'; staleThemeName: string | null };
```

Logic: if `declaredTheme` is set and `discovered.has(declaredTheme)` →
`{ source: 'space-toml', staleThemeName: null }`. If `declaredTheme` is
set but not discovered → `{ source: 'inherited', staleThemeName: declaredTheme }`
(declared but fell through). If `declaredTheme` is undefined →
`{ source: 'inherited', staleThemeName: null }`. The resolver's return
contract (`{ theme, warnings }`) is **unchanged** — this is the
deliberate alternative to widening it (§3).

## 8. Testing

Three layers, matching subsystem 5's discipline.

### Unit tests

**`space-config-write.test.ts`** — exercises the writer against a real
`os.tmpdir()` directory. Covers:

- **Happy paths**, one per shape of `SpaceConfigUpdate`:
  - `{ theme: 'amber-default' }` against a directory with no prior
    `space.toml` → file created with exactly
    `theme = "amber-default"\n`.
  - `{ theme: 'amber-editorial' }` against a directory whose
    `space.toml` had `host = "x.example.com"` → file rewritten with
    two lines, `host` first, `theme` second.
  - `{ prefix: '/notes', theme: 'amber-editorial' }` (preserved prefix
    + new theme) → two lines, `prefix` first, `theme` second.
  - `{ default: true }` (existing theme cleared, preserved default) →
    one line, `default = true`.
  - `{}` (empty update — operator picked "Use install default" on a
    space whose `space.toml` had only `theme`) → `space.toml` is
    *deleted* on disk.
  - `{}` against a directory with no prior `space.toml` → no-op (no
    file written, no error).
- **Canonical ordering** explicitly asserted: write
  `{ default: true, theme: 'x' }` → bytes are `default = true\ntheme = "x"\n`,
  not the input-object key order.
- **Comment / unknown-key loss** explicitly asserted: pre-seed
  `space.toml` with `# comment\nhost = "x"\nfoo = "bar"\n`, write
  `{ host: 'x', theme: 'y' }`, assert resulting bytes contain neither
  `# comment` nor `foo = `. This is the canonical-re-serialize trade-
  off (§4), captured in code so a future reviewer can't accidentally
  "fix" it.
- **TOML escaping** (defense — these inputs can't actually arrive via
  the form): `theme: 'has"quote'` → escaped correctly, round-trips
  through `smol-toml.parse` cleanly. Fuzz set: ~200 strings (Unicode,
  control chars, emoji, RTL, zero-width joiners) round-trip through
  write → `readSpaceConfig` → read-back equals input.
- **Atomicity — tmp-then-rename**: install a `writeFile` stub that
  succeeds on the tmp path but the `rename` stub throws → assert
  `space.toml` retains its prior content, the tmp file is gone, and
  the function returns `{ kind: 'error', code: 'write_failed' }`.
- **Atomicity — tmp leak on writeFile failure**: stub `writeFile` to
  throw → assert no tmp file lingers (the `unlink(tmp).catch(() => {})`
  ran).
- **`permission_denied` vs `write_failed`** distinguished by errno
  (synthesize an `EACCES` error and an `EIO` error, assert different
  return codes).
- **Concurrent-write race** (defense — not a contract assertion): two
  `writeSpaceConfig` calls fire in parallel, each writes a different
  `theme`. Asserts: final on-disk content matches one of the two
  writes (last-rename-wins), no torn write, no leftover tmp files.

## 10. Telemetry and logging

One new subsystem tag on the existing `lib/server/logger.ts` child-
logger pattern: `subsystem: "theme-picker"`. All structured (pino's
default), no new sinks.

- **info, pick-attempted** — `{ slug, themeSubmitted, actor: user.id }`.
  Logged at the start of the action handler, after the permission gate,
  before the validator runs. `themeSubmitted` is the *literal* radio
  value: `null` for the empty-string "use install default" submission
  (logged as `null`, not `""`, so a JSON-aware log reader treats it as
  an explicit "unset" rather than an empty field), or the submitted
  theme name as a string.
- **info, pick-succeeded** —
  `{ slug, themeWritten, prevTheme, durationMs }`. Logged after the
  writer returns OK and before the redirect. `themeWritten` is `null`
  for "use install default" or the written theme name. `prevTheme` is
  the theme value on disk before the write (read from the `load`-time
  `space.toml` snapshot — also `null` for fallthrough). `durationMs`
  is wall-clock from action entry to redirect.
- **warn, pick-rejected** — `{ slug, code, themeSubmitted, actor }`
  where `code` is one of the form-rejection codes from §6 (just
  `theme_not_discovered`). Helps operators diagnose "owner X tried to
  pick theme Y and it bounced" without dumping form bodies.
- **error, pick-failed** —
  `{ slug, code, errno, syscall, themeSubmitted }` for
  `permission_denied` / `write_failed` from the writer. Error-level
  because these need operator attention (filesystem permissions, disk
  full, watch holding a stale tmp).
- **warn, post-write-warning** — pass-through of any `LoadWarning`
  emitted by `applySpaceConfigChange` after the picker's write,
  tagged with `{ slug, code, themeWritten }`. The resolver already
  logs these at warn level via subsystem 3's pattern; we re-emit
  with the `theme-picker` subsystem tag so a chronological log read
  makes the chain "operator picked → wrote → watcher fired →
  warning" visible. §6 documents these as essentially impossible in
  practice; this is defensive logging that gives a fast diagnostic
  if they ever fire.

The picker page's `load` and the dashboard's `load` don't log — reads
of the registry and discovered themes are silent, matching every
other read path.

No new metrics. Amber doesn't have a metrics pipeline.

No log of the raw HTTP form body. Slug is the actionable identifier;
`themeSubmitted` is the one user-controlled value worth capturing for
diagnosis.

## 11. Risks and open questions

### Risk: watcher event timing vs. the post-redirect GET

The action returns a 303 to `/admin/spaces/<slug>/theme`. By the time
the redirected GET arrives, chokidar should have observed the on-disk
write and `applySpaceConfigChange` should have completed. In practice
this is 20–80ms on Linux, 50–200ms on WSL2 — well within a browser's
redirect round-trip. The risk is a pathologically slow watcher (locked
filesystem, sleep-throttled VM): the redirected GET sees the
*previous* theme for one render. Mitigation: the integration test
polls up to 500ms; if the test ever flakes, the polling bound or the
test fixture is wrong, not the design. In production a stale-by-one-
request redirect is benign — the operator's next interaction shows
the new theme.

### Risk: chokidar misses the rename

Tmp-then-rename should fire chokidar's `change` (or `add` if
`space.toml` didn't exist) for the destination path. On some Linux
filesystems with inotify quirks, `rename` can produce a
`delete + create` pair. The watcher's switch in `watcher.ts:133` only
catches `rel === 'space.toml'` — both `delete` and `create` for that
exact path dispatch `space_config_change`, so both halves of the rare
split-event path produce the same dispatch. Defense: e2e test asserts
the public site reflects the new theme after the picker submits; if
chokidar silently swallows the event on a given platform, this catches
it. Fallback: subsystem 6's writer could `space.apply`-notify the in-
process `Space` directly (the §5 "what we don't gain by bypassing the
watcher" path), kept in reserve.

### Risk: lost comments / unknown keys on first theme change

§4 documents this as the explicit cost of canonical re-serialize. An
operator who hand-edited `space.toml` with comments loses them the
first time they touch the picker. Mitigation: the §3 picker page
doesn't currently surface a warning ("option A" from the brainstorming
dialogue, not option C). If operators complain after first ship, the
fix is to add a one-line notice ("This will rewrite space.toml in
canonical form") above the submit button — three lines of code, no
design change.

### Risk: two-tabs race / out-of-band edits

Two owners in two tabs each picking a different theme produces last-
write-wins on disk; the watcher's idempotent apply settles to
whichever rename completed last. No corruption, no UI lockup, but the
losing tab shows a stale "Active" badge after redirect until a manual
reload. We don't add optimistic-concurrency tokens (YAGNI). Similarly,
an operator hand-editing `space.toml` between the picker's `load` and
`action` has their hand-edits to the routing fields clobbered (the
action wrote the *load-time* snapshot of host/prefix/default plus the
new theme). Same trade-off as subsystem 5's "trust the validator"
stance, documented in the writer's doc-comment.

### Risk: hand-written TOML escape accepts bad input

§4's escape rules — `"`, `\`, the named `\n`/`\r`/`\t`, and `\uXXXX`
for the remaining control chars — cover the cases that matter for TOML
basic strings (they're imported verbatim from subsystem 5's shipped
`escapeTomlBasic`). The fuzz test in §8 (~200 strings:
Unicode, control chars, emoji, RTL, zero-width joiners) round-trips
through write → `smol-toml.parse` → read-back. If a category fails,
the fix is escape-rule expansion in `toml-escape.ts`; if escape isn't
enough, the discovered theme set already constrains the only string
that can land there. The host/prefix values come from
`readSpaceConfig`'s already-parsed strings, which round-trip cleanly
through `smol-toml`.

### Risk: theme directory removed at runtime by an out-of-band restart

Themes are discovered once at cold start (`docs/themes.md`). Within a
single process run, the discovered set is fixed. If an operator
restarts the server with a theme directory renamed/removed and a
space's `space.toml` still references it, the picker `load`
correctly shows the stale-theme warning (§3), the resolver chain
falls through, the picker can fix it. No runtime race exists; the
cross-restart case is handled by UI.

### Open question: ship the "this will rewrite in canonical form" notice in v1?

§3 currently doesn't surface a warning when an existing `space.toml`
has comments or unknown keys. Brainstorming picked "silent canonical
re-serialize", not "canonical + UI notice". **Lean: defer the
notice.** Worth confirming during spec review.

### Open question: theme thumbnails / preview images

§3 shows name + version + author + description for each theme — no
thumbnail. Adding a thumbnail would require a new convention (a
`theme.png` at theme-root? a `screenshot` field in `theme.toml`?).
Neither shipped theme has one. **Lean: defer to a future PR if a
community-themes contribution surfaces the need.** Not subsystem 6's
job.

### Open question: a generic per-space "Settings" sub-route shell

Subsystem 6 ships `/admin/spaces/[slug]/theme` as a direct route. A
hypothetical near-future need might want a `.../settings` shell with
theme as one tab among several. **Lean: don't pre-build the shell.**
YAGNI. If a second settings-shaped page lands, refactor `/theme`
into a `/settings/theme` sibling.

### Open question: should the picker offer "view current site" too?

The dashboard chrome already shows a "view your site ↗" link beside
the active-theme summary. The picker page shows the same link in its
own status block (§3). **Lean: keep the picker's link.** Three lines,
no design change; symmetric with the dashboard.

### Open question: v0.5 release closure

After subsystem 6 lands, all six v0.5 authoring-layer subsystems
have shipped. Whether to immediately declare v0.5 as a release
milestone (retrospective, version tag, README sweep, possibly a
`v0.6` direction note) is a separate decision from subsystem 6's
build commit. **Lean: subsystem 6 does not make the call.** The
build's final step revises CLAUDE.md per §9 and stops.
