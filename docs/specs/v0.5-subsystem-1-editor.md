# v0.5 subsystem 1 — the editor — spec

Status: **agreed design**, ready for a plan session.
Input: the round-trip spike (`spike/editor-roundtrip`, `SPIKE_NOTES.md`,
verdict GO). This spec turns that proof into a buildable design for
subsystem 1 of the v0.5 authoring layer: an authenticated in-browser
WYSIWYG editor that writes plain markdown to disk.

The spike answered *is it possible*. This spec answers *what exactly are
we building*. It does not write implementation code and does not revise
`CLAUDE.md` — it names the revisions; the build makes them.

---

## 1. What this subsystem is

A small authenticated admin surface in the existing SvelteKit app that
lets a person edit and create the space's markdown pages from a browser,
without a terminal or git. It changes how files are *written*, not what
is *stored*: content stays plain `.md` on disk, frontmatter stays YAML,
and every other editor — and git — keeps working on the same files.

The editor is "just another writer to the filesystem." The existing
watcher → loader → render pipeline picks up its writes unchanged; the
spike proved this end to end.

### Non-goals

Explicitly out of scope for subsystem 1. Each is either a later
subsystem, a later increment, or deliberately not Amber's job:

- The editor *build* itself — this document is the spec; the build is a
  separate cycle after a plan session.
- Subsystems 2–6 (auth, multi-space routing, permissions, space
  creation, theme picker). This spec names their seams and builds none
  of them.
- Creating new **directories**. v1 new-page creation places pages into
  directories that already exist (see §8).
- Editing `slug`, and anything redirect-related. URL changes are a
  separate concern.
- Asset / image upload and media management. The loader already does not
  index assets; the editor does not either.
- Multi-user, invited authors, per-space permissions — subsystem 4.
- Autosave, version history, draft snapshots. Git is the history.
- Real-time collaboration.
- Conflict *merge*. Concurrency is detect-only (§7); the user chooses
  reload or overwrite.
- A preview pane. The public route already renders the page — it is the
  preview.
- Styling and polish beyond minimal admin chrome plus Crepe's bundled
  frame theme.

## 2. Surface and routes

All editor routes live under `/admin`. They are reached two ways
("both", by decision):

| Route | Purpose |
|---|---|
| `/admin` | Admin index: list of the space's pages (drafts marked), a "New page" link. |
| `/admin/edit/[...path]` | The editor for one page. `[...path]` is the page's public URL path. |
| `/admin/new` | New-page form. |
| `/admin/api/page/[...path]` | `PUT` save endpoint (§5). Not user-visible. |

**Page-list index** (`/admin`) enumerates `Space.pages` — drafts
included, drafts visibly marked (the loader produces; the admin index is
a consumer that *shows* drafts, unlike the public nav which hides them).

**Inline edit affordance.** Rendered public pages show an "Edit this
page" link **only when the request is authenticated**. The link is a
plain `<a href="/admin/edit/...">` emitted server-side after the auth
check — no page content crosses to the client as data on the public
route. The public render path stays server-only and unchanged.

**Path resolution.** `[...path]` is the page's public URL (e.g.
`/admin/edit/about` edits the page served at `/about`). The editor's
`+page.server.ts` resolves URL → source file through the `Space` index,
which already owns that mapping. A path that resolves to no page is a
404; a path that resolves to a draft is editable (drafts are in the
index by design).

## 3. The frontmatter boundary

The spike's verbatim-split strategy is kept. The file is split
server-side with the loader's existing regex
(`splitFrontmatter` in `lib/space/load.ts`) into a verbatim frontmatter
block and a body. **The prose editor only ever touches the body.**

Frontmatter editing is a **separate UI surface**: a side panel next to
the editor. Editable fields in v1 are exactly:

- `title`
- `draft` (boolean)
- `date`

`slug` is deliberately not editable (URL changes are a redirect concern,
out of scope). Any other keys in the file — `updated`, `description`,
`auto_index`, redirects, anything theme- or future-facing — are the
loader's `extra` bucket and **always pass through untouched**.

### Recombination on save

The save (§5) carries the body always, and the frontmatter only when the
panel was edited:

- **Frontmatter panel untouched** → the save omits `frontmatter`. The
  server re-prepends the verbatim frontmatter block byte-for-byte. The
  YAML is structurally unchanged — comments, key order, indentation all
  preserved. This is the common case.
- **Frontmatter panel edited** → the save includes `frontmatter`. The
  server re-reads the file from disk, parses its frontmatter, applies the
  three editable known fields over the parsed result (preserving all
  `extra` keys), and re-serializes the YAML block. This **reformats** the
  YAML — hand-written comments and key order are not preserved. That is
  accepted and documented: editing frontmatter through the panel is
  explicitly a "rewrite the YAML" action; not opening the panel leaves
  the block verbatim.

### Unparseable frontmatter

The loader admits pages whose YAML frontmatter fails to parse — it keeps
them in the index with empty `frontmatter` and a `frontmatter_parse_error`
warning. Such a page can therefore be opened in the editor. Behavior:

- The body editor works normally — the body split is purely structural
  and does not depend on the YAML parsing.
- A body-only save (no `frontmatter` in the request) re-prepends the
  frontmatter block **verbatim**, so the malformed YAML is preserved
  byte-for-byte. Nothing is lost.
- The **frontmatter side panel is disabled / read-only**, with a message:
  the YAML cannot be parsed, so fix it directly in the file to re-enable
  the panel. The panel never silently drops the keys it cannot read, and
  the save endpoint rejects a request that carries `frontmatter` for a
  page whose on-disk YAML does not parse (`422`) rather than overwriting
  the unparseable block.

This keeps "filesystem is truth": the editor never destroys frontmatter
it could not understand.

### Body-edge whitespace

The recombine step guarantees **exactly one `\n`** between the closing
`---` of the frontmatter and the first byte of the body: leading newlines
are stripped from the editor body and exactly one is inserted. This
removes the spike's only diff-on-a-no-op-save (Crepe trims the leading
blank line; the server now makes the result deterministic regardless).

## 4. Round-trip normalization — accepted behavior

The spike proved the round-trip is a *formatter*, not a data-loss
machine: idempotent, no loss, every construct survives as clean markdown.
The cosmetic normalizations (bullet `-`→`*`, table-cell padding, hard
break `  `→`\`) are accepted as documented formatter output.

The one normalization with a rendering effect is **tight lists → loose
lists** (loose lists wrap each `<li>` in `<p>`, adding vertical spacing).
Decision: **accept it as documented formatter behavior; do not tune
remark-stringify.** Justification:

- The spike proved it consistent and idempotent — it reaches a fixed
  point on first save, it does not progressively degrade.
- Tuning remark-stringify to preserve tightness fights Crepe's internal
  ProseMirror document model and is fragile to upstream changes.
- The output is still valid, portable, human-readable markdown.
- The rendered difference is a small spacing change a theme can flatten
  in CSS if it cares.
- It is reversible: a later increment can revisit serializer tuning.
  Accepting now is not a lock-in.

The spec records this so the build documents it for users rather than
treating it as a bug.

## 5. Save transport

A dedicated endpoint, not a form action (form actions are FormData-shaped
by convention, and multipart is ruled out — see below):

```
PUT /admin/api/page/[...path]
Content-Type: application/json
If-Match: <sha256 hex of the whole file as read at editor load>

{ "body": "<markdown body>", "frontmatter": { "title": "...", "draft": false, "date": "..." } }
```

- **Raw JSON request body, never `multipart/form-data`.** The spike
  proved multipart normalizes field-value newlines to CRLF (RFC 2046),
  producing a mixed-ending file. JSON round-trips `\n` faithfully and
  `crepe.getMarkdown()` already emits clean `\n`. JSON is chosen over
  raw `text/plain` because the save must carry the body *and* optionally
  the structured frontmatter in one request.
- `frontmatter` is **omitted** when the panel was not edited (§3).
- The `If-Match` header carries the SHA-256 of the entire file captured
  when the editor loaded it. Concurrency handling in §7.

**Server save sequence:**

1. Resolve `[...path]` → source file via the `Space` index.
2. Read the current file; compute its SHA-256.
3. If `If-Match` is present and does not match → respond `409` (§7).
4. Split the current file with `splitFrontmatter`.
5. Recombine (§3): verbatim FM block if `frontmatter` omitted; else
   re-serialized FM merged over the on-disk `extra` keys. Enforce the
   one-`\n` body-edge rule.
6. Write the file.
7. Respond `200` with the new file's SHA-256 so the editor can keep
   editing without a reload.

The write lands on disk; the existing watcher → `Space.apply()` → render
pipeline picks it up. The editor does not touch the cache or the index
directly — filesystem is truth, exactly as the spike demonstrated.

## 6. New-page creation

`/admin/new` is a server-rendered form. Inputs:

- **Directory** — a picker of directories that already exist in the
  content tree. No free-text path; no new-directory creation in v1.
- **Filename** — the `.md` basename (`.md` appended if the user omits
  it).
- **Title** — seeds the `title` frontmatter field.
- **Draft** — a checkbox; seeds `draft: true` when checked.

**Validation**, on the server, before any write:

- Reserved prefixes and names are rejected: `_*` and `.*` anywhere,
  `amber.toml`, `themes/`, `.amber/`.
- The target file must not already exist.
- The derived URL must not collide with an existing page (`duplicate_url`
  would otherwise be the result — reject up front instead).
- `index.md` is allowed as a filename (folder-with-index pattern); since
  `slug` is not an editable field in v1, the "`slug` on `index.md`" load
  error is not reachable from this form.

**On submit:** the server writes the new `.md` file to disk with minimal
frontmatter (`title`, `date` set to today, `draft` if checked) and an
empty body — the file exists on disk *before* the editor opens it
(filesystem is truth). The server then redirects to
`/admin/edit/<new-url-path>`. The watcher indexes the new file through
`Space.apply()` like any other external creation.

## 7. Concurrency and external edits

A file can change under an open editor: a `git pull` in production, the
watcher, another editor, a direct filesystem write. The behavior:

- **The watcher does not auto-reload an open editor.** Auto-reloading
  would silently discard the user's in-progress edits. The editor holds
  whatever it loaded; staleness is a save-time concern only.
- **Detection is at save time, via `If-Match`.** The editor captured the
  file's SHA-256 at load and sends it as `If-Match`. The server compares
  it against the current on-disk file (§5 step 3).
- **Mismatch → `409 Conflict`.** The editor shows a conflict notice and
  offers two explicit choices:
  - *Reload* — discard the in-editor changes and reload the current
    disk content (fresh body, fresh frontmatter, fresh hash).
  - *Overwrite* — re-issue the `PUT` with `If-Match: *`, which the
    server treats as an unconditional write.
- **No merge.** Three-way merge is out of scope. 409 is detect-only; the
  user decides.

This is the "loader produces; consumers decide" posture applied to
conflicts: the watcher keeps the index honest, the editor decides what to
do about a divergence.

## 8. Auth seam (dependency on subsystem 2)

The editor is a remote filesystem-write surface and cannot ship
unauthenticated. Auth itself is subsystem 2 (`better-auth`), built
separately. This spec specifies only the **seam**.

- All `/admin/*` routes (pages and the `PUT` API) sit behind a single
  server-side guard, `requireAuthor()`, enforced in a shared
  `admin/+layout.server.ts` (and the API endpoint calls the same helper).
  Unauthenticated requests get a redirect to login / `401`.
- Subsystem 2 implements the body of `requireAuthor()`. The routes do not
  change when it lands — only the helper's internals.
- **Interim, before subsystem 2 exists:** `requireAuthor()` is gated by
  an `AMBER_DEV_UNSAFE=1` environment flag. Unset (the default) → the
  guard denies all `/admin/*` access. Set → the guard allows access, so
  subsystem 1 can be built, run, and tested end to end before auth lands.
  When the flag is set the app logs a loud `logger.warn` at boot, and the
  flag is documented as **never for production**. Subsystem 2 removes the
  interim branch.

This makes subsystem 1 independently buildable and verifiable rather than
strictly blocked on subsystem 2, at the cost of one clearly-marked,
loudly-warned dev flag.

## 9. The Crepe dependency

The editor uses Milkdown's **Crepe** (`@milkdown/crepe`, `@milkdown/core`,
`@milkdown/kit`), the library the spike validated. Crepe is
ProseMirror-based and pulls Vue and remark into its dependency tree.

Containment:

- Crepe is **client-only** and **dynamic-imported** — loaded in
  `onMount`, never in the SSR bundle (the spike pattern).
- It is bundled **only into the `/admin/edit` route**. It does not enter
  the public render path's bundle at all.

This is the subsystem that confronts two `CLAUDE.md` lines. The spec
*names* the revisions; the build *makes* them (no `CLAUDE.md` edit in
this spec session):

- **"Server-only for content."** The public render path stays
  server-only (`+page.server.ts`, content as rendered HTML only). The
  editor is a separate, authenticated surface that necessarily receives
  body content as editable data. The rule is revised to scope it to the
  public render path; the revision lands with this subsystem's code.
- **"Every dependency justifies its presence" + the UI-component-library
  scope guard.** Crepe is a specialized markdown editor, not a generic UI
  component library or CSS framework — and it is the load-bearing
  component of the entire authoring-layer direction, already proven by a
  risk-first spike. It is adjacent enough to the scope guard that the
  guard is revised here, with this subsystem, to permit the editor
  library specifically. Justification: the dependency is contained to one
  route's client bundle and earns its place as the thing subsystem 1
  exists to deliver.

## 10. Components

Each unit has one purpose and a defined interface; the pure helpers are
unit-testable in isolation.

| Unit | Responsibility |
|---|---|
| `routes/admin/+layout.server.ts` | Calls `requireAuthor()`; gates the whole admin surface. |
| `routes/admin/+page.server.ts` / `.svelte` | Page-list index over `Space.pages`. |
| `routes/admin/edit/[...path]/+page.server.ts` | Resolve URL→file, read, `splitFrontmatter`, return body + parsed editable FM + `extra` + file hash. |
| `routes/admin/edit/[...path]/+page.svelte` | Dynamic-import + mount Crepe on the body; frontmatter side panel; save + 409 handling. |
| `routes/admin/api/page/[...path]/+server.ts` | `PUT` save: `If-Match` check, recombine, write, return new hash. |
| `routes/admin/new/+page.server.ts` / `.svelte` | New-page form, validation, create-and-redirect. |
| `lib/server/editor.ts` | Pure helpers: `hashFile`, `recombine(fm, body)` with the one-`\n` rule, `reserializeFrontmatter(known, extra)`. No I/O, no globals — unit-tested. |
| `lib/server/auth.ts` | `requireAuthor()` seam + the `AMBER_DEV_UNSAFE` interim branch. |
| `routes/[...path]/+page.svelte` (+ layout) | Conditional "Edit this page" link, server-gated on auth. |

Resolving URL→file reuses the `Space` index; the editor adds no second
content index.

## 11. Testing

Per `apps/web/CLAUDE.md`, pure functions get unit tests and route modules
get at least one runtime smoke test.

- **`lib/server/editor.ts`** — unit tests: recombine preserves verbatim
  FM, enforces exactly one body-edge `\n`, re-serialization preserves
  `extra` keys, hashing is stable.
- **Idempotent round-trip** — load a page body, save it unedited, assert
  the file is byte-identical except the documented body-edge
  normalization; a second save is a no-op. (The spike's `about.md` /
  `stress.md` evidence, made a permanent test.)
- **Concurrency** — load, mutate the file out-of-band, save with the
  stale `If-Match` → assert `409`; then save with `If-Match: *` → assert
  the write lands.
- **New-page validation** — reserved names, existing file, duplicate URL
  each rejected before any write.
- **Route smoke** — `/admin`, `/admin/edit/...`, the `PUT` endpoint
  exercised against the example-space fixture (runtime, not just direct
  handler import — route module shape is only validated at request time).
- The auth seam is tested with `AMBER_DEV_UNSAFE=1`; the real auth tests
  belong to subsystem 2.

## 12. Open questions for the operator

None block a plan session. Recorded for visibility:

- **Editing-conflict UX wording** — the exact copy of the 409 conflict
  notice and the reload/overwrite affordances is a build-time detail, not
  a spec decision.
- **Admin index ordering** — the page list's sort order (alpha, by
  `date`, by path) is unspecified; pick a sensible default at build time.
- **`date` field input** — whether the FM panel's `date` is a date
  picker or a text field is a build-time UX detail; the stored value is
  the loader's coerced ISO 8601 string either way.

These are deliberately left to the build; none change the architecture.
