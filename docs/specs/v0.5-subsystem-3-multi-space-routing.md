# v0.5 subsystem 3 — multi-space routing — spec

Status: **agreed design**, ready for a plan session.
Input: the v0.4 `getSpace()` registry refactor (`apps/web/src/lib/server/space.ts`),
which made the runtime path-keyed and named subsystem 3 as the module that
puts a request-to-space resolver in front of it. This spec turns that
half-built substrate into a buildable design.

The spec answers *what exactly are we building*. It does not write
implementation code and does not revise `CLAUDE.md` — it names the revisions;
the build makes them.

---

## 1. What this subsystem is

The runtime piece that lets one Amber instance serve multiple spaces from a
single process. A **resolver** runs once per request in `hooks.server.ts`,
maps the request to a space root, and hands the resolved space to every
downstream handler. The v0.4 registry already loads and caches `Space`
instances by absolute path; subsystem 3 is the resolver, the config that
drives it, and the route-level plumbing that threads the resolved space
through admin, theme, sitemap, robots, and asset handlers.

Resolution is **host-first with path fallback**:

- A request whose `Host` header matches a space's declared `host` resolves
  to that space.
- Otherwise, the **default space** (one space marked `default = true`)
  serves the host. If the path starts with `/<prefix>/…` for some
  prefix-mode space, that space wins and the prefix is stripped before the
  handler sees the URL.
- No default space + no host match → SvelteKit's built-in 404.

The admin surface (`/admin/*`) and the auth handler (`/api/auth/*`) live on
the operator's `AMBER_PUBLIC_URL` and are **not** resolved against spaces.
The admin gets one global sign-in (subsystem 2 unchanged) with a
space-picker that scopes editing to one space at a time via
`/admin/spaces/[slug]/…` routes.

### Non-goals

Explicitly out of scope for subsystem 3. Each is either a later subsystem,
a later increment, or deliberately not Amber's job:

- **Multi-user, invited authors, per-space permissions** — subsystem 4.
  Subsystem 3 keeps the single-admin model from subsystem 2 and lets that
  one admin edit every loaded space.
- **Space creation through the UI** — subsystem 5. Operators add spaces by
  mkdir + editing `space.toml`, then restarting. Subsystem 3 does not write
  `space.toml`.
- **Wildcard or regex hosts** (`*.example.com`, `^foo-.*$`). Exact-match
  `host` strings only. Operators who want wildcards put them in Caddy and
  point the traffic at a `default = true` space.
- **Per-space authentication providers.** All spaces share the global auth
  config from subsystem 2. A space cannot declare its own Google client.
- **Theme sharing between spaces.** Each space carries its own `themes/`
  directory. Subsystem 3 does not introduce a shared theme registry.
- **Hot reload of routing config.** Adding or removing a space requires a
  restart. The existing watcher continues to pick up content changes
  *inside* an already-loaded space; new spaces under `AMBER_SPACES_DIR` are
  not detected at runtime. Trivially addable later if needed; explicitly
  deferred here.
- **Cross-space links rendered as internal.** A markdown link to
  `https://notes.example.com/post` written in amber's content renders as a
  plain external link. The renderer doesn't know about Amber's space graph.

## 2. Surface and routes

Three categories of incoming URL:

| Category | Example | Resolver behavior |
|---|---|---|
| Admin / auth (operator origin) | `https://example.com/admin/...` `https://example.com/api/auth/...` | Short-circuit: skip space resolution; handler runs on the global admin context. |
| Per-space host | `https://amber.example.com/about` | Host match → amber's space, `mountPath = /about`. |
| Path-prefix on default host | `https://example.com/scratch/post-1` | Default host + prefix match → scratch's space, `mountPath = /post-1`. |
| Unclaimed host + no prefix | `https://random.example.com/x` | Default space serves `/x`. No default → SvelteKit 404. |

The admin/auth short-circuit is **host-aware**. A request arriving at
`amber.example.com/admin` is *not* a request for amber's `/admin` page (no
such route exists in the public space); the resolver detects it as an
admin URL on the wrong host and 302-redirects to `AMBER_PUBLIC_URL/admin`.
This keeps better-auth's session cookie on a single host (subsystem 2's
invariant) without making `/admin` a forbidden URL on per-space hosts.

The admin route table after this subsystem:

| Group | Route | Slug-scoped? |
|---|---|---|
| `(public)` | `/admin/login`, `/admin/setup` | No — global, unchanged from subsystem 2. |
| `(authed)` | `/admin` | No — space-picker landing (or 302 to the lone space in single-space mode). |
| `(authed)` | `/admin/account` | No — global, unchanged from subsystem 2. |
| `(authed)` | `/admin/spaces/[slug]` | Yes — page-list index for one space. |
| `(authed)` | `/admin/spaces/[slug]/edit/[...path]` | Yes — editor for one page. |
| `(authed)` | `/admin/spaces/[slug]/new` | Yes — new-page form. |
| `(authed)` | `/admin/spaces/[slug]/api/page/[...path]` | Yes — save endpoint, JSON only. |
| (root) | `/api/auth/[...path]` | No — better-auth handler, unchanged. |

The old subsystem-2 admin URLs (`/admin/edit/[...path]`, `/admin/new`,
`/admin/api/page/[...path]`) become **redirect-only shims** for one
version. They 302 to the equivalent `/admin/spaces/[slug]/…` URL in the
operator's default space. The shims may be removed in v0.6.

### Per-space public routes

The existing public route table (`/[...path]`, `/sitemap.xml`,
`/robots.txt`, `/themes/[name]/[...file]`, asset routes) is **not
changed**. The resolver runs before these handlers, populates
`event.locals.space` with the resolved `Space` instance, and rewrites
`event.url.pathname` to `mountPath`. Every handler reads
`event.locals.space` instead of calling `getSpace()` with no args.

## 3. The resolver

The resolver is a pure function over the resolver index + the request, with
no I/O of its own:

```
resolve(index, host, pathname)
  → { kind: 'admin', adminPath }                  // /admin/* or /api/auth/*
  → { kind: 'admin-elsewhere', redirectTo }       // /admin/* on a non-admin host
  → { kind: 'space', space, mountPath }           // a space serves the request
  → { kind: 'not-found' }                         // no host match, no prefix match, no default
```

The resolver index is built once at boot from the loaded registry. Its
shape:

```
{
  adminHost: string,                  // host of AMBER_PUBLIC_URL
  byHost:    Map<host, Space>,        // exact-match host index
  prefixes:  Array<{ prefix, space }>, // sorted by prefix length desc
  default:   Space | null
}
```

Algorithm, in order:

1. If `pathname` starts with `/admin/` (or equals `/admin`) or starts with
   `/api/auth/`: if `host === adminHost` → `kind: 'admin'`; else →
   `kind: 'admin-elsewhere'` redirecting to `adminHost + pathname + search`.
2. `byHost.get(host)` exists → `kind: 'space'` with that space,
   `mountPath = pathname`.
3. `default == null` → `kind: 'not-found'`.
4. For each prefix in `prefixes` (longest first): if `pathname === prefix`
   or `pathname.startsWith(prefix + '/')` → `kind: 'space'` with that
   space, `mountPath = pathname.slice(prefix.length) || '/'`.
5. Fall through to default: `kind: 'space'`, default space,
   `mountPath = pathname`.

Longest-prefix-first matters: a request to `/scratch-archive/x` must not
match a `prefix = "/scratch"` space. Sorting at index-build time keeps the
hot path branchless.

### Where the resolver runs

Two SvelteKit primitives, both backed by the same pure `resolve(...)`
function. The split exists because SvelteKit's `event.url` is read-only
in `handle`; URL rewriting has its own hook.

1. **`reroute` hook** (`src/hooks.ts`, runs on every request before route
   matching). For `kind: 'space'` requests with a non-empty prefix, returns
   `mountPath` so SvelteKit's route matcher sees the stripped path. For
   `admin`, `admin-elsewhere`, and `not-found` it returns the original URL
   — `handle` does the redirect / 404 / short-circuit work.
2. **`handle` hook** (`hooks.server.ts`, runs after `reroute`). Recomputes
   the resolver result against the *original* URL (carried via a header or
   by re-reading `event.url` + remembering the rewrite) and:
   - On `admin-elsewhere`, returns `Response.redirect(redirectTo, 302)`.
   - On `not-found`, calls SvelteKit's `error(404, ...)`.
   - On `admin`, sets nothing space-specific and forwards to the admin /
     auth handlers exactly as subsystem 2 left them.
   - On `space`, sets `event.locals.space = resolved.space` (and
     `event.locals.mountPath = mountPath` for handlers that want the
     post-rewrite path explicitly), then forwards.

After the reroute + handle pair runs, every existing public handler
(page, sitemap, themes, assets) reads `event.locals.space` instead of
calling `getSpace()` with no args. `event.url.pathname` already carries
the mounted (prefix-stripped) path because `reroute` rewrote it; no
handler needs to know about prefixes.

The double-resolve (once in `reroute`, once in `handle`) is cheap: the
resolver is a pure function over a small index, and the original URL +
host are available in both hooks. If the duplication ever becomes a real
cost, the rewritten URL can carry the resolution decision in a header
that `handle` reads — but the spec calls for the simpler "resolve twice"
shape at the implementation start.

## 4. Routing config — `space.toml` additions

Three new optional top-level keys, alongside the existing `theme`:

```toml
# space.toml — every field optional
theme = "amber-editorial"
host = "amber.example.com"     # exactly-one-host, no wildcards
prefix = "/scratch"             # mutually exclusive with host
default = true                  # at most one space across the whole runtime
```

### Rules

Enforced at scan time. Each violation emits a `LoadWarning` (§7) and drops
the offending field; the space stays in the registry but loses the bad
route hook. **The space is never dropped from the registry over a routing
problem — it just becomes unreachable via that route.**

- `host` and `prefix` set on the same space → `space_routing_conflict`,
  both dropped, space is unreachable (still listed in the picker).
- Two spaces with the same `host` → `space_routing_duplicate_host`,
  first-loaded wins, the loser's `host` is dropped.
- Two spaces with the same `prefix` → `space_routing_duplicate_prefix`,
  first-loaded wins.
- Two spaces with `default = true` → `space_routing_duplicate_default`,
  first-loaded wins; subsequent `default` flags are dropped.
- `prefix` not starting with `/`, equal to `/`, ending in `/`, or
  containing `?`/`#` → `space_routing_invalid_prefix`, dropped.
- `host` containing a scheme, port, path, or wildcard → currently rejected
  with the same `space_routing_invalid_host` warning. Bare host strings
  only (`amber.example.com`, not `https://amber.example.com:443/`).
- `host` equal to the admin host (derived from `AMBER_PUBLIC_URL`) →
  `space_routing_admin_host_collision`, the `host` field is dropped. The
  admin/auth short-circuit always wins on the admin host, so a space
  claiming that host would silently never serve traffic. The warning
  surfaces the misconfiguration loudly. The space itself stays in the
  registry and remains reachable as a default or via prefix.
- A space with no `host`, no `prefix`, and `default ≠ true` →
  `space_routing_unreachable` (info-level). The space is loaded and
  appears in the admin picker but never serves a public request.

### Why per-space `space.toml` and not a top-level routing file

"Filesystem is truth" applied to identity. Each space directory carries
its own routing rule next to its theme rule; adding a space means
`mkdir` + edit one file. The conflict-resolution rules above accept that
two spaces *could* claim the same host or prefix and define a deterministic
fallback rather than a fatal boot error. This matches the existing
`duplicate_url` semantics in the loader (first-wins, warned, fail-soft).

### Slug — the space's admin identifier

The space's **directory name** is its slug. No `slug` field in
`space.toml`. The constraint:

```
^[a-z0-9][a-z0-9-]{0,62}$
```

Directories whose names don't match emit `space_routing_invalid_slug`
and are dropped from the registry entirely (no slug → no admin URL → no
way to reach the space). The constraint is intentionally narrow to keep
admin URLs unambiguous and shell-safe.

Renaming a directory changes the slug, which changes the admin URL.
That's acceptable: admin URLs are not bookmarked by visitors, and the
rule avoids Amber owning a second naming convention.

## 5. Discovery and env

One new env var, **mutually exclusive** with the existing
`AMBER_SPACE_PATH`:

- `AMBER_SPACES_DIR=/srv/amber/spaces` — points at a directory whose
  immediate subdirectories are spaces. Subsystem 3 cold-start lists this
  directory; every subdirectory containing an `amber.toml` is loaded into
  the registry. Subdirectories without `amber.toml` are skipped silently
  (matches the existing reserved-prefix scan behavior).

Mutual exclusion:

- Exactly one of `AMBER_SPACE_PATH` or `AMBER_SPACES_DIR` must be set.
- Neither set → boot fails with a single-line error that names both vars
  and explains the choice. (Same pattern subsystem 2 introduced for
  `AMBER_PUBLIC_URL`.)
- Both set → boot fails. We do not silently prefer one over the other —
  the operator's intent is ambiguous and the safe stance is to refuse.

Subsystem 3 does not introduce any other env vars. `AMBER_PUBLIC_URL`,
`AMBER_AUTH_SECRET`, and the Google OAuth pair keep their subsystem-2
meanings.

### Symlinks under `AMBER_SPACES_DIR`

Allowed. The scan calls `readdir` then `stat` (not `lstat`) on each entry,
so a symlink that resolves to a space directory is a valid space.
Operators who want to keep spaces on a separate volume can symlink them
in.

## 6. Single-space backward compatibility

`AMBER_SPACE_PATH` mode keeps every observable behaviour from v0.4. The
resolver is still wired in, but with a degenerate index:

- One space in `byHost`? No — the lone space has no declared host. Instead
  the resolver treats it as if `default = true` was set, regardless of what
  the space's `space.toml` says about `host` / `prefix` / `default`. Those
  fields are *ignored* in single-space mode with one info-level log line
  per ignored field (so an operator who later flips to multi-space sees
  what was being ignored).
- The space-picker UI does not render. `/admin` 302-redirects to
  `/admin/spaces/[lone-slug]`. The lone slug is the directory name if it
  passes the slug regex; otherwise the literal string `default`. An
  invalid directory name does **not** crash the boot — single-space mode
  pre-dates the slug regex and we don't retro-enforce it.
- The old admin URLs (`/admin/edit/[...path]`, `/admin/new`,
  `/admin/api/page/[...path]`) keep working via 302 redirects to the
  per-space equivalents. These are pure redirects with no logic; they
  exist to avoid breaking subsystem-2 bookmarks for one version and may be
  deleted in v0.6.

The migration story for a single-space operator who wants multi-space:

1. Stop the container.
2. Move the space directory under a new parent: `mv /srv/amber-space /srv/amber/spaces/`.
3. Replace `AMBER_SPACE_PATH=/srv/amber-space` with
   `AMBER_SPACES_DIR=/srv/amber/spaces` in `.env`.
4. Edit the space's `space.toml` to add `default = true` (and optionally a
   `host`).
5. Start the container.

No data conversion. No `.amber/auth.db` change. Same admin row, same
sessions invalidated only by the restart (not by the migration itself).

## 7. `LoadWarning` codes introduced

New codes documented in CLAUDE.md alongside the existing enum. Each has a
single defined trigger:

- `space_routing_conflict` — a space declares both `host` and `prefix`.
- `space_routing_duplicate_host` — two spaces declare the same `host`.
- `space_routing_duplicate_prefix` — two spaces declare the same `prefix`.
- `space_routing_duplicate_default` — two spaces declare `default = true`.
- `space_routing_invalid_host` — `host` value isn't a bare host string.
- `space_routing_admin_host_collision` — `host` equals the admin host
  derived from `AMBER_PUBLIC_URL`.
- `space_routing_invalid_prefix` — `prefix` value isn't a clean
  leading-slash path.
- `space_routing_invalid_slug` — directory name doesn't satisfy the slug
  regex. Unlike the others, this one drops the space from the registry
  entirely.
- `space_routing_unreachable` — a space has no `host`, no `prefix`, and is
  not `default`. Informational; the space is still in the registry and
  reachable via admin slug.

The "unreachable codes are bugs" rule still applies: any code shipping
without a code path that can trigger it is a bug, not a placeholder.

## 8. CLAUDE.md revisions

Each revision lands with the subsystem's code, not before. The spec names
the revisions; the build PR makes them.

1. **"Space directory path comes from `AMBER_SPACE_PATH`"** — revised to
   "either `AMBER_SPACE_PATH` (single-space mode) or `AMBER_SPACES_DIR`
   (multi-space mode); mutually exclusive; boot fails if neither or both
   are set." The "no hardcoded paths" rule is unchanged.
2. **`LoadWarning` codes** — nine new codes documented per §7.
3. **Manifest is never silently rewritten** — unchanged. The resolver
   only *reads* `space.toml`. The pending v0.5+ revision note about
   space-creation/theme-picker UIs writing `space.toml` continues to wait
   for those subsystems.
4. **Roadmap shape** — v0.5 subsystem 3 marked **(shipped)** when this
   lands. The "v0.4 `getSpace()` registry refactor (half-unblocked this)"
   parenthetical is updated to past-tense closure ("the v0.4 registry made
   the runtime path-keyed; subsystem 3 lit it up").
5. **`amber_version`** — no bump. `space.toml` is hand-edited config, not
   a migrated on-disk shape; adding optional fields is forward-compatible.
   `amber_version` exists for content-format migrations and stays at its
   current value until one of those happens.

The "scope guards" section is **not** revised. Subsystem 3 introduces no
new build-time deps and adds no Compose services — both guards still
hold.

## 9. Testing

Vitest unit tests (run under Bun per `apps/web/CLAUDE.md`) cover the
resolver as a pure function plus the per-handler integration points:

### Resolver unit tests

A throwaway resolver index, no real spaces. Cover:

- host match wins over default
- prefix match (longest-first) under default
- admin host short-circuit
- admin-on-wrong-host → redirect to admin host
- no default + no host match → not-found
- single-space mode (degenerate index) → every path falls through to the
  lone space

These are pure function tests; they don't touch the filesystem or the
registry.

### Discovery / config-validation tests

A fixture parent directory under `apps/web/fixtures/multi-space/` with
several intentionally-broken spaces:

- Two spaces claiming the same host → one wins, warning emitted.
- A space with `host` and `prefix` both set → both dropped, warning.
- A space with `prefix = "foo"` (no leading slash) → dropped, warning.
- A space whose directory name has uppercase letters → dropped from
  registry, warning.
- A space marked `default = true` next to another also marked default →
  first wins, warning.

The fixture is read by both the resolver-index builder test and the
registry loader test.

### Route integration tests

Existing route tests (`page.test.ts`, `themes/.../server.test.ts`,
`sitemap.xml/...`, the admin page-list and editor tests) get a thin
wrapper that drives them through a resolver. The mechanical change at
each call site (`getSpace()` → `event.locals.space`) is covered by these
tests existing and continuing to pass; the tests themselves don't have
to know about the resolver beyond setting up `event.locals.space`.

### E2E (smoke)

The opt-in e2e suite (`AMBER_E2E=1`) gets a second fixture: two spaces
side-by-side under a `multi-space-fixture/`, one with `host =
"site-a.localtest.me"`, one with `default = true`. The smoke:

- Boots the prod bundle with `AMBER_SPACES_DIR=...fixtures/multi-space-fixture`.
- `Host: site-a.localtest.me` → serves site-a's `index.md`.
- `Host: 127.0.0.1` (unclaimed) → serves the default space.
- `/admin` on either host → 302 to `AMBER_PUBLIC_URL/admin`.
- The space-picker at `/admin` lists both spaces.
- Editing through `/admin/spaces/site-a/edit/about` writes to site-a's
  filesystem and not the default space's. (The PUT save assertion already
  exists; this extends it with a `slug` parameter.)

The single-space e2e suite (today's `admin.e2e.test.ts`) is preserved as
a second smoke run with `AMBER_SPACE_PATH` set — that guards the
backward-compat redirects.

## 10. Telemetry / logging

One subsystem tag, `subsystem: "resolver"`, on a child logger from the
existing `lib/server/logger.ts`. Log lines:

- **boot, info** — resolver index summary: `{ spaces: N, hosts: [...],
  prefixes: [...], default: slug | null }`. One line at startup, useful
  for operator diagnosis.
- **request, debug** — `{ host, path, resolved: slug | "admin" |
  "not-found", mountPath }`. Debug-level (off by default) — useful when
  tracking down "why did this request go to space X."
- **warn** — every `LoadWarning` code from §7 surfaces here at warn level
  with `{ code, slug, ... }`. Consistent with how the loader already
  emits its warnings.

No new metrics, no new log sinks. The existing pino setup carries this.

## 11. Risks and open questions

### Risk: the resolver in front of better-auth

better-auth's cookie / origin / CSRF checks are tightly coupled to
`AMBER_PUBLIC_URL`. The admin/auth short-circuit (§3 step 1) has to be
exact — any path under `/api/auth/*` that the resolver fails to short-
circuit will break sign-in. The build spike should land **the resolver
plus a passing subsystem-2 e2e suite first**, before any per-space route
refactor. If sign-in / sign-out / OAuth callback still work end-to-end
under multi-space mode with the resolver active, the rest of the
subsystem is mechanical.

### Risk: `AMBER_PUBLIC_URL` as both auth origin and admin host

Subsystem 2 already conflates these. Subsystem 3 inherits that
conflation. If we ever want to split them (admin on `admin.example.com`,
public canonical URL still `example.com`), that's a subsystem-2 follow-up,
not subsystem 3's problem. Spec acknowledges the coupling and does not
try to fix it.

### Open question: should the picker show admin-only routes?

A space that's `default = true` and a space that's `host`-routed both
have a public URL the picker can display. A space marked unreachable
(no host / no prefix / not default) has no public URL — the picker shows
"(unreachable — admin only)" next to its slug. This is intentional and
helps staging.

### Open question: what about `Forwarded` / `X-Forwarded-Host` headers?

Caddy in production sets `X-Forwarded-Host`. The resolver should read
the host from SvelteKit's `event.url.host`, which SvelteKit derives from
`ORIGIN`/forwarded headers per its adapter-node docs. The build should
verify this works against the e2e suite's `Host:` header injection (the
suite explicitly sets `ORIGIN=$base` to keep better-auth happy; same
discipline applies here).

## 12. Build order

Risk-first, matching subsystem 1 / 2's discipline:

1. **Spike: the resolver + reroute + handle pair, alongside subsystem 2's
   existing single-space behaviour.** Wire the pure `resolve()` function
   into both `src/hooks.ts` (`reroute`) and `src/hooks.server.ts` (`handle`)
   with a degenerate index that always returns `{ kind: 'space', space:
   getSpace(), mountPath: pathname }`. Run the full subsystem-2 e2e suite.
   If sign-in / sign-out / OAuth still pass — and the public page route
   still serves content — the seam is safe.
2. **Discovery + scan + registry plumbing.** `AMBER_SPACES_DIR` env, the
   parent-dir scan, the routing fields on `space.toml`, the warning
   codes. Resolver-index builder. Unit tests only — no route changes yet.
3. **Wire `event.locals.space` through every public handler.** Mechanical
   replacement of `getSpace()` with `event.locals.space` in the page
   route, sitemap, robots, themes, assets. Tests already cover these
   handlers; they just need a thin `event.locals.space` fixture.
4. **Per-space admin routes.** New `/admin/spaces/[slug]/...` tree.
   Backward-compat redirect shims from the old admin URLs.
5. **Picker UI at `/admin`.** Server-rendered list. Single-space mode
   redirects past it.
6. **E2E smoke fixture + multi-space e2e suite.** New fixture under
   `apps/web/fixtures/multi-space-fixture/`; new e2e file exercising the
   resolver and the picker end-to-end.
7. **CLAUDE.md revisions and roadmap-shape closure** land with the final
   step.

Each step is small enough to land as its own commit. Step 1 is the only
one that gates the rest — if it fails, the design is reconsidered before
any of 2-7 lands.
