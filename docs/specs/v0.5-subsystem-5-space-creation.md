# v0.5 subsystem 5 â€” space-creation UI â€” spec

Status: **agreed design**, ready for a plan session.
Input: the multi-space resolver shipped in subsystem 3
(`apps/web/src/hooks.ts`, `hooks.server.ts`, the per-space admin tree
under `/admin/(authed)/spaces/[slug]/...`) and the install-admin /
per-space permission model shipped in subsystem 4
(`apps/web/src/lib/server/permissions.ts`, the `isInstallAdmin` column on
`user`). Subsystem 3 named "space creation through the UI" as out of
scope and pointed at subsystem 5; subsystem 4 named install-admin as the
operator tier for filesystem-mutating actions. This spec turns those two
promises into a buildable design.

The spec answers *what exactly are we building*. It does not write
implementation code and does not revise `CLAUDE.md` â€” it names the
revisions; the build makes them.

---

## 1. What this subsystem is

The runtime piece that lets the install-admin create a new space â€”
directory, manifest, routing config, and a first content file â€” from
inside the admin UI, and have the new space appear in the picker, the
resolver, and (if routing was set) on the public web **without
restarting the process**.

Subsystem 5 owns three new things and revises one existing one:

- A new admin route `/admin/new-space` (install-admin only, multi-space
  mode only) that hosts the create form.
- A new `lib/server/space-create.ts` module that takes the validated
  form data and writes the directory tree atomically-ish (directory
  created last, rolled back on any error).
- A new `addSpace(absPath)` function on `lib/server/space.ts` that loads
  the new space into the path-keyed registry and rebuilds the resolver
  index in place.
- A revision to subsystem 3's "hot reload of routing config is a
  non-goal" â€” that non-goal stops applying for *creation*. Deletion,
  rename, and out-of-band edits to existing `space.toml` files still
  require a restart.

### Non-goals

Explicitly out of scope for subsystem 5. Each is either a later
subsystem, a later increment, or deliberately not Amber's job:

- **Editing an existing space's `space.toml`** (host, prefix, default,
  theme) from the UI. Subsystem 6 (theme picker) revises the file for
  the theme field; broader routing edits stay manual. After-creation
  changes to routing still require a restart.
- **Renaming or deleting a space.** Both are filesystem operations the
  operator does at the shell, then restarts. Adding the inverse of
  creation is its own future increment.
- **Migrating a single-space install to multi-space mode.** Subsystem 5
  hides itself in single-space mode and points to the doc that explains
  the migration. Migration is operator work at the shell, not a UI
  flow.
- **Choosing a theme at create time.** Subsystem 6.
- **Inviting a first editor at create time.** The install-admin can do
  that from the per-space members page (subsystem 4) after creation.
- **Hot-reload of the *full* config surface.** Only the *just-created*
  space is hot-added. Existing `space.toml` edits, removed directories,
  manual out-of-band `mkdir` under `AMBER_SPACES_DIR`, etc. still need
  a restart.

## 2. Surface and routes

One new route, one chrome change, no shims to maintain.

| Group | Route | Purpose |
|---|---|---|
| `(authed)` | `/admin/new-space` | The create form. GET renders the form, POST handles validation + creation + hot-add. Install-admin only; everyone else gets 403. Returns 404 in single-space mode. |

After successful create, the POST handler issues
`redirect(302, '/admin/spaces/<slug>')` â€” straight to the per-space
dashboard the operator already knows. The new space's scaffolded
`index.md` is the first thing they'll see; one click from there opens
the existing editor.

### Chrome additions

- The picker page at `/admin` (already lists spaces in `+page.svelte`)
  gains a "New space" button at the top of the list, visible only when
  `locals.user.isInstallAdmin && discoveryMode === 'multi-space'`. For
  install-admins on an empty multi-space install (`emptyState ===
  'no-spaces-loaded'`), the empty-state copy gains a "Create the first
  space" affordance pointing at the same route. The `no-memberships`
  branch is unchanged â€” non-install-admins never see a create
  affordance.
- The admin chrome (top-level `+layout.svelte`) does **not** get a new
  link. Install-admin-only routes are reached from contextually-related
  pages, not surfaced in the global nav. Matches subsystem 4's
  `/admin/users` shape.

No backward-compat shims â€” there's no prior URL to redirect from.

## 3. The create form

Single page, single column, install-admin-only, plain admin-chrome
aesthetic. Matches `/admin`'s `+page.svelte` rather than introducing a
new visual register. Amber's admin is utilitarian by design.

### Fields, in order

1. **Title** â€” text input, required, autofocused. Becomes
   `[site].title` in `amber.toml` and the `# {title}` heading in the
   scaffolded `index.md`.
2. **Slug** â€” text input, required. Auto-derived from Title (lowercased,
   non-`[a-z0-9-]` characters replaced with `-`, consecutive `-`
   collapsed, leading `-` stripped) and editable â€” the operator can
   override anything we derived. Validated against
   `^[a-z0-9][a-z0-9-]{0,62}$` from subsystem 3.
3. **How is it reached?** â€” single radio group with four exclusive
   options, each on its own row with a one-line muted description:
   - *Path prefix on the default site* â†’ reveals a `prefix` input
     (defaulted to `/<slug>`).
   - *Its own host* â†’ reveals a `host` input.
   - *This is the default site* â†’ no field. The option is disabled if
     a default space already exists; the row shows
     "currently: `<slug>`" inline.
   - *Admin-only for now* â†’ no field. Helper line: "Loads into the
     admin but doesn't serve public traffic yet. Add `host` or
     `prefix` to `space.toml` later."
4. **Submit** (primary) and a **Cancel** link back to `/admin`.

### Validation timing

- Slug regex: on **blur**, not keystroke (keystroke validation flashes
  errors mid-type).
- Collisions (slug-exists-on-disk, host already claimed, prefix already
  claimed, prefix is reserved, host equals admin host, default already
  set): **server-side on submit**, against the in-process registry â€”
  cheap, single source of truth, no client/server skew.

### Errors

Per-field errors sit directly below the input they relate to,
`role="alert"`, `aria-live="polite"`. Form-level errors (mkdir failed,
permission denied on `AMBER_SPACES_DIR`) sit above the submit button so
they're seen before re-submitting.

### Helper text

Sits under each input (not inside placeholder text â€” placeholders
disappear on focus and fail screen readers).

- Slug: "becomes the directory name and `/admin/spaces/<slug>`".
- Host: "bare host, no scheme or port. Example:
  `notes.example.com`."
- Prefix: "path the space is mounted at, e.g. `/notes`."

### Motion

Revealed conditional inputs animate in over ~200ms ease-out (opacity +
8px translate-from-below via `@starting-style`, no layout-property
animation). The submit button gets `transform: scale(0.97)` on
`:active` and a spinner during the async POST. No page-entry animation
â€” this is admin chrome, used repeatedly. No animation on
keyboard-initiated form submission.

## 4. The disk-writing module â€” `lib/server/space-create.ts`

A single exported function:

```ts
export async function createSpace(
  input: ValidatedCreateInput
): Promise<
  | { kind: 'ok'; absPath: string }
  | { kind: 'error'; code: CreateErrorCode; detail?: string }
>;
```

`ValidatedCreateInput` is the form's submitted shape *after* server-side
validation has already happened â€” the function trusts its input.
Splitting validation from writing keeps the writer pure-ish (it does
I/O but doesn't know about the registry or HTTP).

### Inputs

```ts
{
  slug: string,
  title: string,
  routing:
    | { kind: 'host';   host: string }
    | { kind: 'prefix'; prefix: string }
    | { kind: 'default' }
    | { kind: 'admin-only' }
}
```

### What it writes

In this order, all inside `<AMBER_SPACES_DIR>/<slug>/`:

1. `mkdir(absPath, { recursive: false })` â€” fails fast if the directory
   already exists. We check earlier in the validator; the `EEXIST` here
   is the race-condition backstop.
2. `amber.toml`:
   ```toml
   amber_version = "0.1"

   [site]
   title = "<title escaped>"
   ```
   Nothing else. No nav block, no description, no author â€” those are
   editor-driven additions the operator makes later. `[redirects]` is
   omitted; CLAUDE.md keeps the table reserved but doesn't require it
   to exist.
3. `space.toml`:
   - `kind: 'host'` â†’ `host = "<host>"`
   - `kind: 'prefix'` â†’ `prefix = "<prefix>"`
   - `kind: 'default'` â†’ `default = true`
   - `kind: 'admin-only'` â†’ file is **not written**. The space loads
     with no `space.toml`; the theme-resolution chain falls through
     cleanly (the loader already tolerates a missing `space.toml`).
4. `index.md`:
   ```markdown
   ---
   title: "<title escaped>"
   ---

   # <title>

   Welcome to <title>. This is your homepage â€” open the editor to
   make it your own.
   ```

### Rollback

A `try` wraps steps 2-4; on any throw, `rm -rf` the directory created
in step 1 (we know it didn't exist before us â€” step 1 used
`recursive: false`) and return
`{ kind: 'error', code: 'write_failed', detail: ... }`. We don't
partially recover; the operator re-submits the form.

### Error codes

The writer's, distinct from `LoadWarning`:

- `dir_already_exists` â€” race with another writer or out-of-band mkdir
  (validator missed it).
- `write_failed` â€” any I/O error during file writes. Detail carries
  the underlying message for log surfacing.
- `permission_denied` â€” `EACCES`/`EPERM` from `mkdir` or write.
  Separated so the form can show a targeted "Amber doesn't have write
  access to `AMBER_SPACES_DIR`" message instead of generic
  "write failed".

### TOML escaping

We hand-write the file as a string, not via a TOML serializer
dependency. Title is the only user-controlled value that lands in TOML;
we escape it for TOML basic strings (`\`, `"`, control chars) inline.
No new build-time dep, ~10 lines of code, unit-tested with a fuzz set.

### Why a separate module from the route handler

Three reasons. (1) The handler does HTTP, the writer does I/O â€”
different failure surfaces. (2) The writer is unit-testable against a
real `os.tmpdir()` directory without standing up a SvelteKit request.
(3) A future "import an existing space" feature (off-roadmap) would
reuse the same scaffolder shape.

## 5. The registry hot-add API â€” `lib/server/space.ts`

One new exported function on the v0.4 registry module:

```ts
export async function addSpace(absPath: string): Promise<Space>;
```

### Contract

Loads the space directory at `absPath` via the existing
`Space.load(absPath)` path, inserts it into the path-keyed `Map`,
recomputes the resolver index from the updated registry, returns the
loaded `Space`. Throws if `absPath` is already in the registry
(programmer error â€” caller must dedupe), or if `Space.load()` rejects
(corrupt `amber.toml`, missing directory, etc.).

### What "recompute the resolver index" means

Subsystem 3 built the index once at boot from the loaded registry (host
map, prefix list sorted by length desc, default pointer, admin host).
The resolver itself is a pure function; rebuilding the index calls
`buildResolverIndex(...)` from `lib/server/resolver-index.ts` â€”
unchanged. The boot result currently lives as a `const resolverIndex`
inside `hooks.server.ts`'s module scope. For `addSpace()` to mutate
it, the build moves that holder into a tiny shared module â€”
`lib/server/resolver-index-holder.ts` â€” with `get`/`set` accessors,
matching the existing pattern used by `$lib/reroute-prefixes` and
`$lib/server/default-space`. `hooks.server.ts` reads via the getter
on each request; `addSpace()` calls the setter after rebuilding.
Single-assignment swap, no locking needed in Node's single-threaded
model.

`addSpace()` also updates two adjacent pieces of mutable boot state:

- `setReroutePrefixes(newIndex.prefixes.map((p) => p.prefix))` â€”
  without this, SvelteKit's `reroute` hook wouldn't strip the
  new space's prefix off matching requests.
- `setDefaultSlug(computeDefaultSlug(newIndex))` â€” keeps the
  v0.4-compat admin shims pointing at the right space if the
  new space takes the `default` role (defense-in-depth â€” the
  form blocks this case, but the writer trusts the validator).

All three updates are part of `addSpace()`'s atomic-feeling sequence
â€” they happen synchronously, in order, after `Space.load()` succeeds
and before `addSpace()` returns. If the writer rolls back later,
none of these were called; if `addSpace()` itself throws (e.g. a
duplicate-path assertion at the top), none were called either.

### Routing-config validation at add-time

The newly-loaded space may carry a `space.toml` with `default = true`,
a `host`, or a `prefix`. Subsystem 3's `LoadWarning` rules apply
unchanged â€” first-loaded wins for `default`/`host`/`prefix` collisions,
the loser's field is dropped. Concretely: if a default space already
exists in the registry and the new space also claims `default`, the
new space's `default` is dropped and
`space_routing_duplicate_default` is logged. The form already prevents
this case (the "default site" radio option is disabled when one
exists), so this is defense-in-depth, not an expected path.

### Failure surface

If `Space.load()` throws (e.g. the just-written `amber.toml` somehow
doesn't parse), the registry is *not* mutated, the resolver index is
*not* rebuilt, and the error propagates to the route handler â€” which
then triggers the `space-create.ts` rollback (`rm -rf` the partial
directory). Net effect of a failed `addSpace`: filesystem state
untouched, runtime state untouched.

### Watcher coupling

The v0.1 watcher operates *inside* a loaded `Space` instance (file
events under that space's content dir). Adding a new `Space` to the
registry causes a fresh watcher to spin up via the existing
`Space.load()` path â€” no new code. Removing a watcher on rollback
isn't needed because `Space.load()` failed before the watcher was
attached.

### Why on `space.ts` and not a new module

The registry IS `space.ts`. The hot-add lives where the `Map` is.
Adding a new module to wrap a single function makes the call graph
harder to follow. Subsystem 3 didn't do that for `getRegistryEntries`;
subsystem 5 doesn't do it for `addSpace`.

### What this doesn't do

Called out explicitly so the non-goal is concrete:

- No `removeSpace(absPath)`. Deletion still requires a restart.
- No `reloadSpace(absPath)`. Out-of-band `space.toml` edits to existing
  spaces still require a restart.
- No filesystem watcher on `AMBER_SPACES_DIR` itself. Only spaces
  created through the UI hot-add; a manual `mkdir` under
  `AMBER_SPACES_DIR` still requires a restart.

## 6. Validation rules and new error codes

Validation runs server-side at form submission, before
`space-create.ts` touches the disk. Two failure surfaces â€”
**input-shape errors** (rejected at the form, recoverable, user
re-submits) and **load-time warnings** (subsystem 3's existing
`LoadWarning` enum, emitted from `addSpace`'s call to `Space.load`).

### Form-rejection codes

Returned to the form as field-level errors. Not loaded into the
registry, not emitted as `LoadWarning` â€” these are user input
problems.

| Code | Trigger | UI surface |
|---|---|---|
| `title_empty` | Title is empty after trim. | Below title: "Title is required." |
| `slug_invalid` | Slug doesn't match `^[a-z0-9][a-z0-9-]{0,62}$`. | Below slug: "Use lowercase letters, digits, and hyphens. Must start with a letter or digit." |
| `slug_taken` | A directory at `<AMBER_SPACES_DIR>/<slug>` already exists (loaded or not). | Below slug: "A directory named `<slug>` already exists. Choose another name." |
| `host_invalid` | Host contains a scheme, port, path, or wildcard, or is empty. | Below host: "Bare host only, no scheme or port. Example: `notes.example.com`." |
| `host_is_admin` | Host equals the admin host derived from `AMBER_PUBLIC_URL`. | Below host: "This is the admin host. The admin/auth endpoints always win here â€” pick a different host." |
| `host_taken` | Another loaded space already declares this `host`. | Below host: "`<host>` is already used by `<other-slug>`." |
| `prefix_invalid` | Prefix not starting with `/`, equal to `/`, ending in `/`, or contains `?`/`#`. | Below prefix: "Path like `/notes`. No trailing slash." |
| `prefix_reserved` | Prefix is `/admin`, `/api`, `/themes`, or a well-known file path. | Below prefix: "`<prefix>` collides with a reserved Amber path." |
| `prefix_taken` | Another loaded space already declares this `prefix`. | Below prefix: "`<prefix>` is already used by `<other-slug>`." |
| `default_taken` | The "default site" option was submitted, but a default space already exists. | Defense-in-depth â€” the radio option is also disabled in the UI for this case. Server-side: "A default space already exists." |

These reuse the same string predicates and reserved-prefix list as
subsystem 3's `LoadWarning` validators, but emit a different *code
shape* because the response shape is different (form error vs.
registry warning).

### Write-time codes

The two `space-create.ts` error returns surface as form-level errors
above the submit button:

| Code | Trigger | UI surface |
|---|---|---|
| `permission_denied` | `EACCES`/`EPERM` writing under `AMBER_SPACES_DIR`. | Above submit: "Amber doesn't have write access to `<AMBER_SPACES_DIR>`. Fix the directory permissions and try again." |
| `write_failed` | Any other I/O error. Detail (errno, syscall) is logged but not shown. | Above submit: "Couldn't create the space. Check the server logs for details." |

### Load-time warnings

`addSpace()` calls `Space.load()`, which can emit subsystem-3 warnings
(`space_routing_*`, `frontmatter_parse_error`, `duplicate_url`, etc.).
None of these *block* creation â€” the space loads, the warning is
logged with `subsystem: "resolver"`, the form's success redirect still
fires.

This is intentional: the writer just produced a clean `amber.toml` +
(optional) `space.toml`, so warnings here are highly unusual
(filesystem corruption, racing edits). If they happen, the space is in
the registry and editable; we don't throw away a successfully-created
space over a non-fatal warning.

### One new `LoadWarning` code

None. The hot-add path reuses subsystem 3's full warning vocabulary.
The form-rejection codes above are a separate enum (they don't go
through `LoadWarning` because they're caught before the loader runs).
The CLAUDE.md "every `LoadWarning` code has a defined trigger" rule
stays clean.

## 7. Permission gating, single-space mode, chrome integration

Three small things bundled because each is a few lines.

### Permission gate

The route's `+page.server.ts` `load` and the form action both check:

```ts
if (!locals.user?.isInstallAdmin) throw error(403, 'install-admin only');
```

This matches subsystem 4's `/admin/users` exactly â€” install-admin is
the operator tier, and creating a directory on the host filesystem is
operator-tier work. The `(authed)` layout already 302s anonymous users
to `/admin/login`, so the only case this guards is a signed-in
non-install-admin (a space owner or editor) hitting the URL directly.

The check happens in `load` so signed-in editors get a 403 page rather
than seeing a form they can't submit. The form action repeats the same
check â€” defense in depth against a stale session or an unusual request
shape.

### Single-space mode

The existing registry already tracks discovery mode (`AMBER_SPACE_PATH`
vs `AMBER_SPACES_DIR`) implicitly. We expose a thin getter:

```ts
export function getDiscoveryMode(): 'single-space' | 'multi-space';
```

In the route's `load`, before the permission check:

```ts
if (getDiscoveryMode() === 'single-space') throw error(404, 'not found');
```

404, not 403. The subsystem is *hidden*, not *refused*. Per Â§1
non-goals, migrating an install from single-space to multi-space is a
documented manual operation â€” the user clicks no UI to get there. A
404 keeps the route invisible; the operator who reads docs and sets
`AMBER_SPACES_DIR` discovers the route by then having access to it.

### Chrome integration â€” the picker page

`/admin/(authed)/+page.svelte` already lists the loaded spaces. Two
small changes:

1. The page-load shape (`+page.server.ts`) gains two fields:
   `canCreate: boolean` (= `user.isInstallAdmin &&
   discoveryMode === 'multi-space'`) and `discoveryMode`. The existing
   `spaces` list, `emptyState`, and `user` fields stay.
2. The template gains a "New space" anchor styled as a button â€”
   `<a href="/admin/new-space">` â€” rendered only when `canCreate`. No
   JavaScript needed. Sits at the top of the page next to the
   `<h1>Spaces</h1>` heading.

The empty-state branches stay almost unchanged. For install-admins on
an empty multi-space install (`emptyState === 'no-spaces-loaded' &&
canCreate`), the copy gains a "Create the first space" affordance
pointing at the same route. The `no-memberships` branch is unchanged â€”
non-install-admins never see a create affordance.

### One follow-on edit

`/admin/(authed)/+page.server.ts` gains a single `getDiscoveryMode()`
call. The new logic is roughly three lines added to the existing
`load`. No new files, no new exports beyond `addSpace` (Â§5),
`getDiscoveryMode` (this section), and the new route's pair.

## 8. Testing

Three layers, matching subsystem 3's discipline.

**`space-create.test.ts`** â€” exercises the writer against a real

### Picker page test update

`picker.test.ts` already exists. One added assertion: install-admin in
multi-space â†’ `canCreate: true`; editor â†’ `canCreate: false`;

### E2E (opt-in)

- Browsing to `/notes` on the default host serves the scaffolded

- The picker now lists `notes` alongside the original spaces.

This is the **load-bearing test** â€” it exercises the spec's headline
claim ("creates a space and it's reachable without a restart")
end-to-end in a real Chromium against the real prod bundle. If
anything in Â§4 (writer) or Â§5 (`addSpace` + resolver-index rebuild)

### Smoke gate

The e2e file is part of the existing `AMBER_E2E=1` opt-in project;
`bun run test:smoke` continues to be the operator's pre-deploy gate.
No CI changes â€” Amber doesn't have CI by design.

## 11. Risks and open questions

### Risk: resolver-index swap races a concurrent request

`addSpace()` rebuilds and swaps `currentIndex` with a single
assignment. Node's event loop is single-threaded, so a request
handler reading `currentIndex` between the rebuild and the swap is
impossible at the language level â€” JS code between `await` points
runs atomically. The risk is only theoretical (a future migration to
worker threads, or some Bun-specific surprise). The build verifies
the assumption with a stress test: 50 concurrent requests during an
`addSpace()` call, all should resolve consistently to either pre-add
or post-add state, never to an inconsistent in-between. If Bun
surprises us, the fallback is wrapping the swap in a
`Promise.resolve().then(...)` to defer to a microtask boundary â€”
cheap, doesn't change the public API.

### Risk: out-of-band edits between form-load and form-submit

The "default site already exists" check (and the host/prefix
already-taken checks) reads the live registry on form submit. Between
the `load` that disabled the radio option and the `action` that
re-checks, another install-admin in another tab could have created a
default space. The server re-check catches this â€” the form re-renders
with `default_taken` instead of mis-creating. Worth noting in Â§6's
table; not worth special handling.

### Risk: `amber.toml` parser accepting what we wrote

We hand-write TOML. The escape rules for basic strings (`"..."`) cover
the cases that matter â€” quote, backslash, control chars. The build
adds a fuzz test: ~1000 random titles (Unicode, control chars, emoji,
RTL, zero-width joiners) round-trip through write â†’ `Space.load()` â†’
read-back equals input. If a category fails, escape it; if escape
isn't enough, restrict the character set with a `title_invalid` form
code.

### Open question: `index.md` frontmatter or no frontmatter?

Â§4 scaffolds an `index.md` with a YAML frontmatter block containing
`title:`. The loader doesn't *need* it on `index.md` (the URL is `/`,
the title falls back to the manifest's `[site].title`). It's there so
the editor â€” opened immediately after create â€” shows a frontmatter
block the operator can edit (description, draft toggle, date) rather
than a blank file. Counter-position: ship a clean Markdown file with
just the `#` heading; the operator adds frontmatter when they want
it. **Lean: keep the frontmatter scaffold.** Worth confirming in the
spec review.

### Open question: should we offer a "preview where this serves" hint?

The form's routing radio could show a calculated public URL ("this
space will serve `https://example.com/notes`") next to the chosen
option. Nice affordance. Adds 10-20 lines of derivation logic (admin
host + prefix â†’ URL). Left out of Â§3 to keep the form focused;
reasonable to add. **Lean: defer â€” easy follow-up after first ship
if operators ask.**

### Open question: the empty-state install-admin copy

Picker page, install-admin, multi-space mode, zero spaces. Today the
copy is "No spaces loaded. Ask your administrator to add one." â€”
written before subsystem 5 existed. Â§7 says install-admins get
"Create the first space" copy. The exact wording (button label,
surrounding sentence) isn't in this spec; it's a copy decision worth
getting right but not architecture. Punt to implementation, flag for
review.
