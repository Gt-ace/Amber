# v0.5 subsystem 4 — invites and per-space permissions — spec

Status: **agreed design**, ready for a plan session.
Input: the single-admin model shipped in subsystem 2
(`apps/web/src/lib/server/auth.ts`, `auth-config.ts`, `auth-queries.ts`) and
the per-space admin tree shipped in subsystem 3
(`apps/web/src/routes/admin/(authed)/spaces/[slug]/...`). Subsystem 2 named
"multi-user invites and per-space permissions" as out of scope and pointed
at subsystem 4; subsystem 3 wired a single-admin "edit every loaded space"
posture and named subsystem 4 as the module that scopes editing per user.
This spec turns those two promises into a buildable design.

The spec answers *what exactly are we building*. It does not write
implementation code and does not revise `CLAUDE.md` — it names the revisions;
the build makes them.

---

## 1. What this subsystem is

The opt-in multi-user layer over Amber's authoring surface. After this
subsystem, an Amber install holds:

- One **install-admin** — the user claimed at `/admin/setup`. Implicitly
  owns every loaded space; never appears in the `member` table. This is
  the same row subsystem 2 already creates; subsystem 4 adds an
  `isInstallAdmin` column to it.
- Zero or more **invited users** — created by redeeming a per-space invite
  URL. Each carries an explicit `member(user_id, space_slug, role)` row
  for every space they belong to.

Two per-space roles:

- **Owner** — edits content, generates invites for the space, changes
  member roles, removes members, revokes pending invites. Future
  subsystems (5, 6) will hang space-settings and theme-picking off the
  owner role too.
- **Editor** — edits content. Cannot invite, change roles, or remove
  members. Cannot see the members list.

Invites are **bearer one-time URLs**, generated by an owner or the install-
admin, copied out of the UI and delivered out-of-band (chat, email outside
Amber, in person). They carry a space, a role, an expiry, and a single-use
token. There is no SMTP in Amber and subsystem 4 does not add one.

The seam at the call-site stays familiar: the existing `requireAuthor()`
keeps its meaning (authenticated at all), and a new
`requireSpaceAccess(event, slug, role?)` joins it as the per-space gate.
The per-space `[slug]/+layout.server.ts` already resolves the slug to a
`Space`; subsystem 4 makes it also resolve the user's role and stash it on
`event.locals`.

### Non-goals

Explicitly out of scope for subsystem 4:

- **In-app email**. No SMTP, no email-based password reset, no email-based
  invite delivery. Same posture as subsystem 2. Invite URLs are copied
  from the UI and delivered out-of-band by the operator.
- **Promoting an invited user to install-admin**. There is exactly one
  install-admin row per install in v0.5; transferring it is a CLI-only
  operation (subsystem 4 does not ship a `transfer-admin` CLI, only a
  `grant-ownership` one — §10). A future subsystem may add UI for it.
- **Demoting the install-admin** through the UI. The CLI escape hatch
  exists; the UI never offers the operator a way to lock themselves out.
- **Page-level or path-level permissions**. Space-level is the only
  granularity. A user with edit access to a space can edit every page in
  it, including drafts.
- **Draft-preview on public URLs for non-install-admins**. Drafts still
  return 404 on the public render path. Editors and owners preview drafts
  by navigating into the editor at `/admin/spaces/[slug]/edit/<path>`,
  unchanged from subsystem 3.
- **An audit-log table or UI**. Subsystem 4 emits structured pino log
  lines for member-mutation events (§11); a queryable audit table is not
  shipped here.
- **Group / team abstractions**. Each user gets explicit per-space rows.
  No "invite a group" or "team admin" tier.
- **Per-user session lifetime tuning, account locking, login throttling
  beyond better-auth's defaults**. Inherits subsystem 2's posture.
- **An "I forgot which email I used" recovery path**. The invitee
  chooses their own email at redemption; the operator can re-invite if
  it's lost.

## 2. Surface and routes

Three new public routes (no session required), several new authenticated
routes, and one minor extension to the existing public render path.

| Group | Route | Notes |
|---|---|---|
| `(public)` | `/admin/invite/[token]` | Invite-redemption screen. Load returns 410 Gone on expired/redeemed/unknown tokens; otherwise renders the claim form (email+password, optional Google button). Action creates the user, marks the invite redeemed, sets the session, and 302s to `/admin/spaces/[slug]`. |
| `(authed)` | `/admin` | Unchanged route, behaviour scoped: lists only spaces the current user can access. Install-admin sees every loaded space. A user with zero memberships sees the empty-picker copy. Single-space mode (one loaded space, current user has access) still 302s through to `/admin/spaces/[slug]`. |
| `(authed)` | `/admin/spaces/[slug]/members` | Owner-only (plus install-admin). Lists members and their roles, lists pending invites, lets the actor generate an invite (returns the URL once, copy-to-clipboard affordance), revoke a pending invite, change a member's role, or remove a member. |
| `(authed)` | `/admin/users` | Install-admin only. Install-level user list: every user row, last sign-in (from better-auth's `session` table), space memberships count, "Delete user" with hard confirmation. Where install-admins go when they need to know who's on their server. |
| `(authed)` | `/admin/account` | Existing route, gains a "Delete my account" affordance. Blocked when `isInstallAdmin` is true — operator's only path to leaving the seat is the CLI. |
| `(authed)` | `/admin/spaces/[slug]` `/admin/spaces/[slug]/edit/[...path]` `/admin/spaces/[slug]/new` `/admin/spaces/[slug]/api/page/[...path]` | Existing routes. Gain a single shared gate: the per-space layout calls `requireSpaceAccess(event, slug)`; the inner mutators (`/edit`, `/new`, the PUT save endpoint) call `requireSpaceAccess(event, slug, 'editor')` defensively. Owners and editors both pass; non-members 403. |
| (root) | `/[...path]` (public render) | Unchanged route. The inline "Edit this page" link now probes `canEdit(event, locals.space)` instead of `isAuthor(event)`. Behaviour for the install-admin and an editor on the matching space: link appears. For any other signed-in user: link absent. For a signed-out visitor: link absent (unchanged). |

### What's deliberately not on the route table

- **No `/admin/invites` install-level pending-invite index.** Pending
  invites are listed per-space on the `members` page. An install-admin
  who wants to audit every pending invite reads the `invite` table
  directly or runs the existing logger (§11).
- **No `/admin/spaces/[slug]/settings`.** Space-settings UI is
  subsystem 5/6. Subsystem 4 does not introduce a settings page just to
  hang "members" off it.

### Picker filtering

`/admin` currently lists every entry from `getRegistryEntries()`. After
subsystem 4 the same load filters to:

- If `locals.user.isInstallAdmin` → every entry.
- Else → entries the user has a `member` row for.

The single-space-mode 302 still fires when the *visible* list has exactly
one entry (so a normal editor with one space lands on it directly; a
zero-membership editor lands on the empty-picker copy).

## 3. Permission model

### Roles, in code

```ts
export type SpaceRole = 'owner' | 'editor';

// What landed on event.locals after the per-space layout runs:
type ResolvedAccess =
  | { kind: 'install-admin' }              // implicit, no member row
  | { kind: 'member';     role: SpaceRole } // explicit row, role known
  | { kind: 'none' };                       // not a member, not install-admin
```

### `isInstallAdmin` on the user row

A boolean column on better-auth's `user` table, added via a one-shot
migration the build owns (better-auth's Kysely adapter allows extra
columns; we apply our migration after better-auth's on boot — see §5 for
the migration mechanics). Set to true exactly when the row was inserted
via the `/admin/setup` action. The user-create hook in
`auth-config.ts` is extended so that subsequent creations (via invite
redemption) leave the flag unset.

The CLI `grant-ownership` does not touch this flag (it only creates
`member` rows). The flag exists only as the seam between subsystem 2's
"single-admin" identity and subsystem 4's per-space membership world.
Future "transfer install-admin" work would touch it; not in scope here.

### The permission seam — `lib/server/permissions.ts`

A new module alongside `lib/server/auth.ts`. Exports:

```ts
// Throwing guards used by route handlers and the per-space layout.
export function requireSpaceAccess(
  event: RequestEvent,
  slug: string,
  minimumRole?: SpaceRole // default: any role passes
): ResolvedAccess;        // returns the resolved access; never null

// Non-throwing probes used by the public render path.
export function canEdit(event: RequestEvent, space: Space): boolean;
export function canRead(event: RequestEvent, space: Space): boolean;
```

`requireSpaceAccess()` looks at `event.locals.user` first (must be
non-null, else `error(401)`), then:

1. If `user.isInstallAdmin` → return `{ kind: 'install-admin' }`.
2. Else query `member` for `(user.id, slug)`. Found → return
   `{ kind: 'member', role }`. Not found → `error(404, ...)` (deliberate:
   we don't tell non-members whether the space exists; same posture the
   existing `[slug]/+layout.server.ts` already takes for unknown slugs).
3. If `minimumRole = 'editor'` → both roles pass. If
   `minimumRole = 'owner'` → only owner passes; editor returns
   `error(403, ...)`.

The "404 on unknown OR not-a-member" merge is a deliberate disclosure
choice. It matches the existing layout behaviour for unknown slugs and
keeps the response shapes consistent across the (unknown-space-slug,
known-space-but-not-a-member) pair.

`canEdit()` and `canRead()` are pure reads of `event.locals.role` (set
by the layout once per request, see below) and `event.locals.user`. They
never query.

### Where the role is resolved

The per-space `[slug]/+layout.server.ts` already runs on every
`/admin/spaces/[slug]/*` request and stashes `locals.space`. Subsystem 4
extends it:

```ts
const access = requireSpaceAccess(event, params.slug);
event.locals.access = access;
event.locals.role = access.kind === 'install-admin' ? 'install-admin'
                 : access.kind === 'member'        ? access.role
                 : null;
```

After the layout runs, every nested route reads `locals.role` for its own
gating decisions (the `members` route checks `role === 'owner' ||
'install-admin'`; the `/new` and `/edit` routes check
`role !== null` — already enforced by `requireSpaceAccess()`'s default).

For the **public render path**, there is no `[slug]/+layout.server.ts` —
the resolver from subsystem 3 already populates `locals.space`. The
`canEdit()` probe in `routes/[...path]/+page.svelte`'s "Edit this page"
link runs against `locals.user` and `locals.space.path` directly; it
does *not* run the same `requireSpaceAccess()` heavy path because it
must remain a single DB lookup (one `member` query) on the hot path.

### Install-admin is implicit, never a `member` row

Inserting an install-admin row into `member` is a code smell — the
single source of truth for "this user is the operator" is the
`isInstallAdmin` column on the `user` row. Subsystem 4 enforces this:

- The setup action does **not** insert a `member` row for the new admin.
- The CLI `grant-ownership` refuses to insert a `member` row for a user
  whose `isInstallAdmin` is true (it's a no-op with a clear message).
- A future "demote install-admin" feature would, at that point, want to
  decide whether to materialize their implicit ownerships into rows;
  subsystem 4 doesn't pre-empt that decision.

## 4. Invite lifecycle

### Token generation

An owner (or install-admin) clicks "Invite" on
`/admin/spaces/[slug]/members`. They pick the role (`owner` or
`editor`). The action:

1. Generates a 32-byte random token via `crypto.randomBytes(32)`,
   base64url-encoded for URL safety.
2. Inserts a row into `invite` with `(token_hash, slug, role,
   expires_at = now + 7 days, created_by = actor.id)`.
3. Returns the **URL** (not just the token) to the action's payload:
   `{ inviteUrl: '<AMBER_PUBLIC_URL>/admin/invite/<token>' }`.
4. The Svelte page renders the URL once with a copy-to-clipboard button
   and a "this URL is only shown once" notice.

The DB stores **the hash** of the token, not the token itself. The token
lands in two places: the operator's clipboard and (briefly) the page that
generated it. If the DB is dumped, the tokens can't be replayed against
the redeem endpoint. (`bcrypt`/`argon2` overkill for a 256-bit random
token; a single SHA-256 is enough — preimage resistance is what we need,
not work factor.)

Expiry is fixed at **7 days**. No env var. Operators who need a different
window can regenerate; the design accepts that v0.5 will not satisfy
every preference here.

### Token redemption

`/admin/invite/[token]` is a public route in the `(public)` group, like
`/admin/login` and `/admin/setup`.

**Load** (`+page.server.ts`):

- SHA-256 the token from the URL; look up by `token_hash`.
- Not found → `error(410, 'This invite link is no longer valid.')`.
- `redeemed_at` is non-null → `error(410, 'This invite has already been used.')`.
- `expires_at < now` → `error(410, 'This invite has expired.')`.
- Otherwise render the claim form with the space's title and the role
  shown for context (so the invitee knows what they're accepting).

**Action** (`POST`):

The action runs inside a SQLite transaction on `auth.db`:

1. Re-fetch the invite by `token_hash`. Repeat the validity checks. If
   any fail (race against expiry, race against another redemption),
   return 410. Subsystem 2 already taught us that the *transactional
   re-check* is the only correct way to close a TOFU window.
2. Call better-auth's `auth.api.signUpEmail()` (or the Google sign-in
   path with the invite-state token — see below). The user-create hook
   in `auth-config.ts` is extended: it checks whether the request
   context carries a `pending-invite-id` and allows the creation if so.
   The hook still rejects any sign-up that arrives without either being
   the first admin or carrying a valid invite-id.
3. Insert `member(user_id, slug, role)` from the invite.
4. `UPDATE invite SET redeemed_at = now, redeemed_by = user.id WHERE
   token_hash = ?`. The single-use property comes from `redeemed_at`
   being non-null on a second attempt; the action's transactional
   re-check (step 1) closes the race.
5. Set the session cookie (better-auth's standard flow).
6. 302 to `/admin/spaces/[slug]`.

If any step fails after the user row was created (DB I/O error between
steps 2 and 4, say), the transaction rolls back. The better-auth API
calls run inside this transaction by sharing the singleton DB handle.

**Google sign-in path** uses the same state-token pattern subsystem 2
introduced for `/admin/setup` Google claim. The state is opaque to
better-auth; subsystem 4's callback intercepts it. The state encodes the
invite-id (NOT the token); the callback re-validates the invite
(by id, server-side) before letting the user-create hook through.

### Revocation

Owners (and install-admin) revoke a pending invite from the members
page: `DELETE FROM invite WHERE id = ?`. The URL becomes 410 Gone on the
next click. There is no "regenerate" — revoke and create a fresh one.

### Expiry cleanup

A boot-time sweep deletes invites where `redeemed_at IS NOT NULL` AND
`redeemed_at < now - 30 days`, and where `expires_at < now - 30 days`.
The 30-day buffer keeps recent history visible to operators reading the
table by hand, and bounds the table's growth. No scheduled tick; the
sweep runs on boot only. This is the same posture as the render-cache
vacuum (`lib/space/cache.ts`).

### What's deliberately not in the invite

- **A specific email the invite is bound to.** Bearer URL; the invitee
  picks their own email at redemption. (User answered this in the
  brainstorm.)
- **A "max uses" count.** Single-use only.
- **A "this invite also grants you these other spaces" payload.**
  One invite, one space, one role. Multi-space invites would compose
  trivially (issue several invites and send them as a list); not worth
  designing for here.

## 5. Storage — `auth.db` additions

Three changes to the install-root `auth.db`:

### 5.1. `isInstallAdmin` on `user`

A new column. Boolean (SQLite `INTEGER`), `DEFAULT 0`, `NOT NULL`. The
setup action sets it to 1 for the row it creates. The invite-redemption
action and the user-create hook leave it at 0.

### 5.2. `member` table

```sql
CREATE TABLE member (
  id          TEXT PRIMARY KEY,           -- ULID or uuidv7; opaque to the app
  user_id     TEXT NOT NULL,              -- FK to user.id; not enforced (better-auth's Kysely doesn't ship with FKs)
  space_slug  TEXT NOT NULL,              -- the directory name; matches the slug regex
  role        TEXT NOT NULL CHECK (role IN ('owner', 'editor')),
  created_at  INTEGER NOT NULL,           -- unix ms
  created_by  TEXT,                       -- user.id of inviter, or NULL when granted via CLI
  UNIQUE (user_id, space_slug)
);

CREATE INDEX member_by_user  ON member(user_id);
CREATE INDEX member_by_space ON member(space_slug);
```

The `UNIQUE (user_id, space_slug)` constraint guarantees one role per
(user, space). Role changes are an `UPDATE`, not an insert. Removing a
member is a `DELETE` by row id.

### 5.3. `invite` table

```sql
CREATE TABLE invite (
  id          TEXT PRIMARY KEY,           -- ULID
  token_hash  TEXT NOT NULL UNIQUE,       -- SHA-256(token), hex
  space_slug  TEXT NOT NULL,
  role        TEXT NOT NULL CHECK (role IN ('owner', 'editor')),
  expires_at  INTEGER NOT NULL,           -- unix ms
  created_at  INTEGER NOT NULL,
  created_by  TEXT NOT NULL,              -- user.id of inviter
  redeemed_at INTEGER,                    -- nullable; non-null = consumed
  redeemed_by TEXT                        -- nullable; user.id of redeemer
);

CREATE INDEX invite_by_space     ON invite(space_slug);
CREATE INDEX invite_by_expires   ON invite(expires_at);
```

Lookup by token uses `WHERE token_hash = ?` on the unique index.

### Migration mechanics

Better-auth runs its own Kysely migrations on boot from
`auth-config.ts`. Subsystem 4's migrations are **applied after**
better-auth's, by an `applyAmberAuthMigrations()` call that runs once
after `runMigrations()` in `getAuth()`. The mechanism: a small
`amber_migrations` table tracking applied migration ids, and a hand-written
list of SQL strings. This is `bun:sqlite` direct execution, no migration
framework — matches subsystem 2's "no ORM, no abstractions we don't need"
posture.

If `amber_migrations` shows a version newer than the running build, boot
fails with a clear "auth-DB schema is newer than this Amber build" error
— same pattern subsystem 2 documented for better-auth's own schema.

### Why both tables in `auth.db` and not per-space TOML

The "filesystem is truth" rule applies to content. User identity is
install-level (`auth.db` is the established install-root exception).
Permissions reference `user.id`, so they're naturally install-level too.
Per-space `members.toml` would either duplicate IDs (drift risk) or
reference email strings (a different rename problem). The cleaner shape
is: one DB at the install root, all identity and permissions live there.

Subsystem 4 does not revise the `space.toml` schema, and does not add any
new `LoadWarning` codes that affect the loader. Two new auth-side warning
codes appear (§7) but those flow through the auth logger, not the
loader's warning enum.

### Orphan rows

If a space directory disappears (operator `mv`'d or deleted it), the
corresponding `member` and `invite` rows become orphans (no matching
slug in `getRegistryEntries()`). Subsystem 4's posture:

- **Never auto-delete** orphans. The operator may be moving spaces
  between installs, or has temporarily renamed one.
- **Emit a boot-time log line** per orphan slug:
  `subsystem: 'permissions', code: 'orphan_membership', slug, count`
  and `'orphan_invite'` likewise. Info-level. One line per slug, not per
  row.
- The orphan rows are not exposed in any UI list (the picker filters by
  loaded spaces; orphan rows don't appear). The members page is reached
  only via a valid slug, so orphan rows are unreachable there too.

If the directory reappears under the same slug, the rows re-attach
automatically — they were keyed by the slug all along. This is the
"filesystem is truth, DB is best-effort" stance applied to permissions.

## 6. The auth seam — what changes

### `lib/server/auth.ts` — unchanged

`requireAuthor()` and `isAuthor()` keep the same signatures and
behaviour. They answer "is this a signed-in admin user at all?" — the
right question for routes that don't have a space yet (`/admin/users`,
`/admin/account`).

### New module: `lib/server/permissions.ts`

Owns `requireSpaceAccess()`, `canEdit()`, `canRead()`, and the small
helpers around the `member` table (`getRole`, `listMembers`,
`upsertMember`, `removeMember`). The members/invite admin routes and the
CLI both go through this module — never raw SQL — so the access-control
logic has one home.

### `auth-config.ts` — user-create hook extended

The hook stays at the same insertion point (`databaseHooks.user.create.before`).
It currently checks "is this the first user?" and rejects otherwise.
Subsystem 4 extends it to:

```
if no users exist → allow (setup flow)
else if a valid pending invite is in context → allow
else → reject
```

"In context" means the action that called `auth.api.signUpEmail()`
attached a `pending-invite-id` to a thread-local (or, equivalently,
passed it through an option that the hook can read). Better-auth's hook
signature does receive the calling context; the exact API will be picked
in the plan. The hook performs a re-fetch by id to confirm the invite is
still valid at the moment the user row would land. This closes the
"invite was valid when load ran but is now expired" race.

### `routes/[...path]/+page.svelte` — the inline edit link

The "Edit this page" link's visibility check changes from:

```svelte
{#if isAuthor(...)}
  <a href={...}>Edit this page</a>
{/if}
```

to:

```svelte
{#if canEdit(locals.user, locals.space)}
  <a href={...}>Edit this page</a>
{/if}
```

`canEdit()` runs a single `member` query for non-install-admin users.
Cached on `locals.role` if the request also hit a per-space admin layout
(it doesn't, in the public render path — they're different route
groups), so on the public render path it's a fresh lookup per request.
Acceptable: it's the same posture as `isAuthor()` today (one DB lookup
in `hooks.server.ts`), with one additional row read.

### `/admin/spaces/[slug]/+layout.server.ts` — gains role resolution

The existing slug-resolution code stays. After it sets `locals.space`,
it calls `requireSpaceAccess(event, params.slug)` and stashes the result
on `locals.access` and `locals.role` for child routes.

### `/admin/+page.server.ts` — picker filtered

The picker load filters `getRegistryEntries()` to the user's accessible
spaces (install-admin: all; otherwise: spaces with a `member` row).
Single-space 302 remains, against the *filtered* list.

## 7. Warning / error codes

Two new codes emitted via the `subsystem: 'permissions'` pino logger:

- `orphan_membership` (info) — boot-time scan found `member` rows whose
  `space_slug` doesn't match any loaded space. Emitted once per slug,
  with row count. Never auto-deleted.
- `orphan_invite` (info) — same but for `invite` rows.

Two new mutation-event codes, also pino-only, at info level:

- `invite_generated` — `{ actorId, slug, role, inviteId }`.
- `invite_redeemed` — `{ inviteId, userId, slug, role }`.
- `invite_revoked` — `{ actorId, inviteId, slug }`.
- `member_added` — `{ actorId, userId, slug, role, via: 'invite' | 'cli' }`.
- `member_role_changed` — `{ actorId, userId, slug, from, to }`.
- `member_removed` — `{ actorId, userId, slug }`.
- `user_deleted` — `{ actorId, userId }`.

No additions to the loader's `LoadWarning` enum. Permissions live in
the auth layer; the loader stays untouched.

## 8. Configuration — env-var surface

**No new environment variables.** Subsystem 4 inherits every var
subsystem 2 introduced (`AMBER_AUTH_SECRET`, optional Google pair,
`AMBER_PUBLIC_URL`) and every var subsystem 3 introduced
(`AMBER_SPACE_PATH` / `AMBER_SPACES_DIR`).

The "invite TTL is 7 days, fixed" decision is what keeps the env surface
flat. If a future operator pushes back, an `AMBER_INVITE_TTL_DAYS` is a
trivial addition; not in v0.5.

## 9. Concurrency and edge cases

**Invite redemption race.** Two browsers open the same invite URL.
Browser A submits first, the action transactionally creates the user
and marks `redeemed_at`. Browser B submits a moment later; the re-fetch
inside its transaction sees `redeemed_at` non-null and returns 410. The
single-use property holds. (Same shape as subsystem 2's setup-race
re-check.)

**Invite expires between load and action.** The load rendered the form;
by the time the user clicked submit, expiry passed. The action's
transactional re-check returns 410. The user sees the 410 page; the
operator re-issues.

**Role change while the user is editing.** Owner X demotes editor Y
mid-edit. Y's session is *not* revoked (user choice). Y's next request
to `/admin/spaces/[slug]/api/page/[...path]` (PUT) goes through
`requireSpaceAccess(event, slug, 'editor')`; the role is still editor,
so the save succeeds. The risk we accept: a freshly-removed-then-re-
added editor between PUT requests is a no-op race, not a security
issue.

**Owner removes themselves.** Allowed. They lose access to the space.
The space's other owners (or the install-admin's implicit ownership) is
unaffected. If they were the only explicit owner, the space falls back
to install-admin-only — the install-admin can re-grant ownership via
the CLI or by re-inviting.

**Install-admin tries to remove themselves from a space.** They have no
explicit `member` row to remove. The action is a no-op with a clear
message ("you are the install-admin; ownership is implicit"). The
members page does not render their row.

**Self-delete from `/admin/account`.** Cascades: all of the user's
`member` rows are deleted, all of their sessions are revoked, the
`user` row and `account` rows (better-auth's term, not Amber's) are
deleted. Better-auth's API has the call. Install-admin is blocked from
this path.

**Last-owner removal.** A space with one explicit owner removes them.
The members page shows zero owners. The space is still reachable by the
install-admin (implicit) and editable by remaining editors (who keep
their edit capability). This is intentional: install-admin is always
the floor.

**Slug renamed under the install-admin's feet.** Operator `mv`s a space
directory while the server is running. Memberships keyed by the old
slug become orphans (see §5). The watcher doesn't re-resolve memberships
because the watcher doesn't touch `auth.db` (subsystem 2 invariant
preserved). The orphan log fires on the next boot. Editors with a
membership on the old slug lose access until they're re-invited under
the new slug, or the operator runs the CLI grant.

**A user whose only auth method is an unlinked Google account, removed.**
Their `account` rows still exist; their `member` rows are gone. They can
still sign in but see the empty-picker. If the install-admin later
re-invites them with the same email, better-auth's email-uniqueness
catches the collision; the operator either re-uses the existing user
(by inviting their email — the user-create hook's "is there a pending
invite?" path needs to *find* an existing user before creating, see plan)
or deletes the stale row first.

**User-create hook + existing user.** When a non-existent-user redeems
an invite, the flow is "create + member-insert." When an *existing* user
(say, removed and re-invited) opens an invite URL while signed out, the
redemption form lets them sign in instead of signing up; on success the
existing user gets a fresh `member` row. The redemption load checks for
this case (form has a "I already have an account, sign in" affordance);
the plan picks the cleanest UX.

- **Invite round-trip** — install-admin signs in, navigates to

## 13. Risks and open questions

### Risk: the user-create hook's "is there a pending invite?" check

Better-auth's hook signature is what it is. The plan needs to confirm
the cleanest way to thread invite-id context into the hook — options
are (a) a module-scoped `AsyncLocalStorage` set by the action,
(b) passing the invite-id through better-auth's session.cookies API
extensions, or (c) doing the user-row insertion ourselves and bypassing
`signUpEmail()`. Option (a) is the closest to how the setup action
already shapes its context; the build spike should land it first and
prove the hook resolves the right invite per-request.

### Risk: install-admin's implicit ownership across new spaces

Adding a space by `mkdir` + restart is the current pre-subsystem-5
path. The install-admin auto-owns it because `requireSpaceAccess()`
short-circuits on the flag, not on a row. Edge case: an operator who
manually edits `auth.db` to clear `isInstallAdmin` and forgets to
materialize the implicit ownerships into rows will silently lose access
to every space. The CLI `grant-ownership` is the documented recovery,
but the *demote-install-admin* path doesn't go through it — there's no
UI for demotion. Subsystem 4's posture: don't expose demotion in the
UI; if an operator pokes `auth.db` directly they own the consequence.
Documented in `docs/self-hosting.md`.

### Risk: a removed editor mid-PUT-save

Covered in §9 — accepted; the alternative (revoke sessions on role
change) was the user-rejected option. Documented here so the build
doesn't accidentally re-add session-revocation.

### Open question: should the members page show the install-admin row?

A user reading the members page on a space sees the explicit `member`
rows. The install-admin's implicit ownership is *invisible* there. This
is the user-facing surface of "install-admin is implicit, never a row."
The members page could synthesize a "Install admin (—)" entry at the
top so editors know the operator exists. Decision deferred to the build
— pure UI choice, no architectural weight either way.

### Open question: invite URL host when `host`/`prefix` is set on a space

Invite URLs are minted with `AMBER_PUBLIC_URL` as their base because the
`/admin/invite/[token]` route lives on the admin host (it's an
authenticated-creation route, not a content route). A space with a
declared `host` does NOT mint invites under its own host. Subsystem 3's
admin-host-collision rule (`space_routing_admin_host_collision`)
already protects against the converse. The plan documents this in the
"how to share an invite URL" copy.

### Open question: the picker's "I have zero spaces" copy

Pure UX. The picker load knows whether the empty list is because the
user is new (no memberships) or because the install is empty (no
spaces loaded). The copy differs slightly:

- No memberships, install has spaces → "You don't have access to any
  spaces yet. Ask your administrator to invite you."
- Install has zero spaces → "No spaces loaded. Ask your administrator
  to add one."

Decided in the build.

## 14. Build order

Risk-first, matching subsystem 1–3's discipline:

1. **Schema + permission seam.** The `applyAmberAuthMigrations()`
   runner, the `isInstallAdmin` column on `user`, the `member` and
   `invite` tables, and `lib/server/permissions.ts` with
   `requireSpaceAccess` / `canEdit` / `canRead`. Unit tests against a
   throwaway DB. No route changes yet. This is the spike: if the
   seam shape doesn't fit better-auth's hook surface cleanly, the
   design is reconsidered before any route lands.
2. **`/admin/spaces/[slug]/+layout.server.ts` gains role resolution.**
   The existing tests still pass because install-admin short-circuits
   (subsystem 3's e2e suite signs in as the install-admin); the layout
   stash adds `locals.role`. The PUT save endpoint and the editor /
   new routes start reading `locals.role`.
3. **The picker filter + public render path probe.** `/admin/+page.server.ts`
   filters the list. `routes/[...path]/+page.svelte` switches the edit
   link to `canEdit()`.
4. **Invite generation + redemption.** `/admin/spaces/[slug]/members`
   for owner-side actions; `/admin/invite/[token]` for the public
   claim route. User-create hook extended. Google-callback invite
   state if the install has Google configured (testable with stubs).
5. **`/admin/users` install-admin surface, `/admin/account` delete-
   self, CLI `grant-ownership`.** The last three pieces that don't
   block any earlier piece.
6. **CLAUDE.md revisions + roadmap closure.** Lands in the final
   commit.

Each step is a single commit. Step 1 is the only one that gates the
rest — if better-auth's hook surface can't carry invite-id context
cleanly, steps 2–6 wait while the design is revisited.
