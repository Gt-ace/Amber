# Theme asset cache-busting + live theme previews

Status: in progress. Branch: `worktree-theme-cache-bust-and-previews`.

Two focused changes to the v0.5 theming surface: a correctness fix for a
stale-cache bug, and a usability feature (live previews in the theme picker).

## Problem 1 — stale theme assets (the reported bug)

Observed: "select one theme but the font/background of another theme shows;
a hard refresh fixes it." Confirmed by reproduction (single-space,
`AMBER_PUBLIC_URL=localhost`).

Root cause: theme assets are served by
`routes/themes/[name]/[...file]/+server.ts` with
`Cache-Control: public, max-age=3600` at a URL (`/themes/<name>/theme.css`)
that **does not change when the theme's bytes change**. So when a theme's
`theme.css` is edited in place (or the file a name points at otherwise
changes), the browser keeps serving the previous bytes for up to an hour —
the page shows a previous theme's font/colors until a hard refresh. The
route's own header comment already flags the gap: the files "don't carry
content-hashed names, so `immutable` / year-long `max-age` would be wrong."

Confirmed *not* the cause (ruled out by reproduction): the SQLite render
cache (theme-independent — `getOrRenderHtml` keys on body bytes only,
`pageRenderCacheKey` folds in template source), the picker post-redirect-get
race (resolves in ~50 ms, faster than a redirect round-trip), and the
`data-amber-theme` light/dark attribute (it only changes color *mode*, never
which theme loads — a hard refresh wouldn't fix it).

### Fix

Version the asset URL so it changes whenever the bytes change:

- `lib/space/themes.ts`: add `themeAssetVersion(theme, file)` — returns the
  file's `mtimeMs` (base-36) via `statSync(join(theme.path, file))`, or `'0'`
  when the path is empty (built-in theme) or the stat fails. One `stat` per
  page load is cheap and reflects edits, `git pull` checkouts, and image
  rebuilds. Switching themes already changes the `<name>` segment, so this
  covers both edit-in-place and switching.
- `routes/+layout.server.ts`: append `?v=<version>` to `themeCssHref` and
  `themeJsHref`.
- `routes/themes/[name]/[...file]/+server.ts`: when the request carries a
  `v` query param, serve `public, max-age=31536000, immutable` (the URL is
  now content-versioned, so a long immutable cache is correct and faster);
  without `v`, keep `public, max-age=3600` for un-versioned direct hits and
  `@font-face` URLs.

Out of scope: `@font-face` URLs inside a theme's CSS (e.g. amber-brand's
`/fonts/*.woff2`) are not versioned. Fonts are effectively immutable and not
implicated in the reported symptom (which font *family* shows is decided by
which `theme.css` loads, now correctly busted).

## Problem 2 — no preview in the theme picker

The picker (`/admin/spaces/[slug]/theme`) lists themes by name + metadata
only; an operator must save then open the public site to see a theme. Add a
**live mini-render** preview per theme.

### Design

- `theme/+page.server.ts` load builds, per discovered theme, a
  self-contained preview document using the *same* primitives the real
  render path uses:
  - `readTemplate(theme, 'chrome')` split at `CONTENT_SLOT`, with
    `renderTemplate` filling a fixed sample site title + nav,
  - `readTemplate(theme, 'page')` rendered with a sample title/date and a
    fixed markdown body run through `render()` (heading, paragraph with a
    link and inline code, a short list, a blockquote),
  - a `<link rel="stylesheet" href="/themes/<name>/theme.css?v=<version>">`
    in the document `<head>` (versioned, same as the public path).
  The assembled HTML string is returned per theme as `previewHtml`.
- `theme/+page.svelte`: each theme option renders its `previewHtml` in a
  `<iframe sandbox srcdoc={previewHtml} loading="lazy" tabindex="-1">`,
  scaled down (CSS `transform: scale`) to a small card and
  `pointer-events: none`. The `sandbox` attribute carries no `allow-*`
  tokens — the preview runs no scripts and needs none; a same-site
  `<link>` stylesheet still loads in an opaque-origin srcdoc iframe.

Sample content and the preview-document assembly live in a small helper
(`lib/space/theme-preview.ts`) so it is unit-testable independently of the
route.

### Notes / edges

- Built-in fallback theme (`path === ''`): previewable via `readTemplate`'s
  built-in branch; its `assetBase` may be null, in which case the preview
  omits the stylesheet link (renders unstyled — acceptable; it is the floor).
- Multi-space: the picker is an install-level `/admin` surface; the
  preview's `/themes/<name>/theme.css` request resolves install-level
  (shared themes) the same way the asset route already documents. Per-space
  *override* previews under a non-default space are a known minor edge, out
  of scope here.

## Test plan

- `themeAssetVersion`: returns mtime-derived string for a real file; `'0'`
  for empty path / missing file; value changes after a write.
- Asset route: `?v=...` → `immutable` long max-age; no `v` → `max-age=3600`.
- `+layout.server.ts`: `themeCssHref` / `themeJsHref` carry `?v=`.
- `theme-preview.ts`: assembled document contains the versioned stylesheet
  link, the sample heading/link, and is a single well-formed `<html>` doc.
- Picker load returns `previewHtml` per theme.
- Visual: Playwright render of the picker shows distinct fonts/colors per
  preview iframe; reproduction harness confirms an edited `theme.css` serves
  fresh bytes (new `?v=`) without a hard refresh.
