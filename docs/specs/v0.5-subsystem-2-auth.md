# v0.5 subsystem 2 — auth (better-auth) — spec

Status: **agreed design**, ready for a plan session.
Input: the auth seam shipped in subsystem 1 (`apps/web/src/lib/server/auth.ts`),
which named subsystem 2 as the module that fills in real session checking and
removes the interim `AMBER_DEV_UNSAFE=1` branch. This spec turns that promise
into a buildable design.

The spec answers *what exactly are we building*. It does not write
implementation code and does not revise `CLAUDE.md` — it names the revisions;
the build makes them.

---

## 1. What this subsystem is

Real authentication for the `/admin` authoring surface, backing the
`requireAuthor()` / `isAuthor()` seam that subsystem 1 wired through every
admin route. The seam's call sites do not change; only this module's
internals do. The interim `AMBER_DEV_UNSAFE=1` flag is **removed**.

Two sign-in methods, both natively supported by [better-auth][ba]:

- **Email + password** — the primary method. Works offline, no external
  services, familiar.
- **Google OAuth** — optional, enabled when the operator sets a
  client ID / secret pair. Doubles as a password-recovery path.

The first admin is claimed through a one-time `/admin/setup` screen on a
fresh install. Once any admin exists, setup is locked. Subsystem 2 is
**single-admin** — multi-user invites and per-space permissions are
subsystem 4 and not designed here.

[ba]: https://www.better-auth.com

### Non-goals

Explicitly out of scope for subsystem 2. Each is either a later subsystem,
a later increment, or deliberately not Amber's job:

- Multi-user, invited authors, per-space permissions — subsystem 4.
- 2FA, passkeys/WebAuthn, magic links, login throttling beyond
  better-auth's defaults, password complexity policies.
- Email-based password reset. Amber has no SMTP; recovery is linked Google
  or the CLI (§7).
- An audit log of admin actions, or a session-listing / device-management
  UI (better-auth supports the latter; deferred).
- Brute-force / rate-limit protection in the app. That belongs at the
  reverse proxy (Caddy) if anywhere — out of scope here.
- Migration tooling from a hypothetical prior auth system. Subsystem 2 is
  the first auth Amber has ever had.

## 2. Surface and routes

All admin auth lives under `/admin` to keep one chrome, one mental model.
Two route groups split the auth posture:

| Group | Route | Auth required? |
|---|---|---|
| `(public)` | `/admin/login` | No — login form. |
| `(public)` | `/admin/setup` | No, but 404s once any admin row exists. |
| `(authed)` | `/admin` | Yes — page-list index. |
| `(authed)` | `/admin/edit/[...path]` | Yes. |
| `(authed)` | `/admin/new` | Yes. |
| `(authed)` | `/admin/account` | Yes — change password, link/unlink Google, sign out. |
| `(authed)` | `/admin/api/page/[...path]` | Yes (returns 401 JSON, never a redirect). |
| (root) | `/api/auth/[...path]` | Better-auth's handler — sign-in, sign-out, OAuth callbacks. Authenticates itself. |

Route groups are SvelteKit's native pattern for "shared chrome, different
auth posture." The `(authed)/+layout.server.ts` is the single place the
redirect-to-login decision happens; the `(public)` layout has no guard.
`requireAuthor()` keeps doing one thing — throw 401 — and the page-route
layout decides whether to translate that to a redirect.

**Sign-out** is `POST /api/auth/sign-out` (better-auth's standard route).
The admin chrome gets a small "Sign out" affordance that posts to it and
follows the redirect back to `/admin/login`.

## 3. Bootstrap — claiming the first admin

A fresh `docker compose up` against an empty `.amber/auth.db` boots an
Amber with zero admin rows. The behavior:

1. `GET /admin` → the `(authed)` layout sees no session → 302 `/admin/login`.
2. `GET /admin/login` → the load checks the admin count → 0 admins → 302
   `/admin/setup`.
3. `GET /admin/setup` → renders the claim form. The form offers
   email+password and (if Google is configured) "Continue with Google."
4. The form action **re-checks the admin count inside its DB transaction**
   before inserting. If the count became ≥ 1 between the GET and the POST
   (TOFU race), the action returns 409 "Setup is already complete" with a
   link to `/admin/login`.
5. On success: admin row inserted, session cookie set, 302 `/admin`.

Once an admin exists, `/admin/setup` returns 404 — the route is dead. This
is the Ghost / Gitea / Vaultwarden pattern: trust on first use, narrow
window, sealed by a database row.

The Google-only setup path uses the standard OAuth dance with a setup
state token: the callback handler detects the setup-state, creates the
admin if and only if zero admins exist, and otherwise falls back to the
"this Google account isn't the admin" 403 (§5).

## 4. Sign-in flows

### Email + password

```
POST /admin/login   (form action posts to better-auth via internal fetch)
  → better-auth verifies password hash
  → session row inserted, session cookie set
  → 302 to validated ?next= or /admin
```

Wrong password → inline form error, no information leak about whether the
email exists. The single-admin shape makes that distinction largely
moot (the operator knows their own email) but the principle stands.

### Google OAuth

```
GET  /api/auth/sign-in/social/google
  → 302 to Google's authorize URL
GET  /api/auth/callback/google?code=...
  → better-auth exchanges the code
  → admin exists & this Google sub is linked → sign in, 302 to ?next= or /admin
  → admin exists & this Google sub is NOT linked → 403, log at info
  → no admin exists & setup-state present → create admin, link Google, sign in
  → no admin exists & no setup-state → 403 "this Google account isn't the admin"
```

The "create on first social sign-in" default in better-auth is **disabled**
for the Google provider; the only sign-up paths are the setup screen and
the account-linking flow (§6). A stranger signing in with their own Google
account must never silently get an account.

### Session-locals invariant

`event.locals.user` is populated by the root `hooks.server.ts` (better-auth's
recommended pattern), **not** the admin layout. This means:

- The public render path's `isAuthor()` check is one DB lookup per request,
  in the hook, not per call to `isAuthor()`.
- The "Edit this page" link in `routes/[...path]/+page.svelte` works
  exactly as it does today, with the same code path.

### `?next=` validation

The post-login redirect parameter is validated as a same-origin **path**
only: it must start with `/` and not start with `//` (which would let an
attacker write `//evil.example.com/`), must not contain a scheme, and is
URI-decoded once before validation. Invalid `next` falls back to `/admin`.
Same validation in `/admin/setup` and the OAuth callback.

## 5. Storage — `.amber/auth.db`

A new SQLite file at `.amber/auth.db`, separate from `.amber/cache.db`.
Better-auth's SQLite/Kysely adapter owns its schema; its migrations run on
boot. The tables are better-auth's defaults: `user`, `account` (for OAuth
links), `session`, `verification`.

The CLAUDE.md rule that says "`.amber/` is regenerable runtime state, safe
to delete" is **narrowed** by this subsystem (§12): `cache.db` is
regenerable; `auth.db` is not. The existing compose volume already covers
both files; operators backing up the space directory pick up `auth.db`
automatically.

Schema drift handling: if `auth.db` exists with a schema version newer than
the running Amber recognizes, boot fails with a clear error
("auth-DB schema is newer than this Amber build; upgrade Amber or restore
an older `auth.db`"). Better-auth's migrations are forward-only.

The watcher does not touch `.amber/*.db` (it never has). No interaction;
called out here because the editor PR taught us to be explicit about it.

## 6. The auth seam — what changes inside `lib/server/auth.ts`

The exports stay identical. Internals are rewritten:

```ts
// Before (interim, subsystem 1):
export function isAuthor(_event): boolean { return process.env.AMBER_DEV_UNSAFE === '1'; }

// After (subsystem 2):
export function isAuthor(event): boolean { return event.locals.user != null; }
```

`requireAuthor()` becomes:

```ts
export function requireAuthor(event): void {
  if (isAuthor(event)) return;
  error(401, 'Unauthorized — the Amber admin requires authentication.');
}
```

— same signature as today. The `/admin/(authed)/+layout.server.ts` wraps
the throw with a redirect for page routes; the PUT endpoint keeps the
401 it returns today.

The `AMBER_DEV_UNSAFE` env flag, the loud-boot warning, and the test
block that covers them are **all deleted**. No deprecation period — the
flag was always documented as "removed by subsystem 2."

### Account management (`/admin/account`)

A single page behind the auth guard, offering:

- **Change password.** Requires the current password. On success, all
  other sessions for this user are revoked (force re-login on other
  devices — the standard recovery posture).
- **Link Google.** Starts the Google OAuth dance with an "account-linking"
  state token. The callback links the Google sub to the current user's
  `account` rows. Already-linked → no-op with a success message. Linking
  a Google sub that's already linked to a *different* user errors (can't
  happen with one admin, but the invariant is enforced).
- **Unlink Google.** Blocked if no password is set ("Set a password first,
  otherwise you'd be locked out"). The one-admin invariant: an admin must
  always have at least one usable auth method.
- **Sign out.** Posts to `/api/auth/sign-out`.

## 7. Recovery

Two paths, in order of expected use:

1. **Linked Google.** If the admin linked Google at setup or in account
   settings, forgot-password = sign in with Google, then change password
   from `/admin/account`. No extra UI needed beyond the existing login
   button.
2. **CLI reset.** `bun run --cwd apps/web bin/reset-password.ts --email <x>`
   opens `.amber/auth.db` directly, writes a new password hash (using
   better-auth's hashing helper so the format matches the runtime), and
   revokes all existing sessions for that user. Prints the temporary
   password to stdout once. The operator changes it from `/admin/account`
   after signing in.

The CLI documents itself in `docs/self-hosting.md`. It is the deliberate
escape hatch for the AGPL self-hoster who already has shell access — no
in-app "forgot password" flow exists, by design.

There is no other recovery story. Lose the password, lose access to
linked Google, lose shell access → restore from backup, or wipe
`.amber/auth.db` and re-bootstrap. The smallest acceptable surface area.

## 8. Configuration — env-var surface

Three new variables, one removed:

- **`AMBER_AUTH_SECRET`** — required. Random secret used by better-auth to
  sign session cookies. No default. App **refuses to boot** with a clear
  error if it is unset. The example `compose.yaml` shows a placeholder
  value plus a comment telling the operator to replace it; `compose.prod.yaml`
  threads it through from the operator's `.env`.
- **`AMBER_GOOGLE_CLIENT_ID`** / **`AMBER_GOOGLE_CLIENT_SECRET`** —
  optional, but **all-or-nothing**. Both set → the Google plugin is
  registered, the Google button appears on `/admin/login`, `/admin/setup`,
  and `/admin/account`. Neither set → Google is invisible. One without the
  other → fatal boot error ("Google OAuth requires both env vars or
  neither"). Half-configured OAuth is a misconfiguration, not a feature.
- **`AMBER_PUBLIC_URL`** — already exists for the sitemap; reused as the
  OAuth callback base. The documentation tells the operator to register
  `<AMBER_PUBLIC_URL>/api/auth/callback/google` in Google Cloud Console.
- **`AMBER_DEV_UNSAFE`** — **removed**. The flag is no longer read
  anywhere. Mentioned in the release notes / migration guide as a
  one-shot removal: operators who were running with it set need to
  bootstrap a real admin.

## 9. Concurrency and edge cases

**Setup TOFU race** — the `/admin/setup` action re-checks the admin count
inside the DB transaction that would insert the admin (§3). The window
between deploy and the operator claiming the admin is the residual risk;
the documentation tells operators to claim immediately after `docker compose up`.

**Session deleted out-of-band** (e.g., by the reset CLI, or expiry) → the
cookie is still presented by the browser; better-auth fails to resolve a
session for it; `isAuthor()` returns false. The `(authed)` layout
redirects to `/admin/login?next=...`; the PUT endpoint returns 401 JSON.
No special case needed in the seam.

**User row deleted out-of-band** (someone ran SQL against `auth.db`) →
same as above. The cookie's session row is orphaned; better-auth treats
it as invalid.

**Clock skew / past expiry** — better-auth handles. Default session
lifetime: 30 days, refreshed on use. Not retuned here.

**Failed OAuth callback** (user denied, network error, code reuse) →
return to `/admin/setup` or `/admin/login` (whichever started the dance)
with an inline error; no rows are created — better-auth's callback is
transactional.

**Account-linking against an already-linked sub** — blocked, errors
clearly. Can't happen with one admin, but the code still checks.

## 10. Components

| Unit | Responsibility |
|---|---|
| `lib/server/auth.ts` | Same exports (`requireAuthor`, `isAuthor`). Internals read `event.locals.user`. The `AMBER_DEV_UNSAFE` branch is **deleted**. |
| `lib/server/auth-config.ts` | Constructs the `betterAuth(...)` instance: SQLite adapter pointed at `.amber/auth.db`, email+password plugin, Google plugin (registered only when both env vars present), session/cookie config. Boot-time check for `AMBER_AUTH_SECRET`. |
| `lib/server/auth-db.ts` | Opens `.amber/auth.db` via `bun:sqlite`, runs better-auth's migrations on boot, exposes the Kysely instance. Separate from `lib/space/cache.ts` so the two SQLite handles never tangle. |
| `lib/server/next-validator.ts` | Pure helper: validates the `?next=` parameter as a same-origin path; falls back to `/admin`. Unit-tested in isolation. |
| `hooks.server.ts` | Resolves the session cookie once per request; populates `event.locals.user`. Better-auth's recommended pattern. |
| `routes/api/auth/[...path]/+server.ts` | Better-auth's standard handler. No custom code. |
| `routes/admin/(authed)/+layout.server.ts` | The single page-route auth gate: not authenticated → 302 `/admin/login?next=<url>`. Endpoints reach this via their own `requireAuthor()` call. |
| `routes/admin/(public)/login/+page.server.ts` / `.svelte` | Email+password form + Google button (conditional on env). Posts to better-auth. |
| `routes/admin/(public)/setup/+page.server.ts` / `.svelte` | First-run claim. Load 404s when an admin exists; action re-checks under transaction. |
| `routes/admin/account/+page.server.ts` / `.svelte` | Change password, link/unlink Google, sign out. Enforces the one-admin invariant on unlink. |
| `routes/admin/+layout.svelte` | Add a "Sign out" affordance to the admin chrome. |
| `apps/web/bin/reset-password.ts` | CLI escape hatch. Opens `auth.db` directly; writes a hash; revokes sessions; prints a temporary password. Documented in `docs/self-hosting.md`. |

The PUT save endpoint, the editor, the new-page form, and the inline
"Edit this page" link on the public render path **do not change**. They
already call `requireAuthor()` / `isAuthor()` and that's the seam.

## 11. Testing

Per `apps/web/CLAUDE.md`, pure functions get unit tests and route modules
get at least one runtime smoke. The auth seam carries more value at the
boundaries than usual, so the smoke layer carries more weight than usual.

### Unit (`lib/server/*.test.ts`)

- **`auth.test.ts`** (rewritten): `isAuthor()` and `requireAuthor()` against
  a stubbed `event.locals.user`. The `AMBER_DEV_UNSAFE` test block is
  **deleted** — the flag is gone.
- **`auth-config.test.ts`** (new): import the config factory with env
  mocked: no secret → throws; half-configured Google → throws; both
  Google vars present → Google plugin appears in the assembled config;
  neither set → it doesn't.
- **`next-validator.test.ts`** (new, small): same-origin paths pass;
  protocol-relative (`//evil.example.com`), absolute (`https://...`),
  and `javascript:` are rejected. Decoded-once before validation.

### Route handlers (Vitest + direct import, like subsystem 1)

- **`setup/setup.test.ts`** — setup load returns the form when 0 admins;
  returns 404 when ≥ 1 admin exists; action creates row + session on
  valid input; 409 on race (insert an admin between `beforeEach` and the
  action call); inline errors on invalid input. Throwaway `.amber/auth.db`
  per test via `mkdtemp`.
- **`login/login.test.ts`** — action signs in with valid password; rejects
  wrong password with no user-existence leak; `?next=` open-redirect
  attempts fall back to `/admin`.
- **`(authed)/+layout.server.test.ts`** — redirects to
  `/admin/login?next=...` when unauthenticated. Verifies the
  `requireAuthor` signature is identical to subsystem 1's (the seam
  promise).
- **`account/account.test.ts`** — unlinking the only remaining auth
  method is blocked; changing password revokes other sessions; linking
  an already-linked Google sub on a different user errors.
- **`apps/web/bin/reset-password.test.ts`** — writes a hash matching
  better-auth's format, revokes existing sessions, exits non-zero on
  unknown email.

### Route-module smoke (runtime shape)

Per the `apps/web/CLAUDE.md` rule that SvelteKit validates module shape
at request time only: each of `(public)/login/+page.server.ts`,
`(public)/setup/+page.server.ts`, `account/+page.server.ts`, and
`api/auth/[...path]/+server.ts` gets one route-module smoke (`vite preview`
or an `@sveltejs/kit/test` shaped test, matching whatever the build picks).

### E2E (`admin.e2e.test.ts`, extended — `AMBER_E2E=1`)

- **First-run** — fresh `.amber/auth.db`, navigate to `/admin`, expect
  redirect to `/admin/setup`, fill in email+password, submit, expect to
  land in `/admin`.
- **Login round-trip** — using the admin from the first-run scenario,
  sign out, navigate to `/admin/edit/about`, expect redirect to
  `/admin/login?next=/admin/edit/about`, sign in, expect to land back on
  `/admin/edit/about`.
- The existing editor smoke is updated to set `AMBER_AUTH_SECRET` instead
  of `AMBER_DEV_UNSAFE=1`; the first-run scenario claims the admin so
  later scenarios have a session.
- **Google OAuth is not e2e-tested.** Stubbing the provider is more
  scaffolding than the value justifies; the callback handler is unit-tested
  against canned responses instead.

### What's deliberately NOT tested

- Better-auth's cookie signing, session expiry, or hash format. Testing
  a vendored library is testing the library, not Amber.
- Brute-force / rate-limit behavior — out of scope (§1, §9).
- Multi-user behavior — subsystem 4.

## 12. CLAUDE.md revisions this subsystem will land (named, not made)

For the plan and build sessions to action — not edited here:

1. **"`.amber/` holds regenerable runtime state … and is safe to delete."**
   Narrowed: `.amber/cache.db` (and other regenerable artifacts) are safe
   to delete; `.amber/auth.db` is **not** — it holds the admin account and
   sessions. Backup guidance updated accordingly.
2. **"Every dependency justifies its presence."** `better-auth` joins the
   permitted-dependency list, by the same standard the spec applied to
   Crepe in subsystem 1: it is the load-bearing library of this subsystem,
   server-only, and contained to the admin surface plus the auth seam.

Both revisions land *with* this subsystem's code, as CLAUDE.md's
"Roadmap shape" section already prescribes — the spec names them so the
plan accounts for the work.

## 13. Open questions for the operator

None block a plan session. Recorded for visibility:

- **Session lifetime tuning.** Better-auth's default 30-day session is
  reasonable for a personal canvas; this can be retuned at build time
  without affecting the architecture.
- **Sign-out UI placement.** The admin chrome currently has a small
  header bar (`/admin/+layout.svelte`); whether sign-out lives there or
  on `/admin/account` only is a build-time UX detail.
- **The exact CLI command name** (`reset-password`, `recover`,
  `bootstrap`, …) is a build-time detail; the spec leaves it to the build
  to pick one and document it.

These are deliberately left to the build; none change the architecture.
