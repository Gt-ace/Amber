# v0.5 subsystem 5 — space-creation UI follow-ups

Subsystem 5 shipped on branch `feat/v0.5-subsystem-5-space-creation`.
The final whole-branch review was **Approved with non-blocking notes**.
What landed pre-merge — including review-driven polish — is below,
followed by the open follow-ups that did not block deploy.

## Resolved before merge

- **Resolver-index rollback shape (Important).** The first cut of
  `addSpace()` performed three module-level state mutations (`set*`)
  inside a `try` block where the catch only rolled back the registry.
  A future contributor inserting a throwable line between the swaps
  could leave the resolver pointing at a just-closed `Space`.
  Restructured in `74054b2` so all computation completes before any
  `set*` call; the three swaps are bunched at the tail and the catch
  only handles registry rollback.
- **`computeDefaultSlug` divergence (Important).** The inline
  derivation in `addSpace` returned `null` when `nextIndex.default`
  was set but no matching entry was found; `hooks.server.ts`'s local
  function falls back to `path.basename(entries[0].path)`. Unreachable
  today, but a real drift risk. Extracted to a shared
  `computeDefaultSlug(index, entries)` helper in `default-space.ts`
  (`74054b2`); both call sites use it.
- **`CreateSpaceResult` discriminated union (Important).** First cut
  was `{ kind: 'ok' | 'error'; absPath?; code?; detail? }`, requiring
  the route action to use non-null assertions to read `absPath` after
  the kind check. Rewrote as a proper union in `b937dcd`; the route
  reads cleanly.
- **`pattern` attribute Svelte 5 syntax bug.** The form template
  shipped with `pattern="^[a-z0-9][a-z0-9-]{{0,62}}$"` — the double
  curly braces are invalid inside a plain HTML attribute string and
  caused the `bun run build` step to fail (which broke the e2e suite
  at boot). Fixed in `b996f03` to expression-form
  `pattern={"^[a-z0-9][a-z0-9-]{0,62}$"}`. Caught by Task 9's smoke,
  not by Task 7's unit review — `+page.svelte` doesn't run in
  Vitest's server-project import path.
- **`slugDirty` initial state on re-render after server rejection.**
  When a submission failed server-side and the form re-mounted with
  `form.raw.*`, `slugDirty` initialised to `false`. A user who
  hand-edited their slug, hit a write error, then corrected the title
  would silently lose their slug choice when the auto-derive fired
  again. Initial state now seeds from a derive-vs-stored comparison.

## Open follow-ups

### Open — `addSpace-load-warning` subsystem tag

Spec §10 describes a `warn, addSpace-load-warning` event tagged with
`subsystem: 'space-create'` so a chronological log read makes the
"operator submitted → load warned → space added anyway" chain visible.
The current implementation routes `loadEntry`'s warnings through the
shared `subsystem: 'server'` logger (because `loadEntry` is the same
private helper boot uses for `getSpace`). To honour the spec the
warnings need to be returned from `loadEntry` and re-emitted in
`addSpace` with the right tag. Pure observability gap — no correctness
impact.

### Open — route integration test for `addSpace`-fails-after-write

Spec §8 listed: "action write-failure: stub the writer to fail at step
3 → response carries `write_failed`, the partial `<slug>/` directory
was rolled back." The writer's failure modes (`dir_already_exists`,
`permission_denied`, `write_failed`) are unit-tested in
`space-create.test.ts` via the `__forceFailAfter` hook. The route's
`addSpace`-fails-after-write path — where the writer succeeded but
`addSpace` threw, and the route does its own `rmSync` cleanup — is
not exercised through the route action. Add a `vi.spyOn` or `vi.mock`
based test in `new-space.test.ts` that forces `addSpace` to reject
mid-action, asserts the response carries `writeError: 'write_failed'`,
and asserts the partial directory is gone.

### Open — operator-facing copy on the empty-state "Create the first space" CTA

The spec §11 flagged this as a punted decision. Current copy
("No spaces loaded yet. Create the first space to get started.")
ships fine but is worth a second pass once an actual install-admin
sits in front of an empty install. Low priority, no semantic gap.

### Open — host/prefix preview affordance

The spec §11 also flagged: the form's routing radio could show a
calculated public URL ("this space will serve
`https://example.com/notes`") next to the chosen option. Not built;
deferred until operators ask. Easy to add later — admin host +
prefix → URL is 10-20 lines of derivation logic in the template.

### Open — CLAUDE.md "Hot reload" non-goal revision was a no-op

Task 10's spec called for a revision to the subsystem 3
"Hot reload of routing config" non-goal bullet inside CLAUDE.md.
That bullet lives in the subsystem 3 *spec doc*, not in CLAUDE.md
proper — the implementer found nothing to edit and noted it. The
intent (subsystem 5 revises that non-goal for the create-from-UI
path) is correctly carried in CLAUDE.md's roadmap-shape sub-bullet
for subsystem 5 ("Hot-adds the new space into the resolver index").
No further CLAUDE.md change needed; flag is closed.
